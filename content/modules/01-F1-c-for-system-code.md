## 1. Почему системный код — это C, и чем это меняет голову

В C++ ты опираешься на три вещи, которых в системном C нет:

1. **RAII** — деструктор закрывает ресурс сам. В C ты закрываешь его руками, на каждом пути выхода, включая пути ошибок. Забыл один `return` — утечка fd/памяти, которая в долгоживущем демоне копится днями.
2. **Типобезопасность шаблонов и перегрузок.** В C `void*`, макросы и `_Generic` вместо дженериков. Ошибка типа не ловится компилятором, а стреляет в рантайме.
3. **Менее агрессивный UB в повседневном коде.** В C *повседневные* конструкции (целочисленная арифметика, приведения указателей, неинициализированные переменные) — это поле UB, которое современный оптимизатор эксплуатирует жёстко.

Третий пункт — ключевой сдвиг мышления. **UB в C — это не «программа упадёт». Это «компилятор имеет право считать, что этого не случится, и переписать код исходя из этого».** Отсюда баги, которые: появляются только в `-O2`, исчезают под отладчиком, зависят от версии компилятора и «чинятся» добавлением `printf`. Весь модуль крутится вокруг умения такие места *видеть глазами*.

### Минимальная модель трансляции (нужна для всего остального)

Прежде чем про UB — три понятия, на которых стоит весь C и которые в ядре всплывают постоянно.

**Translation unit (TU)** — один `.c` после препроцессора (с раскрытыми `#include`). Компилятор видит ровно один TU за раз и ничего не знает про другие. Отсюда `extern`, заголовки и линковка.

**Linkage (связывание имени):**
- *external* — имя видно из других TU. Глобальные функции/переменные по умолчанию. В ядре — то, что попадает в `EXPORT_SYMBOL`.
- *internal* — `static` на файловом уровне: имя видно только в своём TU. В ядре `static` повсюду именно ради этого — не засорять глобальное пространство имён.
- *none* — локальные переменные, параметры.

**Storage duration (время жизни):**
- *automatic* — локальные без `static`; живут до конца блока. Стек.
- *static* — `static`-локальные и все файловые; живут всю программу. Инициализируются нулём по умолчанию (важно: в C это гарантия стандарта).
- *allocated* — `malloc`; живут до `free`. Здесь и водятся утечки.
- *thread* (`_Thread_local`) — по одному экземпляру на поток (это Ф3, TLS).

> **Грабли C, которых нет в C++ так явно:** `static`-локальная инициализируется *один раз*, при первом проходе. Но в отличие от C++ это **не потокобезопасно** само по себе — нет «magic statics». В многопоточном системном коде ленивая инициализация через `static` локальную — гонка. Используй `call_once`/`pthread_once`.

## 2. Undefined / Unspecified / Implementation-defined — три разных зверя

Их постоянно путают, а разница принципиальная (§3.4 стандарта).

| Категория | Определение | Можно полагаться? | Пример |
|---|---|---|---|
| **Implementation-defined** | Поведение на выбор реализации, но она **обязана задокументировать** выбор. | Да, если читал доку компилятора/ABI. | Знаковость `char`; размер `int`; что делает `>>` на отрицательном (арифметический сдвиг — на gcc/clang). |
| **Unspecified** | Реализация выбирает из набора вариантов, документировать **не обязана**, и выбор может отличаться от случая к случаю. | Нет, но программа остаётся валидной. | Порядок вычисления аргументов функции; порядок вычисления `a + b`. |
| **Undefined (UB)** | Стандарт **не накладывает никаких требований**. Любое поведение легально, включая «как ты ожидал». | **Никогда.** Наличие UB на исполняемом пути делает невалидной всю программу. | Знаковое переполнение; выход за границы массива; разыменование NULL; гонка данных; нарушение strict aliasing. |

Ключевая фраза стандарта про UB: *«no requirements»*. Это и есть лицензия оптимизатору. Практический закон, который надо вбить:

> **Компилятор оптимизирует в предположении, что UB не происходит.** Если ты написал UB, компилятор не «делает что-то странное назло» — он делает корректный вывод из ложной посылки «этого не бывает», и из ложной посылки следует что угодно.

### Канонический пример: компилятор удаляет твою проверку

```c
#include <limits.h>
int will_overflow(int x) {
    return x + 1 < x;   // "проверка на переполнение"
}
```

Знаковое переполнение — UB. Значит компилятор вправе считать, что `x + 1` никогда не переполняется, значит `x + 1 < x` **всегда ложно**, значит функция компилируется в `return 0;`. Твоя проверка переполнения испаряется на `-O2`. Под `-O0` она «работает» (опирается на wrap-around железа) — отсюда «в дебаге всё ок, в релизе баг».

```
$ gcc -O2 -S → will_overflow:  xor eax, eax;  ret      # буквально return 0
```

Правильно проверять *до* операции, на области определения, либо через билтины:

```c
int add_checked(int a, int b, int *out) {
    return __builtin_add_overflow(a, b, out);  // 1 если переполнилось, *out корректен по модулю
}
```

### Три примера UB, которые молча ломают релизную сборку (это вопрос 1 самопроверки)

**(1) Знаковое переполнение** — разобран выше. Лечение: `-fsanitize=signed-integer-overflow`, билтины, `-ftrapv` (ловушка) или `-fwrapv` (определить как wrap, ценой потери оптимизаций).

**(2) Чтение неинициализированной переменной с indeterminate value.** Не просто «мусор» — компилятор может считать, что путь, читающий неинициализированное, недостижим:

```c
int f(int cond) {
    int x;                 // indeterminate
    if (cond) x = 10;
    return x;              // если cond==0 — UB, не "вернёт мусор"
}
```

Под оптимизацией функция, вызванная с `cond==0`, может вернуть что угодно, и хуже — компилятор может выкинуть саму ветку. Лечение: инициализируй при объявлении; `-Wmaybe-uninitialized`, MSan.

**(3) Нарушение strict aliasing** (раздел 5). Запись через указатель «не того» типа и чтение через другой — компилятор считает, что они не пересекаются, и переупорядочивает/кэширует. Лечение: `memcpy`, `-fno-strict-aliasing` (так собирается само ядро Linux, об этом ниже), `-fsanitize=...` тут слабо ловит — нужен глаз и `-Wstrict-aliasing`.

> Другие частые: разыменование NULL (компилятор выводит «раз разыменовали — значит не NULL» и удаляет последующую проверку — реальный CVE в ядре, `tun` driver, 2009), сдвиг на ширину типа или больше (`x << 32` для 32-битного — UB), деление `INT_MIN / -1`, выход за границы массива, использование указателя после `free`, `memcpy` перекрывающихся областей (нужен `memmove`), две модификации объекта без точки следования (`i = i++`).

### Инструмент дня: UBSan

```c
// ub.c
int main(void){ int x = 0x7fffffff; return x + 1; }
```
```
$ clang -fsanitize=undefined -O1 ub.c -o ub && ./ub
ub.c:1:54: runtime error: signed integer overflow: 2147483647 + 1 cannot be represented in type 'int'
```
UBSan — *рантайм*: ловит только то UB, что реально исполнилось на твоих входах. Поэтому он напарник тестов/фаззинга, а не замена чтению кода. Что ловит UBSan, а что ASan/TSan — вопрос 8 самопроверки, держи в голове: **UBSan — логические UB (переполнение, сдвиги, неверные приведения), ASan — память (OOB, use-after-free, утечки), TSan — гонки данных.**

## 3. Целочисленные типы, продвижения и конверсии

Здесь живёт больше системных багов, чем в любой другой теме C. Разберём по слоям.

### 3.1 Ранги и продвижение (integer promotion)

Каждый целочисленный тип имеет **conversion rank** (§6.3.1.1): `bool < char < short < int < long < long long`, со знаковыми/беззнаковыми одного ранга наравне.

**Integer promotion:** в большинстве выражений типы *уже́ `int`* (`char`, `short`, битовые поля, `bool`) автоматически продвигаются до `int` (или `unsigned int`, если `int` не вмещает все значения). Это происходит **до** любой арифметики.

Следствие, которое ломает мозг новичкам:

```c
unsigned char a = 0xFF, b = 0xFF;
unsigned char c = a * b;     // a,b → int; int(255)*int(255)=65025 (влезает в int!); потом усечение до uchar
// арифметика идёт в int, а не в unsigned char — переполнения uchar тут НЕТ
```

И обратное — продвижение делает поведение неинтуитивным для `~`:

```c
uint8_t x = 0;
printf("%d\n", ~x);     // НЕ 255. x → int 0; ~0 == -1 (int). Печатает -1.
if (~x == 0xFF) { ... } // ложно! ~x == -1
```

### 3.2 Usual arithmetic conversions (приведение операндов бинарной операции)

Когда оба операнда уже не уже́ `int`, применяются *usual arithmetic conversions* (§6.3.1.8): операнды приводятся к общему типу по рангу, причём **при равном ранге и разной знаковости побеждает беззнаковый**. Это источник классического бага:

```c
int i = -1;
unsigned u = 0;
if (i < u) puts("i < u"); else puts("i >= u");   // печатает "i >= u" !!!
// i приводится к unsigned: -1 → 4294967295, которое > 0
```

В системном коде это особенно злобно с `size_t` (беззнаковый):

```c
size_t n = some_count();
for (size_t i = n - 1; i >= 0; i--) { ... }   // БЕСКОНЕЧНЫЙ ЦИКЛ: size_t всегда >= 0
```
```c
if (strlen(s) - 1 < some_int) { ... }   // если s пустая: 0-1 = SIZE_MAX, сравнение в size_t
```

**Дисциплина:** `-Wsign-conversion -Wsign-compare`; не смешивай знаковое и беззнаковое в сравнениях; для индексов «назад» используй другой паттерн (`for (size_t i = n; i-- > 0;)`).

### 3.3 Знаковость `char` и сдвиги

- **`char` имеет implementation-defined знаковость.** На x86-Linux — знаковый; на ARM по умолчанию — беззнаковый. Код `char c = getchar(); if (c == EOF)` ломается на платформах со знаковым char для байта 0xFF vs EOF — поэтому `getchar()` возвращает `int`, а не `char`. Для байтов всегда `unsigned char` или `uint8_t`.
- **Сдвиг `>>` отрицательного** — implementation-defined (gcc/clang: арифметический). **Сдвиг на ширину типа и больше — UB** (`uint32_t x; x << 32`). **Сдвиг отрицательным числом — UB.**
- **`x << 31` для `int`**, где результат не влезает в `int`, — UB (до C23 заход в знаковый бит — отдельная тонкость). Для битовых масок используй беззнаковые литералы: `1u << 31`, `1ull << 40`.

### 3.4 Фиксированные типы и форматы

В системном коде — только `<stdint.h>`: `uint32_t`, `int64_t`, `uintptr_t` (целое, в которое влезает указатель), `intmax_t`. Для размеров — `size_t`, для разностей указателей — `ptrdiff_t`, для смещений в файле — `off_t`. Печать через `<inttypes.h>`:

```c
uint64_t v = ...;
printf("%" PRIu64 "\n", v);          // переносимо
printf("%zu\n", sizeof(x));          // size_t
printf("%p\n", (void*)ptr);          // указатель — обязательно (void*)
```

> **Практика 3.A:** напиши `bool in_range_i32(long v)`, возвращающую истину, если `v` помещается в `int32_t`, без UB и без зависимости от ширины `long`. Проверь на `INT32_MIN`, `INT32_MAX`, их соседях.

## 4. Strict aliasing и правило эффективного типа

Самая «невидимая» тема и любимый источник «работало 5 лет, потом обновили gcc и сломалось».

### 4.1 Правило

§6.5p6–7: у каждого объекта в памяти есть **эффективный тип**. Обращаться (читать/писать) к объекту можно только через lvalue **совместимого** типа, с оговорёнными исключениями. Разрешено всегда:

- тип, совпадающий с эффективным (с точностью до `const`/`volatile`/знаковости);
- **`char`-семейство** (`char`, `signed char`, `unsigned char`) — через них можно читать **любой** объект побайтно. Это легальная «лазейка» для сериализации;
- тип агрегата/union, содержащего совместимый член.

Всё остальное — **UB**, и компилятор это эксплуатирует так: «указатели разных, несовместимых типов не указывают на один объект» → можно не перечитывать память, можно переупорядочить.

```c
// Классика: "type punning" через указатель — UB
float bit_hack(float f) {
    uint32_t i = *(uint32_t*)&f;   // читаем float-объект через uint32_t* — НАРУШЕНИЕ
    i |= 0x80000000u;
    return *(float*)&i;            // и обратно
}
```

Почему это реально стреляет:

```c
void scale(float *f, int *i) {     // компилятор: f и i разных типов → не алиасятся
    *i = 5;
    *f = 2.0f;                      // вправе НЕ перечитывать *i после этого
    *i += 1;                        // мог закэшировать *i == 5 → запишет 6, даже если f==(float*)i
}
```

### 4.2 Легальные способы (вопрос 2 самопроверки)

**(1) `memcpy`** — канонический и переносимый. Компилятор его распознаёт и при `-O2` генерирует тот же один `mov`, что и грязный каст, — нулевая цена:

```c
uint32_t f2u(float f) { uint32_t u; memcpy(&u, &f, sizeof u); return u; }
```

**(2) Доступ через `char*`/`unsigned char*`** — разрешён стандартом для побайтной интерпретации (сериализация, хеши, дампы).

**(3) `union` — но осторожно с различием C и C++!**
- **В C** чтение из *другого* члена union, чем тот, что записан («type punning через union»), — **разрешено** и хорошо определено (footnote в §6.5.2.3): значение реинтерпретируется по представлению.
- **В C++** это формально UB (там легален только `memcpy` или `std::bit_cast`). Поэтому привычка из C++ «делай memcpy» — самая переносимая.

```c
// Легально в C:
union { float f; uint32_t u; } pun = { .f = 3.14f };
uint32_t bits = pun.u;
```

**(4) `-fno-strict-aliasing`** — выключить оптимизацию целиком. **Ядро Linux собирается именно с этим флагом** (`Makefile`: `KBUILD_CFLAGS += -fno-strict-aliasing`), потому что низкоуровневый код массово кастует указатели. То есть в коде ядра ты можешь кастовать — но в своём userspace-коде по умолчанию правило действует, и полагаться на «как в ядре» нельзя.

> **Вывод-дисциплина:** в переносимом userspace — `memcpy` для реинтерпретации битов; `char*` для побайтного разбора; помни, что в ядре правило отключено флагом, а не отсутствует.

> **Практика 4.A:** реализуй `fast_inv_sqrt` (легендарный Quake-хак) дважды — грязным кастом и через `memcpy` — и сравни ассемблер на godbolt под `-O2 -fstrict-aliasing`. Убедись, что `memcpy`-версия даёт тот же код и не ломается.

## 5. `restrict` — обещание оптимизатору

`restrict` (§6.7.3.1) — квалификатор указателя (только указателей). Семантика-контракт: **«в течение жизни этого указателя к объекту, на который он указывает, не обращаются ни через какой другой указатель»**. Это *обещание программиста компилятору*, не проверяемое. Нарушил — UB.

Зачем: позволяет компилятору не перечитывать память и векторизовать. Классика — `memcpy` (её прототип `void *memcpy(void *restrict dst, const void *restrict src, size_t n)` — отсюда требование непересечения, иначе бери `memmove`).

```c
// Без restrict компилятор обязан учитывать, что a и b могут пересекаться:
void add(size_t n, float *a, const float *b) {
    for (size_t i = 0; i < n; i++) a[i] += b[i];   // вынужден перечитывать с оглядкой на алиасинг
}
// С restrict — обещаем, что a и b не пересекаются → свободная векторизация:
void add_r(size_t n, float *restrict a, const float *restrict b) {
    for (size_t i = 0; i < n; i++) a[i] += b[i];
}
```

На godbolt `add_r` под `-O3` разворачивается в SIMD, `add` — нет (или с проверкой перекрытия). Это и есть вопрос «чем `restrict` помогает оптимизатору» из самопроверки фундамента: **снимает консервативное предположение об алиасинге**, давая переупорядочивание и векторизацию.

> **Грабли:** если вызвать `add_r(n, p, p)` — ты нарушил обещание, поведение не определено. `restrict` — инструмент горячих циклов, а не «добавить везде для скорости».

## 6. Управление ресурсами без RAII

В ядре нет деструкторов. Каждый ресурс (память, fd, блокировка, refcount) освобождается вручную, на **каждом** пути выхода. Ошибки здесь — №1 источник утечек и use-after-free в системном коде. Разберём идиомы, которые используются в самом ядре.

### 6.1 Идиома «goto cleanup» (single exit, лестница освобождения)

В C++ ты бы положился на стек деструкторов. В C каноничен **обратный goto-разворот**: ресурсы освобождаются в порядке, обратном захвату, через метки.

```c
int do_work(const char *path) {
    int ret = -1;                       // по умолчанию — ошибка
    char *buf = NULL;
    FILE *f = NULL;

    buf = malloc(BUFSZ);
    if (!buf)
        goto out;                       // ещё ничего не захвачено сверх buf
    f = fopen(path, "rb");
    if (!f)
        goto out_free_buf;              // прыжок на нужную ступень

    if (fread(buf, 1, BUFSZ, f) == 0)
        goto out_close;                 // ошибка — но f и buf освободим

    ret = 0;                            // успех

out_close:
    fclose(f);
out_free_buf:
    free(buf);
out:
    return ret;
}
```

Почему так, а не «free на каждом return»:
- **DRY:** освобождение в одном месте, не дублируется по веткам;
- **порядок:** метки выстроены в обратном порядке захвата — корректный teardown;
- **это буквально стиль ядра** (`Documentation/process/coding-style.rst` прямо рекомендует goto для cleanup). Читая исходники драйверов, ты увидишь именно это.

Тонкости:
- `free(NULL)` безопасен (no-op) — поэтому можно инициализировать всё `NULL`/`-1` и иметь единую метку `out`, безопасно «освобождая» незахваченное. Но для fd `close(-1)` вернёт `EBADF` — fd инициализируй `-1` и оборачивай `if (fd >= 0) close(fd);`.
- Не прыгай goto *через* объявление с инициализацией внутри блока (в C это допустимо синтаксически, но переменная будет с indeterminate value).

### 6.2 Дисциплина владения (ownership)

RAII в C++ делает владение типобезопасным. В C владение — **соглашение, записанное в комментарии и имени**:

- Кто аллоцировал — обычно тот и освобождает; если нет — это должно быть в контракте функции явно («caller takes ownership of returned buffer, must free()»).
- Суффиксы-конвенции: `*_create`/`*_destroy`, `*_alloc`/`*_free`, `*_get`/`*_put` (refcount, как `kref` в ядре).
- После `free(p)` — **`p = NULL;`** (защита от double-free и use-after-free; `free(NULL)` безопасен).
- Возвращая владение — документируй. Принимая указатель — документируй, «занимаешь» (borrow) или «забираешь» (move).

### 6.3 Частые дефекты и чем ловить

| Дефект | Симптом | Инструмент |
|---|---|---|
| Утечка (не дошли до `free`) | растёт RSS демона | ASan (`LeakSanitizer`), valgrind `--leak-check=full` |
| Double-free | краш/коррупция кучи | ASan |
| Use-after-free | «иногда» неверные данные/краш | ASan |
| Использование после `realloc` старого указателя | UAF (realloc мог переместить) | ASan; правило: `p = realloc(p, n)` только во временную, проверь, потом присвой |
| `realloc(p, 0)` | impl-defined (может вернуть NULL ≠ ошибка) | не используй для «освободить» |
| Неосвобождённый fd | `EMFILE` под нагрузкой | strace, `/proc/<pid>/fd`, статический анализ |

```c
// realloc без потери буфера при ошибке:
char *tmp = realloc(buf, newsz);
if (!tmp) { /* buf ещё валиден, не потерян */ goto err; }
buf = tmp;
```

> **Практика 6.A:** напиши `char *read_whole_file(const char *path, size_t *out_len)` с goto-cleanup: открыть, узнать размер, выделить, прочитать, на любой ошибке вернуть NULL без утечек. Прогони под `-fsanitize=address` и valgrind на: несуществующем файле, пустом файле, файле > доступной памяти (замокай malloc-фейл). Это прямой задел под Ф2 (файловый I/O) и капстоун.

## 7. Системный C: то, чего нет в «учебном» C

Короткий, но обязательный набор — это встречается в каждом заголовке ядра.

**`const` vs `volatile`.** `const` — обещание «я не меняю». `volatile` — «значение может измениться вне видимости компилятора, не кэшируй и не выкидывай обращения» (MMIO-регистры, переменная, меняемая обработчиком сигнала). **`volatile` ≠ атомарность и ≠ барьер памяти** — частое заблуждение из embedded; в многопоточности нужны `_Atomic`/барьеры (это C1). В ядре для MMIO — не голый `volatile`, а `readl`/`writel`.

**`sig_atomic_t` / `volatile sig_atomic_t`** — единственный тип, который безопасно трогать из обработчика сигнала (Ф/C4). 

**Выравнивание и layout структур (§6.7.2.1, §6.2.8).** Компилятор вставляет padding для выравнивания членов. Это влияет на `sizeof`, на бинарную совместимость (протоколы, сериализация — никогда не `fwrite(&struct)` без `__attribute__((packed))` или ручной сериализации), на false sharing (C1/C6).

```c
struct bad  { char a; int b; char c; };   // sizeof обычно 12 (паддинг)
struct good { int b; char a; char c; };   // sizeof обычно 8 (упаковано плотнее)
```
`_Alignof(T)`, `alignas`/`_Alignas`, `offsetof(struct, member)` — твои инструменты. `container_of` ядра (вопрос K1) построен ровно на `offsetof`.

**Flexible array member** — «хвост переменной длины», паттерн ядра:
```c
struct msg { uint32_t len; char data[]; };          // C99 FAM
struct msg *m = malloc(sizeof *m + payload_len);     // аллокация с хвостом
```

**Полезный современный C (C11/C23), часто в системном коде:**
- `_Generic` — перегрузка по типу (на нём построены типобезопасные макросы);
- designated initializers `{ .field = x }` — массово в ядре для таблиц `file_operations` и т.п.;
- compound literals `(struct foo){ ... }`;
- `_Static_assert(sizeof(struct hdr) == 8, "ABI!")` — проверка ABI на этапе компиляции;
- `__attribute__((packed, aligned, cleanup))` — GNU-расширения. **`cleanup`** даёт почти-RAII в GCC/Clang и используется в новом ядре (`__free`, `guard()` из `cleanup.h`):
```c
static inline void freep(void *p){ free(*(void**)p); }
#define _autofree __attribute__((cleanup(freep)))
void f(void){ _autofree char *buf = malloc(100); /* free автоматически на выходе */ }
```

> **Практика 7.A:** объяви протокольный заголовок пакета фиксированного бинарного формата (magic, version, len, flags). Подтверди `sizeof` через `_Static_assert`, выведи offset каждого поля через `offsetof`, объясни весь padding. Сериализуй через `memcpy` в `unsigned char buf[]`, не через каст структуры.

<details>
<summary><b>Эталонные ответы (раскрыть после попытки)</b></summary>

1. **Impl-defined** — реализация выбирает и документирует (знаковость `char`). **Unspecified** — выбирает из набора, документировать не обязана, может меняться (порядок вычисления аргументов). **UB** — никаких требований, вся программа невалидна (знаковое переполнение). Практическая разница: на impl-defined можно полагаться, прочитав доку компилятора/ABI; на unspecified — нет, но программа остаётся корректной.
2. Наличие UB на пути исполнения снимает с компилятора любые обязательства на этом пути; он выводит факты из «UB не происходит». `x+1<x`: знаковое переполнение UB ⇒ `x+1` «всегда» `>x` ⇒ выражение всегда `0` ⇒ `return 0`. Проверка исчезает на `-O2`.
3. (а) знаковое переполнение — билтины/`-fwrapv`/UBSan; (б) чтение неинициализированного — инициализация при объявлении/MSan; (в) нарушение strict aliasing — `memcpy`/`-fno-strict-aliasing`/`-Wstrict-aliasing`. (Также: NULL-дереф с последующим удалением проверки, сдвиг ≥ ширины.)
4. Промоушен: `char/short/bool/битполя` → `int` до арифметики. UAC: операнды бинарной операции приводятся к общему типу по рангу; при равном ранге беждает беззнаковый. `-1 < 0u`: `-1`→`UINT_MAX` ⇒ ложно. `~(uint8_t)0`: операнд → `int 0`, `~0 == -1` (int).
5. `size_t` беззнаковый, всегда `>= 0`, после `0` идёт `SIZE_MAX` ⇒ условие никогда не ложно. Корректно: `for (size_t i = n; i-- > 0; ) { use(i); }`.
6. Доступ к объекту — только через совместимый тип, кроме char-семейства и union-члена. Легально для битов: `memcpy`; через `unsigned char*` побайтно; union-punning (в C). C vs C++: union-punning определён в C, UB в C++ (там `memcpy`/`bit_cast`). Ядро: массовые касты низкоуровневого кода → отключают оптимизацию флагом, иначе UB.
7. `restrict`: к объекту не обращаются ни через какой иной указатель в течение жизни указателя. Взамен — снятие предположения об алиасинге → переупорядочивание/векторизация. Нарушение — UB (тихая порча данных под оптимизацией).
8. goto-cleanup: единое место освобождения в обратном порядке захвата, без дублирования по веткам, корректный teardown — стиль ядра. `free(NULL)` no-op ⇒ можно инициализировать всё в NULL и иметь общую метку. `close(-1)` даёт `EBADF` ⇒ fd надо инициализировать `-1` и закрывать под `if (fd>=0)`.
9. `volatile` — не атомарность и не барьер памяти; гарантирует лишь «не кэшировать/не выкидывать обращения». Для потоков нужны `_Atomic`/барьеры; `volatile` не упорядочивает доступы к другим объектам и не даёт атомарности RMW.
10. UBSan — логические UB в рантайме (переполнение, сдвиги, неверные приведения, NULL-дереф). ASan — память (OOB, UAF, double-free, утечки). TSan — гонки. Баг из в.2 поймал бы UBSan (`signed-integer-overflow`), если этот вход реально исполнится.

</details>
