## 1. Почему системный код — это C, и чем это меняет голову

В C++ ты опираешься на три вещи, которых в системном C нет:

1. **RAII** — деструктор закрывает ресурс сам. В C ты закрываешь его руками, на каждом пути выхода, включая пути ошибок. Забыл один `return` — утечка fd/памяти, которая в долгоживущем демоне копится днями.
2. **Типобезопасность шаблонов и перегрузок.** В C `void*`, макросы и `_Generic` вместо дженериков. Ошибка типа не ловится компилятором, а стреляет в рантайме.
3. **Менее агрессивный UB в повседневном коде.** В C *повседневные* конструкции (целочисленная арифметика, приведения указателей, неинициализированные переменные) — это поле UB, которое современный оптимизатор эксплуатирует жёстко.

Третий пункт — ключевой сдвиг мышления. **UB в C — это не «программа упадёт». Это «компилятор имеет право считать, что этого не случится, и переписать код исходя из этого».** Отсюда баги, которые: появляются только в `-O2`, исчезают под отладчиком, зависят от версии компилятора и «чинятся» добавлением `printf`. Весь модуль крутится вокруг умения такие места *видеть глазами*.

### Минимальная модель трансляции (нужна для всего остального)

Прежде чем про UB — три понятия, на которых стоит весь C и которые в ядре всплывают постоянно.

**Translation unit (TU)** — один `.c` после препроцессора (с раскрытыми `#include`). Компилятор видит ровно один TU за раз и ничего не знает про другие. Отсюда `extern`, заголовки и линковка.

**Linkage (связывание имени):**
- *external* — имя видно из других TU. Глобальные функции/переменные по умолчанию. В ядре — то, что попадает в `EXPORT_SYMBOL`.
- *internal* — `static` на файловом уровне: имя видно только в своём TU. В ядре `static` повсюду именно ради этого — не засорять глобальное пространство имён.
- *none* — локальные переменные, параметры.

**Storage duration (время жизни):**
- *automatic* — локальные без `static`; живут до конца блока. Стек.
- *static* — `static`-локальные и все файловые; живут всю программу. Инициализируются нулём по умолчанию (важно: в C это гарантия стандарта).
- *allocated* — `malloc`; живут до `free`. Здесь и водятся утечки.
- *thread* (`_Thread_local`) — по одному экземпляру на поток (это Ф3, TLS).

> **Грабли C, которых нет в C++ так явно:** инициализатор объекта со static storage duration **обязан быть константным выражением** (§6.7.9p4) — поэтому `static`-локальная инициализируется не «при первом проходе», а при загрузке, готовым значением; динамической инициализации (а с ней и C++-проблемы «magic statics») в C просто нет. Зато **ручная** ленивая инициализация — `static T *p = NULL; if (!p) p = make();` — в многопоточном коде это гонка, и никакой компилятор её за тебя не закрывает. Используй `call_once`/`pthread_once`.

## 2. Undefined / Unspecified / Implementation-defined — три разных зверя

Их постоянно путают, а разница принципиальная (§3.4 стандарта).

| Категория | Определение | Можно полагаться? | Пример |
|---|---|---|---|
| **Implementation-defined** | Поведение на выбор реализации, но она **обязана задокументировать** выбор. | Да, если читал доку компилятора/ABI. | Знаковость `char`; размер `int`; что делает `>>` на отрицательном (арифметический сдвиг — на gcc/clang). |
| **Unspecified** | Реализация выбирает из набора вариантов, документировать **не обязана**, и выбор может отличаться от случая к случаю. | Нет, но программа остаётся валидной. | Порядок вычисления аргументов функции; порядок вычисления `a + b`. |
| **Undefined (UB)** | Стандарт **не накладывает никаких требований**. Любое поведение легально, включая «как ты ожидал». | **Никогда.** Наличие UB на исполняемом пути делает невалидной всю программу. | Знаковое переполнение; выход за границы массива; разыменование NULL; гонка данных; нарушение strict aliasing. |

Ключевая фраза стандарта про UB: *«no requirements»*. Это и есть лицензия оптимизатору. Практический закон, который надо вбить:

> **Компилятор оптимизирует в предположении, что UB не происходит.** Если ты написал UB, компилятор не «делает что-то странное назло» — он делает корректный вывод из ложной посылки «этого не бывает», и из ложной посылки следует что угодно.

### Канонический пример: компилятор удаляет твою проверку

```c
#include <limits.h>
int will_overflow(int x) {
    return x + 1 < x;   // "проверка на переполнение"
}
```

Знаковое переполнение — UB. Значит компилятор вправе считать, что `x + 1` никогда не переполняется, значит `x + 1 < x` **всегда ложно**, значит функция компилируется в `return 0;`. Твоя проверка переполнения испаряется на `-O2`. Под `-O0` она «работает» (опирается на wrap-around железа) — отсюда «в дебаге всё ок, в релизе баг».

```
$ gcc -O2 -S → will_overflow:  xor eax, eax;  ret      # буквально return 0
```

Правильно проверять *до* операции, на области определения, либо через билтины:

```c
int add_checked(int a, int b, int *out) {
    return __builtin_add_overflow(a, b, out);  // 1 если переполнилось, *out корректен по модулю
}
```

### Три примера UB, которые молча ломают релизную сборку (это вопрос 1 самопроверки)

**(1) Знаковое переполнение** — разобран выше. Лечение: `-fsanitize=signed-integer-overflow`, билтины, `-ftrapv` (ловушка) или `-fwrapv` (определить как wrap, ценой потери оптимизаций).

**(2) Чтение неинициализированной переменной с indeterminate value.** Не просто «мусор» — компилятор может считать, что путь, читающий неинициализированное, недостижим:

```c
int f(int cond) {
    int x;                 // indeterminate
    if (cond) x = 10;
    return x;              // если cond==0 — UB, не "вернёт мусор"
}
```

Под оптимизацией функция, вызванная с `cond==0`, может вернуть что угодно, и хуже — компилятор может выкинуть саму ветку. Лечение: инициализируй при объявлении; `-Wmaybe-uninitialized`, MSan.

**(3) Нарушение strict aliasing** (раздел 5). Запись через указатель «не того» типа и чтение через другой — компилятор считает, что они не пересекаются, и переупорядочивает/кэширует. Лечение: `memcpy`, `-fno-strict-aliasing` (так собирается само ядро Linux, об этом ниже), `-fsanitize=...` тут слабо ловит — нужен глаз и `-Wstrict-aliasing`.

> Другие частые: разыменование NULL (компилятор выводит «раз разыменовали — значит не NULL» и удаляет последующую проверку — реальный CVE в ядре, `tun` driver, 2009), сдвиг на ширину типа или больше (`x << 32` для 32-битного — UB), деление `INT_MIN / -1`, выход за границы массива, использование указателя после `free`, `memcpy` перекрывающихся областей (нужен `memmove`), две модификации объекта без точки следования (`i = i++`).

### Инструмент дня: UBSan

```c
// ub.c
int main(void){ int x = 0x7fffffff; return x + 1; }
```
```
$ clang -fsanitize=undefined -O1 ub.c -o ub && ./ub
ub.c:1:54: runtime error: signed integer overflow: 2147483647 + 1 cannot be represented in type 'int'
```
UBSan — *рантайм*: ловит только то UB, что реально исполнилось на твоих входах. Поэтому он напарник тестов/фаззинга, а не замена чтению кода. Что ловит UBSan, а что ASan/TSan — вопрос 8 самопроверки, держи в голове: **UBSan — логические UB (переполнение, сдвиги, неверные приведения), ASan — память (OOB, use-after-free, утечки), TSan — гонки данных.**

## 3. Целочисленные типы, продвижения и конверсии

Здесь живёт больше системных багов, чем в любой другой теме C. Разберём по слоям.

### 3.1 Ранги и продвижение (integer promotion)

Каждый целочисленный тип имеет **conversion rank** (§6.3.1.1): `bool < char < short < int < long < long long`, со знаковыми/беззнаковыми одного ранга наравне.

**Integer promotion:** в большинстве выражений типы *уже́ `int`* (`char`, `short`, битовые поля, `bool`) автоматически продвигаются до `int` (или `unsigned int`, если `int` не вмещает все значения). Это происходит **до** любой арифметики.

Следствие, которое ломает мозг новичкам:

```c
unsigned char a = 0xFF, b = 0xFF;
unsigned char c = a * b;     // a,b → int; int(255)*int(255)=65025 (влезает в int!); потом усечение до uchar
// арифметика идёт в int, а не в unsigned char — переполнения uchar тут НЕТ
```

И обратное — продвижение делает поведение неинтуитивным для `~`:

```c
uint8_t x = 0;
printf("%d\n", ~x);     // НЕ 255. x → int 0; ~0 == -1 (int). Печатает -1.
if (~x == 0xFF) { ... } // ложно! ~x == -1
```

### 3.2 Usual arithmetic conversions (приведение операндов бинарной операции)

Когда оба операнда уже не уже́ `int`, применяются *usual arithmetic conversions* (§6.3.1.8): операнды приводятся к общему типу по рангу, причём **при равном ранге и разной знаковости побеждает беззнаковый**. Это источник классического бага:

```c
int i = -1;
unsigned u = 0;
if (i < u) puts("i < u"); else puts("i >= u");   // печатает "i >= u" !!!
// i приводится к unsigned: -1 → 4294967295, которое > 0
```

В системном коде это особенно злобно с `size_t` (беззнаковый):

```c
size_t n = some_count();
for (size_t i = n - 1; i >= 0; i--) { ... }   // БЕСКОНЕЧНЫЙ ЦИКЛ: size_t всегда >= 0
```
```c
if (strlen(s) - 1 < some_int) { ... }   // если s пустая: 0-1 = SIZE_MAX, сравнение в size_t
```

**Дисциплина:** `-Wsign-conversion -Wsign-compare`; не смешивай знаковое и беззнаковое в сравнениях; для индексов «назад» используй другой паттерн (`for (size_t i = n; i-- > 0;)`).

### 3.3 Знаковость `char` и сдвиги

- **`char` имеет implementation-defined знаковость.** На x86-Linux — знаковый; на ARM по умолчанию — беззнаковый. Код `char c = getchar(); if (c == EOF)` ломается на платформах со знаковым char для байта 0xFF vs EOF — поэтому `getchar()` возвращает `int`, а не `char`. Для байтов всегда `unsigned char` или `uint8_t`.
- **Сдвиг `>>` отрицательного** — implementation-defined (gcc/clang: арифметический). **Сдвиг на ширину типа и больше — UB** (`uint32_t x; x << 32`). **Сдвиг отрицательным числом — UB.**
- **`x << 31` для `int`**, где результат не влезает в `int`, — UB (до C23 заход в знаковый бит — отдельная тонкость). Для битовых масок используй беззнаковые литералы: `1u << 31`, `1ull << 40`.

### 3.4 Фиксированные типы и форматы

В системном коде — только `<stdint.h>`: `uint32_t`, `int64_t`, `uintptr_t` (целое, в которое влезает указатель), `intmax_t`. Для размеров — `size_t`, для разностей указателей — `ptrdiff_t`, для смещений в файле — `off_t`. Печать через `<inttypes.h>`:

```c
uint64_t v = ...;
printf("%" PRIu64 "\n", v);          // переносимо
printf("%zu\n", sizeof(x));          // size_t
printf("%p\n", (void*)ptr);          // указатель — обязательно (void*)
```

> **Практика 3.A:** напиши `bool in_range_i32(long v)`, возвращающую истину, если `v` помещается в `int32_t`, без UB и без зависимости от ширины `long`. Проверь на `INT32_MIN`, `INT32_MAX`, их соседях.

## 4. Strict aliasing и правило эффективного типа

Самая «невидимая» тема и любимый источник «работало 5 лет, потом обновили gcc и сломалось».

### 4.1 Правило

§6.5p6–7: у каждого объекта в памяти есть **эффективный тип**. Обращаться (читать/писать) к объекту можно только через lvalue **совместимого** типа, с оговорёнными исключениями. Разрешено всегда:

- тип, совпадающий с эффективным (с точностью до `const`/`volatile`/знаковости);
- **`char`-семейство** (`char`, `signed char`, `unsigned char`) — через них можно читать **любой** объект побайтно. Это легальная «лазейка» для сериализации;
- тип агрегата/union, содержащего совместимый член.

Всё остальное — **UB**, и компилятор это эксплуатирует так: «указатели разных, несовместимых типов не указывают на один объект» → можно не перечитывать память, можно переупорядочить.

```c
// Классика: "type punning" через указатель — UB
float bit_hack(float f) {
    uint32_t i = *(uint32_t*)&f;   // читаем float-объект через uint32_t* — НАРУШЕНИЕ
    i |= 0x80000000u;
    return *(float*)&i;            // и обратно
}
```

Почему это реально стреляет:

```c
void scale(float *f, int *i) {     // компилятор: f и i разных типов → не алиасятся
    *i = 5;
    *f = 2.0f;                      // вправе НЕ перечитывать *i после этого
    *i += 1;                        // мог закэшировать *i == 5 → запишет 6, даже если f==(float*)i
}
```

### 4.2 Легальные способы (вопрос 2 самопроверки)

**(1) `memcpy`** — канонический и переносимый. Компилятор его распознаёт и при `-O2` генерирует тот же один `mov`, что и грязный каст, — нулевая цена:

```c
uint32_t f2u(float f) { uint32_t u; memcpy(&u, &f, sizeof u); return u; }
```

**(2) Доступ через `char*`/`unsigned char*`** — разрешён стандартом для побайтной интерпретации (сериализация, хеши, дампы).

**(3) `union` — но осторожно с различием C и C++!**
- **В C** чтение из *другого* члена union, чем тот, что записан («type punning через union»), — **разрешено** и хорошо определено (footnote в §6.5.2.3): значение реинтерпретируется по представлению.
- **В C++** это формально UB (там легален только `memcpy` или `std::bit_cast`). Поэтому привычка из C++ «делай memcpy» — самая переносимая.

```c
// Легально в C:
union { float f; uint32_t u; } pun = { .f = 3.14f };
uint32_t bits = pun.u;
```

**(4) `-fno-strict-aliasing`** — выключить оптимизацию целиком. **Ядро Linux собирается именно с этим флагом** (`Makefile`: `KBUILD_CFLAGS += -fno-strict-aliasing`), потому что низкоуровневый код массово кастует указатели. То есть в коде ядра ты можешь кастовать — но в своём userspace-коде по умолчанию правило действует, и полагаться на «как в ядре» нельзя.

> **Вывод-дисциплина:** в переносимом userspace — `memcpy` для реинтерпретации битов; `char*` для побайтного разбора; помни, что в ядре правило отключено флагом, а не отсутствует.

> **Практика 4.A:** реализуй `fast_inv_sqrt` (легендарный Quake-хак) дважды — грязным кастом и через `memcpy` — и сравни ассемблер на godbolt под `-O2 -fstrict-aliasing`. Убедись, что `memcpy`-версия даёт тот же код и не ломается.

## 5. `restrict` — обещание оптимизатору

`restrict` (§6.7.3.1) — квалификатор указателя (только указателей). Семантика-контракт: **«в течение жизни этого указателя к объекту, на который он указывает, не обращаются ни через какой другой указатель»**. Это *обещание программиста компилятору*, не проверяемое. Нарушил — UB.

Зачем: позволяет компилятору не перечитывать память и векторизовать. Классика — `memcpy` (её прототип `void *memcpy(void *restrict dst, const void *restrict src, size_t n)` — отсюда требование непересечения, иначе бери `memmove`).

```c
// Без restrict компилятор обязан учитывать, что a и b могут пересекаться:
void add(size_t n, float *a, const float *b) {
    for (size_t i = 0; i < n; i++) a[i] += b[i];   // вынужден перечитывать с оглядкой на алиасинг
}
// С restrict — обещаем, что a и b не пересекаются → свободная векторизация:
void add_r(size_t n, float *restrict a, const float *restrict b) {
    for (size_t i = 0; i < n; i++) a[i] += b[i];
}
```

На godbolt `add_r` под `-O3` разворачивается в SIMD, `add` — нет (или с проверкой перекрытия). Это и есть вопрос «чем `restrict` помогает оптимизатору» из самопроверки фундамента: **снимает консервативное предположение об алиасинге**, давая переупорядочивание и векторизацию.

> **Грабли:** если вызвать `add_r(n, p, p)` — ты нарушил обещание, поведение не определено. `restrict` — инструмент горячих циклов, а не «добавить везде для скорости».

## 6. Управление ресурсами без RAII

В ядре нет деструкторов. Каждый ресурс (память, fd, блокировка, refcount) освобождается вручную, на **каждом** пути выхода. Ошибки здесь — №1 источник утечек и use-after-free в системном коде. Разберём идиомы, которые используются в самом ядре.

### 6.1 Идиома «goto cleanup» (single exit, лестница освобождения)

В C++ ты бы положился на стек деструкторов. В C каноничен **обратный goto-разворот**: ресурсы освобождаются в порядке, обратном захвату, через метки.

```c
int do_work(const char *path) {
    int ret = -1;                       // по умолчанию — ошибка
    char *buf = NULL;
    FILE *f = NULL;

    buf = malloc(BUFSZ);
    if (!buf)
        goto out;                       // ещё ничего не захвачено сверх buf
    f = fopen(path, "rb");
    if (!f)
        goto out_free_buf;              // прыжок на нужную ступень

    if (fread(buf, 1, BUFSZ, f) == 0)
        goto out_close;                 // ошибка — но f и buf освободим

    ret = 0;                            // успех

out_close:
    fclose(f);
out_free_buf:
    free(buf);
out:
    return ret;
}
```

Почему так, а не «free на каждом return»:
- **DRY:** освобождение в одном месте, не дублируется по веткам;
- **порядок:** метки выстроены в обратном порядке захвата — корректный teardown;
- **это буквально стиль ядра** (`Documentation/process/coding-style.rst` прямо рекомендует goto для cleanup). Читая исходники драйверов, ты увидишь именно это.

Тонкости:
- `free(NULL)` безопасен (no-op) — поэтому можно инициализировать всё `NULL`/`-1` и иметь единую метку `out`, безопасно «освобождая» незахваченное. Но для fd `close(-1)` вернёт `EBADF` — fd инициализируй `-1` и оборачивай `if (fd >= 0) close(fd);`.
- Не прыгай goto *через* объявление с инициализацией внутри блока (в C это допустимо синтаксически, но переменная будет с indeterminate value).

### 6.2 Дисциплина владения (ownership)

RAII в C++ делает владение типобезопасным. В C владение — **соглашение, записанное в комментарии и имени**:

- Кто аллоцировал — обычно тот и освобождает; если нет — это должно быть в контракте функции явно («caller takes ownership of returned buffer, must free()»).
- Суффиксы-конвенции: `*_create`/`*_destroy`, `*_alloc`/`*_free`, `*_get`/`*_put` (refcount, как `kref` в ядре).
- После `free(p)` — **`p = NULL;`** (защита от double-free и use-after-free; `free(NULL)` безопасен).
- Возвращая владение — документируй. Принимая указатель — документируй, «занимаешь» (borrow) или «забираешь» (move).

### 6.3 Частые дефекты и чем ловить

| Дефект | Симптом | Инструмент |
|---|---|---|
| Утечка (не дошли до `free`) | растёт RSS демона | ASan (`LeakSanitizer`), valgrind `--leak-check=full` |
| Double-free | краш/коррупция кучи | ASan |
| Use-after-free | «иногда» неверные данные/краш | ASan |
| Использование после `realloc` старого указателя | UAF (realloc мог переместить) | ASan; правило: `p = realloc(p, n)` только во временную, проверь, потом присвой |
| `realloc(p, 0)` | impl-defined (может вернуть NULL ≠ ошибка) | не используй для «освободить» |
| Неосвобождённый fd | `EMFILE` под нагрузкой | strace, `/proc/<pid>/fd`, статический анализ |

```c
// realloc без потери буфера при ошибке:
char *tmp = realloc(buf, newsz);
if (!tmp) { /* buf ещё валиден, не потерян */ goto err; }
buf = tmp;
```

> **Практика 6.A:** напиши `char *read_whole_file(const char *path, size_t *out_len)` с goto-cleanup: открыть, узнать размер, выделить, прочитать, на любой ошибке вернуть NULL без утечек. Прогони под `-fsanitize=address` и valgrind на: несуществующем файле, пустом файле, файле > доступной памяти (замокай malloc-фейл). Это прямой задел под Ф2 (файловый I/O) и капстоун.

## 7. Системный C: то, чего нет в «учебном» C

Короткий, но обязательный набор — это встречается в каждом заголовке ядра.

**`const` vs `volatile`.** `const` — обещание «я не меняю». `volatile` — «значение может измениться вне видимости компилятора, не кэшируй и не выкидывай обращения» (MMIO-регистры, переменная, меняемая обработчиком сигнала). **`volatile` ≠ атомарность и ≠ барьер памяти** — частое заблуждение из embedded; в многопоточности нужны `_Atomic`/барьеры (это C1). В ядре для MMIO — не голый `volatile`, а `readl`/`writel`.

**`sig_atomic_t` / `volatile sig_atomic_t`** — единственный тип, который безопасно трогать из обработчика сигнала (Ф/C4). 

**Выравнивание и layout структур (§6.7.2.1, §6.2.8).** Компилятор вставляет padding для выравнивания членов. Это влияет на `sizeof`, на бинарную совместимость (протоколы, сериализация — никогда не `fwrite(&struct)` без `__attribute__((packed))` или ручной сериализации), на false sharing (C1/C6).

```c
struct bad  { char a; int b; char c; };   // sizeof обычно 12 (паддинг)
struct good { int b; char a; char c; };   // sizeof обычно 8 (упаковано плотнее)
```
`_Alignof(T)`, `alignas`/`_Alignas`, `offsetof(struct, member)` — твои инструменты. `container_of` ядра (вопрос K1) построен ровно на `offsetof`.

**Flexible array member** — «хвост переменной длины», паттерн ядра:
```c
struct msg { uint32_t len; char data[]; };          // C99 FAM
struct msg *m = malloc(sizeof *m + payload_len);     // аллокация с хвостом
```

**Полезный современный C (C11/C23), часто в системном коде:**
- `_Generic` — перегрузка по типу (на нём построены типобезопасные макросы);
- designated initializers `{ .field = x }` — массово в ядре для таблиц `file_operations` и т.п.;
- compound literals `(struct foo){ ... }`;
- `_Static_assert(sizeof(struct hdr) == 8, "ABI!")` — проверка ABI на этапе компиляции;
- `__attribute__((packed, aligned, cleanup))` — GNU-расширения. **`cleanup`** даёт почти-RAII в GCC/Clang и используется в новом ядре (`__free`, `guard()` из `cleanup.h`):
```c
static inline void freep(void *p){ free(*(void**)p); }
#define _autofree __attribute__((cleanup(freep)))
void f(void){ _autofree char *buf = malloc(100); /* free автоматически на выходе */ }
```

> **Практика 7.A:** объяви протокольный заголовок пакета фиксированного бинарного формата (magic, version, len, flags). Подтверди `sizeof` через `_Static_assert`, выведи offset каждого поля через `offsetof`, объясни весь padding. Сериализуй через `memcpy` в `unsigned char buf[]`, не через каст структуры.

## 8. Указатели — глубокое погружение

### 8.1 Что такое указатель на уровне машины

Указатель — это целое число, представляющее адрес байта в виртуальном адресном пространстве процесса. На x86-64 — 64-битное, `sizeof(void*) == 8` на любой платформе с одним адресным пространством. Операция разыменования (`*p`) транслируется в инструкцию обращения к памяти (`mov` в x86 ассемблере): процессор идёт по этому адресу и читает/пишет `sizeof(*p)` байт.

```c
int x = 42;
int *p = &x;          // p хранит адрес x (например, 0x7ffd1234)
*p = 99;              // mov [rax], 99  — запись по адресу
int y = *p;           // mov rax, [rax] — чтение по адресу
printf("%p\n", (void*)p);      // %p требует void*; всегда кастуй явно
printf("%zu\n", sizeof(p));    // всегда 8 на x86-64
printf("%zu\n", sizeof(*p));   // sizeof(int) = 4
```

**Указатель ≠ массив, но имя массива «разлагается» в указатель на первый элемент** при передаче в функцию или в большинстве выражений («array decay»). Исключения: `sizeof(arr)`, `&arr` — работают с массивом как с целым. (Осторожно: стандартный `_Alignof` принимает **только имя типа** в скобках, не выражение — `_Alignof(arr)` для объекта невалиден; с выражением работает лишь GNU-расширение `__alignof__`.) Это источник классического заблуждения «массив = указатель».

### 8.2 Арифметика указателей — точные правила

Арифметика `p + n` сдвигает указатель на `n * sizeof(*p)` байт. Это и есть смысл «типизированного» указателя: компилятор знает шаг.

```c
int arr[5] = {0, 1, 2, 3, 4};
int *p = arr;
p + 1;   // адрес arr[1] = адрес arr[0] + 4 байта (sizeof int)
p + 3;   // адрес arr[3]
```

**Точные правила (§6.5.6):**

1. Арифметика указателей определена только внутри одного объекта (массива или памяти, выделенной `malloc`). Выход за пределы объекта — UB, даже если не разыменовываешь.
2. **«Один за последним» легален для сравнения.** `p == arr + 5` (указатель за последним элементом `arr[4]`) — допустимое сравнение, но разыменовывать `*(arr + 5)` — UB.
3. `p - q` возвращает `ptrdiff_t` (знаковый) — количество элементов между указателями. Оба должны указывать в один и тот же объект.

```c
// Корректный обратный проход:
for (int *p = arr + 5; p != arr; ) {
    --p;
    printf("%d\n", *p);
}

// UB: выход за начало массива
int *bad = arr - 1;    // UB! (не "один за последним" спереди — это исключение только для конца)
```

Разность `char*`-указателей на разные объекты — UB (даже если они случайно рядом в памяти). Компилятор использует это при оптимизации.

### 8.3 void* — беззнаковый переносчик адреса

`void*` — указатель на объект неизвестного типа. **Нет арифметики** (нет `sizeof(*void_ptr)` — неизвестен шаг), **нет разыменования**. Неявно конвертируется в/из любого объектного указателя (в C; в C++ — нет, нужен явный каст).

Применения в стандартной библиотеке:

```c
// malloc: не знает тип — возвращает void*
int *p = malloc(n * sizeof *p);   // в C: каст не нужен и даже нежелателен
                                   // в C++: каст обязателен

// qsort: компаратор работает через void*
int cmp_int(const void *a, const void *b) {
    int x = *(const int*)a;    // каст обязателен перед разыменованием
    int y = *(const int*)b;
    return (x > y) - (x < y); // без переполнения (в отличие от x - y!)
}
qsort(arr, n, sizeof *arr, cmp_int);

// bsearch: то же
int key = 3;
int *found = bsearch(&key, arr, n, sizeof *arr, cmp_int);
```

Для обобщённой функции, принимающей данные произвольного типа, паттерн «void* + size + function pointer» — это вся дженерик-система C. Тип — только у программиста в голове.

### 8.4 const — четыре позиции

`const` сбивает с толку именно потому, что значащих позиций несколько, и они читаются по-разному. Правило: **читай объявление справа налево**.

```c
const int *p;          // "p — указатель на const int":  нельзя менять *p, можно менять p
int * const p;         // "p — const указатель на int":  можно менять *p, нельзя менять p
const int * const p;   // нельзя ни *p, ни p менять
int **pp;              // указатель на указатель на int
const int **pp;        // указатель на указатель на const int

// Важно: int** и const int** НЕ взаимозаменяемы!
void f(const int **pp); // принимает указатель на (указатель на const int)
int *p;
f(&p);  // WARNING: несовместимые типы — &p имеет тип int**, не const int**
```

Почему `int**` ≠ `const int**` — это не очевидно. Предположим, что конверсия была бы разрешена:

```c
const int val = 42;
int *q;
const int **cpp = (const int**)&q;  // если бы разрешили...
*cpp = &val;                         // теперь q указывает на const val
*q = 0;                              // ПИШЕМ в const объект — UB!
```

Компилятор запрещает такую конверсию именно чтобы предотвратить это. Разрешены только преобразования «добавить const» ближе к данным: `int*` → `const int*` — ОК; `int**` → `const int* const*` — ОК; `int**` → `const int**` — нет.

### 8.5 Указатели на функцию

Функция — это тоже объект в памяти с адресом. Указатель на функцию хранит этот адрес и позволяет вызвать её косвенно. Применение: обратные вызовы (callbacks), dispatch tables, имитация виртуальных функций.

```c
// Синтаксис объявления: тип_возврата (*имя)(типы_параметров)
int (*fn)(int, int);    // fn — указатель на функцию (int, int) → int

// typedef делает код читаемым:
typedef int (*binary_op)(int, int);
binary_op op = NULL;

int add(int a, int b) { return a + b; }
int mul(int a, int b) { return a * b; }

op = add;
printf("%d\n", op(3, 4));  // 7 — косвенный вызов

// Массив функций-обработчиков (dispatch table):
typedef void (*handler_t)(int event_data);

void on_connect(int fd)    { /* ... */ }
void on_disconnect(int fd) { /* ... */ }
void on_data(int len)      { /* ... */ }

static const handler_t handlers[] = {
    [0] = on_connect,
    [1] = on_disconnect,
    [2] = on_data,
};
#define HANDLER_COUNT (sizeof handlers / sizeof handlers[0])

void dispatch_event(int type, int data) {
    if ((size_t)type < HANDLER_COUNT && handlers[type])
        handlers[type](data);
}
```

Это эквивалент vtable C++: в C++ `virtual` функции реализованы именно через такой массив указателей в памяти, только созданный компилятором автоматически. `container_of` + dispatch table = самодельный полиморфизм в ядре Linux (`struct file_operations`, `struct net_device_ops` и т.д.).

**Ещё один нюанс:** указатель на функцию **нельзя** конвертировать в `void*` и обратно в переносимом C — стандарт такую конверсию просто не определяет (объектные и функциональные указатели — разные миры; §6.3.2.3 покрывает только объектные). На POSIX-системах (Linux) это разрешено (POSIX явно требует), поэтому `dlsym()` возвращает `void*`. Но технически — расширение, а не гарантия стандарта C.

### 8.6 Нулевой указатель — NULL

NULL — это **null pointer constant**: целочисленная константа, равная нулю (или `(void*)0`). Стандарт гарантирует: при конверсии в указательный тип получается «нулевой указатель», отличный от указателя на любой объект.

**На x86 нулевой указатель имеет нулевое битовое представление.** Но стандарт этого не требует — на некоторых встраиваемых платформах нулевой указатель имеет другое представление. Поэтому нельзя делать:

```c
memset(&ptr, 0, sizeof ptr);  // обнуляет биты, не гарантирует null pointer на всех платформах
// Правильно:
ptr = NULL;                    // стандартная null pointer constant
```

**Разыменование NULL — UB.** Но важнее: **компилятор использует UB разыменования NULL как информацию об оптимизации:**

```c
int *p = get_ptr();
*p = 0;           // разыменовываем p
if (p == NULL)    // если мы дошли до этой строки — p уже был не-NULL
    handle_null(); // компилятор удаляет эту ветку!
```

Это был реальный CVE: тот, кто писал код, думал «проверяю после использования для безопасности», но компилятор знает, что разыменование прошло успешно → p был не-NULL → условие всегда ложно → ветка удалена. Всегда проверяй `!= NULL` **до** разыменования.

> **Практика 8.A:** напиши безопасный `array_get(int *arr, size_t len, size_t idx, int *out)`, возвращающий 0 при успехе и -1 при выходе за границы. Реализуй dispatch table с 4 операциями и событийным циклом.

## 9. Структуры, union, битовые поля

### 9.1 Выравнивание и padding — детально

Каждый тип имеет **требование выравнивания** (`_Alignof(T)` = n, адрес объекта должен быть кратен n). Правила GCC для x86-64:

| Тип | sizeof | _Alignof |
|-----|--------|----------|
| `char` | 1 | 1 |
| `short` | 2 | 2 |
| `int` | 4 | 4 |
| `long` | 8 | 8 |
| `long long` | 8 | 8 |
| `float` | 4 | 4 |
| `double` | 8 | 8 |
| `void*` | 8 | 8 |

**Правила компоновки структуры:**
1. Каждый член размещается по адресу, кратному его `_Alignof`.
2. Перед членом вставляется **padding** (байты-заглушки без значения) для выполнения п.1.
3. В конце структуры добавляется **trailing padding** так, чтобы `sizeof(struct)` было кратно `_Alignof` всей структуры (= максимальному `_Alignof` среди всех членов).

```c
struct bad {
    char   a;  // offset 0,  size 1
               // pad  3    (выравнивание int=4)
    int    b;  // offset 4,  size 4
    char   c;  // offset 8,  size 1
               // pad  7    (выравнивание double=8)
    double d;  // offset 16, size 8
    short  e;  // offset 24, size 2
               // pad  6    (trailing padding, выравнивание struct=8)
};  // sizeof = 32

struct good {   // после оптимальной перестановки: сортировка по убыванию _Alignof
    double d;  // offset 0,  size 8
    int    b;  // offset 8,  size 4
    short  e;  // offset 12, size 2
    char   a;  // offset 14, size 1
    char   c;  // offset 15, size 1
               // trailing pad: 0  (sizeof=16, кратно 8)
};  // sizeof = 16 — вдвое меньше!
```

Разница 2× важна для массивов структур: миллион `struct bad` занимает 32 МБ, миллион `struct good` — 16 МБ.

**Почему нельзя `memcmp` структуры с padding:**
padding-байты имеют неопределённое значение. `memcmp(&s1, &s2, sizeof s1)` сравнивает байты включая padding — структуры с одинаковыми полями могут сравниться как «разные». Для сравнения структур нужно либо `memset(&s, 0, sizeof s)` до заполнения (обнуляет padding), либо поле-за-полем.

### 9.2 `__attribute__((packed))` — когда нужно и когда нет

```c
struct __attribute__((packed)) net_header {
    uint8_t  version;
    uint8_t  ihl;
    uint16_t total_len;   // не выровнен на 2 — может быть нечётным адресе
    uint32_t seq_num;
};  // sizeof = 8, без padding — точно соответствует бинарному формату
```

**Когда нужно:** сетевые пакеты, бинарные форматы (ELF заголовки, PE заголовки, флеш-прошивки), где байтовый макет фиксирован внешним стандартом. `_Static_assert(sizeof(struct net_header) == 8, "ABI!")` обязателен.

**Цена:** на x86 процессор обрабатывает невыровненный доступ аппаратно (медленнее, но работает). На ARM невыровненный доступ может вызвать SIGBUS или быть принципиально другим по скорости. GCC при `-O2` может генерировать несколько инструкций вместо одной для невыровненного `uint32_t`. **Поэтому:** `packed` — только для структур протоколов, где чтение/запись идёт через `memcpy` или явный побайтный разбор, не через прямые обращения к членам.

### 9.3 Битовые поля

```c
struct flags {
    unsigned int read    : 1;
    unsigned int write   : 1;
    unsigned int execute : 1;
    unsigned int _pad    : 5;  // явный padding для ясности
};  // sizeof обычно 4 (один int)
```

**Что определено стандартом:** наличие битовых полей, диапазон значений члена (2^n для unsigned шириной n).

**Что implementation-defined:** порядок битов внутри storage unit (little-bit или big-bit), через сколько storage unit может «перетекать» поле, выравнивание storage unit.

**Поэтому:** битовые поля — **не для сетевых протоколов**. Формат битов на разных компиляторах/платформах различается. Для протоколов — `uint8_t` + ручные сдвиги и маски.

**Хорошее применение:** флаги внутри одной единицы компиляции, где переносимость формата в память не нужна (разные ОС-флаги в структурах ядра, флаги состояния в парсере).

### 9.4 union для type punning и tagged union

В **C** `union` с чтением «не того» члена — определённое поведение (§6.5.2.3 footnote 95 в C11): биты реинтерпретируются. В **C++** — UB (там только `memcpy` или `std::bit_cast`).

```c
// Type punning в C:
union float_bits {
    float    f;
    uint32_t u;
};
uint32_t get_bits(float x) {
    union float_bits pun;
    pun.f = x;
    return pun.u;   // легально в C, UB в C++
}
```

**Tagged union (discriminated union)** — паттерн для AST-нод, токенов, вариантных типов:

```c
typedef enum { TOK_INT, TOK_FLOAT, TOK_IDENT, TOK_OP } token_type;

typedef struct {
    token_type type;     // тег — определяет, какой член union активен
    union {
        long        int_val;
        double      float_val;
        char       *ident;    // для TOK_IDENT: строка
        char        op;       // для TOK_OP: символ операции
    } val;
} token_t;

// Использование:
void print_token(const token_t *t) {
    switch (t->type) {
    case TOK_INT:   printf("INT(%ld)\n", t->val.int_val); break;
    case TOK_FLOAT: printf("FLT(%g)\n",  t->val.float_val); break;
    case TOK_IDENT: printf("ID(%s)\n",   t->val.ident);   break;
    case TOK_OP:    printf("OP(%c)\n",   t->val.op);      break;
    }
}
```

**Размер union** = размер наибольшего члена (плюс возможный trailing padding для выравнивания).

### 9.5 Flexible Array Member (FAM)

```c
struct message {
    uint32_t type;
    uint32_t len;
    char     data[];   // FAM: только последний член, только один в struct
};
```

FAM — массив без указанного размера в конце структуры. Правила (§6.7.2.1):
- Только последний член структуры.
- Структура должна иметь хотя бы один другой член.
- `sizeof(struct message)` не включает FAM (как будто его нет).
- Инициализация структуры с FAM через `= { ... }` — без FAM-части.

Аллокация с хвостом одним `malloc`:

```c
struct message *msg_alloc(uint32_t type, const char *data, size_t len) {
    // sizeof(struct message) + len + 1 — один malloc для заголовка и данных
    struct message *m = malloc(sizeof *m + len + 1);
    if (!m) return NULL;
    m->type = type;
    m->len  = (uint32_t)len;
    memcpy(m->data, data, len);
    m->data[len] = '\0';
    return m;
}
```

Это паттерн `struct iov`, `struct cmsghdr`, `struct sk_buff` в ядре — заголовок и данные как единый блок памяти = нет дополнительного указателя, нет второго `malloc`, нет второго `free`, лучше кэш-локальность.

### 9.6 `container_of` — как устроен макрос ядра

```c
// include/linux/container_of.h (упрощённо):
#define container_of(ptr, type, member) \
    ((type *)((char *)(ptr) - offsetof(type, member)))
```

Идея: если знаем адрес члена `member` в структуре `type`, то адрес самой структуры = адрес члена − смещение члена.

```c
struct list_head {
    struct list_head *next;
    struct list_head *prev;
};

struct task_struct {
    pid_t             pid;
    char              comm[16];
    struct list_head  tasks;    // встроенный узел списка
    // ...
};

// Итерация по ядровому связному списку:
struct list_head *pos;
list_for_each(pos, &task_list) {
    // pos указывает на member tasks, а не на начало task_struct
    struct task_struct *task = container_of(pos, struct task_struct, tasks);
    printf("pid=%d name=%s\n", task->pid, task->comm);
}
```

Это обобщённый связный список без `void*` и без потери типобезопасности — структура данных «встроена» в объект, а не объект содержит указатель на «узел».

> **Практика 9.A:** спроектируй `token_t` tagged union для JSON-лексера (null, bool, number, string, символы `{[]},:`) с правильными размерами. Реализуй `container_of` и проверь на `struct list_item` с embedded `struct node`.

## 10. Строки в C — половина всех CVE

### 10.1 Модель строки: NTBS

**Null-Terminated Byte String (NTBS)** — последовательность байт, заканчивающаяся байтом `'\0'` (нулевым терминатором). Длина строки неизвестна без её прохода (`strlen` = O(n)). В буфере должен быть место для терминатора: строка "hello" требует 6 байт, не 5.

Почему это небезопасно по умолчанию: **большинство стандартных функций не принимают размер буфера как параметр**. Они идут до `'\0'` и пишут/читают дальше, если его нет или буфер мал. Это — половина всех уязвимостей переполнения буфера за историю C.

### 10.2 Опасные функции и замены

| Опасная | Проблема | Замена |
|---------|----------|--------|
| `gets(buf)` | читает без ограничения длины | `fgets(buf, size, stdin)` |
| `strcpy(dst, src)` | пишет без ограничения длины | `strncpy` (осторожно!) или `strlcpy` (POSIX 2024) |
| `strcat(dst, src)` | не проверяет dst размер | `strncat` или ручная проверка |
| `sprintf(buf, fmt, ...)` | переполняет `buf` | `snprintf(buf, size, fmt, ...)` |
| `scanf("%s", buf)` | читает без ограничения | `scanf("%255s", buf)` с явным размером |

**`strncpy` — коварная замена:** `strncpy(dst, src, n)` ГАРАНТИРУЕТ записать ровно `n` байт в `dst`. Если `src` короче `n` — остаток заполняется нулями (это не баг, это feature). Если `src` длиннее `n` — **нулевой терминатор не записывается**! Нужно вручную: `dst[n-1] = '\0';`. Многие разработчики об этом забывают.

```c
// Безопасное копирование — явная гарантия терминатора:
void safe_copy(char *dst, size_t dst_size, const char *src) {
    if (dst_size == 0) return;
    strncpy(dst, src, dst_size - 1);
    dst[dst_size - 1] = '\0';   // всегда!
}
```

### 10.3 Переполнение буфера — механизм

На x86 стековый фрейм выглядит так (упрощённо, стек растёт вниз):

```
высокие адреса:
  [аргументы]
  [return address]  ← перезапись → arbitrary code execution
  [saved rbp]
  [локальные переменные]  ← buf[128]
низкие адреса
```

При переполнении `buf[128]` запись идёт вверх по адресам и перезаписывает `return address`. При возврате из функции процессор прыгает на адрес, управляемый атакующим.

**Защиты (слои):**
- **Stack canary** (`-fstack-protector`): перед `return address` компилятор вставляет случайное значение («канарейку»); перед возвратом проверяет, не изменилась ли. Переполнение буфера перезапишет канарейку → детект.
- **ASLR** (Address Space Layout Randomization): базовые адреса стека/кучи/библиотек случайны → атакующий не знает, куда прыгать.
- **NX/DEP** (No-Execute): страницы стека и кучи помечены как не-исполняемые → shellcode там не выполняется.
- **ASan** (`-fsanitize=address`): при каждой записи/чтении проверяет «shadow memory» — ловит выход за пределы буфера немедленно.

### 10.4 Off-by-one — классика

```c
// Ошибка 1: забытый нулевой терминатор
char buf[5];
const char *src = "hello";
memcpy(buf, src, strlen(src));   // копирует 5 байт, нет места для '\0'!
// buf не является корректной C-строкой

// Правильно:
memcpy(buf, src, strlen(src) + 1);  // +1 для '\0'
// или:
snprintf(buf, sizeof buf, "%s", src);

// Ошибка 2: цикл пишет на buf[n]
char buf[10];
for (int i = 0; i <= 10; i++)  // i <= 10 → пишет buf[10] — выход за пределы!
    buf[i] = 'x';
// Правильно: i < 10

// Ошибка 3: strlen в условии массива
size_t len = strlen(s);
char *copy = malloc(len);    // не хватает места для '\0'!
// Правильно: malloc(len + 1)
```

Off-by-one — частая причина CVE именно потому, что «работает» на большинстве входных данных и проявляется только при максимальной длине.

### 10.5 strtol — правильный error-checking

Стандартная `strtol` имеет нетривиальный контракт. Полностью правильная проверка:

```c
#include <errno.h>
#include <stdlib.h>
#include <limits.h>

int parse_long(const char *s, long *out) {
    if (!s || !*s) return -1;   // пустая строка
    char *endptr;
    errno = 0;                   // сбрасываем перед вызовом!
    long val = strtol(s, &endptr, 10);

    if (errno == ERANGE)         // переполнение (LONG_MIN или LONG_MAX)
        return -1;
    if (endptr == s)             // ни одного символа не распознано
        return -1;
    if (*endptr != '\0')         // trailing garbage: "42abc"
        return -1;

    *out = val;
    return 0;
}
```

Почему `errno = 0` до вызова: стандарт гарантирует, что `strtol` устанавливает `errno = ERANGE` при переполнении, но не гарантирует, что сбрасывает `errno` при успехе. Если `errno` уже содержит ненулевое значение от предыдущей операции — ложное срабатывание.

**`strtok` не реентерабельна:** хранит состояние в `static` переменной. В многопоточном коде и при рекурсии — используй `strtok_r` (POSIX) или `strtok_s` (C11 Annex K).

```c
// НЕ реентерабельно:
char *tok = strtok(str, " ");     // сохраняет static ptr в str

// Реентерабельно:
char *saveptr;
char *tok = strtok_r(str, " ", &saveptr);  // контекст в saveptr
```

### 10.6 Locale-зависимые функции

`toupper`, `tolower`, `isalpha`, `isdigit` — зависят от текущего locale (`LC_CTYPE`). В русской локали `isalpha('А')` может вернуть истину; `toupper('a')` в локали с многобайтной кодировкой — непредсказуемо.

Для протоколов, где нужен ASCII: используй явные сравнения или вспомогательные функции:

```c
// Вместо toupper(c) для ASCII:
static char ascii_upper(char c) {
    return (c >= 'a' && c <= 'z') ? (char)(c - 'a' + 'A') : c;
}

// Вместо isdigit(c) — locale-зависимой:
static int is_ascii_digit(char c) {
    return c >= '0' && c <= '9';
}
```

### 10.7 Форматные строки — format string attack

```c
// УЯЗВИМОСТЬ — если user_input содержит "%n" или "%s%s%s%s%s":
printf(user_input);           // НИКОГДА ТАК!

// Правильно:
printf("%s", user_input);     // формат-строка — литерал, не пользовательские данные
// или:
fputs(user_input, stdout);    // если просто вывод строки
```

`%n` записывает количество выведенных символов по адресу из va_list. Атакующий, контролирующий формат-строку, может:
1. Читать со стека через `%x %x %x %x`.
2. Писать произвольные значения по произвольным адресам через `%n`.

Это был CVE в `wu-ftpd`, `syslog` и множестве других программ в 2000-х. Современный GCC с `-Wformat` и `-Wformat-security` предупреждает о `printf(user_input)`.

### 10.8 Инструменты

**AddressSanitizer (ASan)** — инструментирует код теневой памятью: каждый байт приложения имеет 1 бит тени, отмечающий валидность. При каждом доступе проверка. Ловит: OOB (out-of-bounds read/write), UAF (use-after-free), double-free, stack overflow. Накладные расходы: ~2× по памяти, ~1.5–2× по времени.

```
$ gcc -fsanitize=address -g prog.c -o prog && ./prog
=================================================================
==1234==ERROR: AddressSanitizer: stack-buffer-overflow on address 0x7fff...
READ of size 1 at 0x7fff... thread T0
    #0 0x... in main prog.c:15
```

**Valgrind** — эмулирует выполнение, отслеживает все операции с памятью. Медленнее ASan (~10–20×), но не требует перекомпиляции, ловит больше классов ошибок (uninitialised reads, fd leaks). Для production-бинарей без перекомпиляции.

**UBSan** — ловит UB в строках: `strlen(non_null_terminated)` — не ловит (нет UB в самом strlen, пока не выйдем за объект), но ловит: переполнение при `malloc(strlen(s) + 1)` если `strlen(s) == SIZE_MAX`.

> **Практика 10.A:** напиши `safe_string_copy(char *dst, size_t dst_size, const char *src)` с явным контрактом (возврат 0/−1, гарантия терминатора), тесты на: пустую строку, точно dst_size−1 символов, dst_size символов (усечение), NULL-терминатор в итоге.

## 11. Препроцессор — гигиена макросов

### 11.1 Правила безопасных макросов

Макросы — это текстовая подстановка, не функции. Ловушки:

```c
// ПЛОХО: нет скобок вокруг параметров
#define DOUBLE(x) x * 2
DOUBLE(1 + 2)  →  1 + 2 * 2  =  5  (не 6!)

// ПЛОХО: нет скобок вокруг всего выражения
#define ADD(a, b) a + b
int x = ADD(1, 2) * 3;  →  1 + 2 * 3  =  7  (не 9!)

// ХОРОШО:
#define DOUBLE(x) ((x) * 2)
#define ADD(a, b) ((a) + (b))

// ПЛОХО: двойное вычисление аргументов
#define MAX(a, b) ((a) > (b) ? (a) : (b))
MAX(i++, j++)   // i++ и j++ вычисляются дважды!
                // i инкрементируется 1 или 2 раза в зависимости от сравнения

// ПЛОХО: отсутствие do...while(0) для многострочных макросов
#define SWAP(a, b) \
    int _tmp = (a); \
    (a) = (b); \
    (b) = _tmp;
if (cond) SWAP(x, y); else do_something();
// Раскрывается: if (cond) int _tmp = x; x = y; y = _tmp; else ...
// — только первая строка под if!

// ХОРОШО: do { ... } while(0)
#define SWAP(type, a, b) do { \
    type _swap_tmp = (a); \
    (a) = (b); \
    (b) = _swap_tmp; \
} while(0)
```

### 11.2 Макрос vs inline function

В C99+ **предпочитай `static inline` функцию** вместо макроса:

```c
// Макрос: нет проверки типов, двойное вычисление
#define MAX(a, b) ((a) > (b) ? (a) : (b))

// static inline: проверка типов, аргументы вычисляются один раз
static inline int max_int(int a, int b) { return a > b ? a : b; }
```

Компилятор подставляет `inline`-функцию по месту вызова так же, как макрос — накладных расходов нет. Но: типобезопасность, отладчик видит имя функции, нет двойного вычисления, нет захвата переменных из контекста вызова.

**Когда макрос незаменим:**
- `stringify` (`#x` превращает токен в строку-литерал);
- конкатенация токенов (`##`);
- compile-time типонезависимость через `_Generic`;
- многострочный «шаблон» для генерации кода (X-макросы).

### 11.3 X-макросы — генерация кода

Паттерн: список данных определяется один раз через макрос `X(...)`, потом один и тот же список используется для генерации разных вещей (enum, строковые таблицы, обработчики).

```c
// Список системных вызовов (одно место — не рассинхронизируется)
#define SYSCALL_LIST(X) \
    X(read,  0) \
    X(write, 1) \
    X(open,  2) \
    X(close, 3)

// Генерация enum:
typedef enum {
#define X_ENUM(name, num) SYS_##name = num,
    SYSCALL_LIST(X_ENUM)
#undef X_ENUM
} syscall_t;

// Генерация строковой таблицы:
static const char *syscall_names[] = {
#define X_NAME(name, num) [num] = #name,
    SYSCALL_LIST(X_NAME)
#undef X_NAME
};

// Теперь: syscall_names[SYS_read] == "read"
// Добавить новый syscall — только в SYSCALL_LIST, всё остальное обновится автоматически.
```

Этот же паттерн используется в ядре Linux для таблиц системных вызовов (`arch/x86/entry/syscalls/syscall_64.tbl`), в компиляторах для таблиц opcodes.

### 11.4 `_Generic` — безопасная перегрузка по типу

```c
// tgmath.h-стиль: функция "abs" для разных типов
#define abs_generic(x) _Generic((x), \
    int:        abs(x),       \
    long:       labs(x),      \
    long long:  llabs(x),     \
    float:      fabsf(x),     \
    double:     fabs(x),      \
    default:    fabs(x)       \
)

abs_generic(3)    // вызывает abs(3)
abs_generic(3LL)  // вызывает llabs(3LL)
abs_generic(3.0)  // вызывает fabs(3.0)
```

`_Generic` вычисляет **тип** выражения на этапе компиляции и выбирает соответствующую ветку. Ветки «не выбранные» не компилируются (compile-time selection, не runtime). Это позволяет писать типонезависимые макросы без потери типобезопасности.

### 11.5 Условная компиляция

```c
// Include guards — традиционный способ:
#ifndef MYHEADER_H
#define MYHEADER_H
// ... тело заголовка ...
#endif

// #pragma once — более краткий, но не стандарт C (GCC/Clang поддерживают):
#pragma once

// Feature test macros — включить расширенный POSIX API:
#define _GNU_SOURCE      // GNU-расширения (strdupa, asprintf, ...)
```

## 12. Реальные CVE — UB и типы в боевом коде

Восемь коротких разборов: код + механизм + санитайзер.

### CVE-2009-1897: Linux tun driver — NULL dereference → компилятор удалил проверку

```c
// Упрощённо (drivers/net/tun.c до патча):
static unsigned int tun_chr_poll(struct file *file, poll_table *wait) {
    struct tun_file *tfile = file->private_data;
    struct tun_struct *tun = __tun_get(tfile); // tun может оказаться NULL
    struct sock *sk = tun->sk;             // (1) разыменовываем tun (tun->sk)
    // ...
    if (!tun)                              // (2) проверяем tun после разыменования
        return POLLERR;
    // ... работаем с sk
}
```

Компилятор: «раз мы разыменовали `tun` в (1) без UB — значит `tun != NULL`. Значит проверка (2) всегда ложна». Ветка `return POLLERR` удалена. Атакующий мог создать ситуацию, где `tun == NULL`, и достичь `tun->socket.sk` с нулевым указателем → EoP.

**Тип UB:** разыменование NULL-указателя.
**Что поймало бы:** UBSan с `-fsanitize=null`; KASan в ядре.

#### Почему компилятор «прав», а segfault не происходит

Интуиция «разыменовал NULL → упал с segfault» неверна сразу по двум причинам.

**1. Разыменование NULL — это UB, а не «всегда segfault».** Стандарт C не говорит «программа упадёт». Он говорит: *поведение не определено вообще*. А раз так, компилятор по определению считает, что UB **не происходит** — он оптимизирует в предположении, что программист никогда не пишет программу, которая его вызывает. Дальше работает простая логическая цепочка:

```
(1) выполнилось `tun->sk`          → значит tun не NULL (иначе было бы UB, а UB не бывает)
(2) `if (!tun)`                     → но мы только что доказали, что tun != NULL
(3) следовательно `!tun` == false  → ветка return POLLERR недостижима (мёртвый код)
(4) выкидываем проверку целиком
```

Компилятор формально «прав» относительно стандарта: он сделал валидное преобразование. Виноват код, который сначала разыменовал указатель, а потом стал его проверять — он пообещал компилятору (фактом разыменования), что NULL быть не может, и тут же сам это обещание нарушил.

**2. Почему не падает на самой строке (1)?** Потому что segfault при доступе к адресу 0 — это поведение *операционной системы*, а не закон природы. Адрес `0` обычно не отображён в память, поэтому процессор и ловит исключение. Но:

- На старых ядрах Linux нижние адреса можно было отобразить из пользовательского пространства (`mmap` с `MAP_FIXED` по адресу `0`, при `mmap_min_addr == 0`). Тогда чтение по «нулевому» адресу **не** вызывает fault — оно читает память, которую заранее подготовил атакующий.
- `tun->sk` — это не доступ строго по адресу 0, а по адресу `0 + offsetof(struct tun_struct, sk)`, то есть `NULL + небольшое_смещение`. Атакующему достаточно, чтобы оказалась отображена страница, покрывающая это смещение.

**Полная цепочка эксплойта:**

1. Атакующий через `mmap` отображает страницу по адресу `0` и кладёт туда подконтрольные данные (поддельный `struct tun_struct`).
2. Вызывает `poll()` на устройстве в состоянии, когда `tun == NULL`.
3. Строка (1) `tun->sk` читает атакующую страницу — **fault не возникает**, выполнение продолжается.
4. Проверки `if (!tun) return POLLERR` в бинарнике **нет** — компилятор её удалил.
5. Дальше ядро работает с полями `tun->...`, значения которых задал атакующий → подмена указателей, запись в произвольную память → повышение привилегий (EoP, Elevation of Privilege — повышение привилегий).

**Мораль:** опасность не в том, что «программа упадёт», а в том, что она **не упадёт** — UB позволил компилятору удалить защитную проверку, а отображённая нулевая страница позволила разыменованию «успешно» прочитать чужие данные. Правильный код всегда проверяет указатель **до** первого разыменования.

**Как чинили (факты из NVD/LWN):** уязвимость затронула ядра Linux **2.6.30 и 2.6.30.1**. Помимо исправления самого кода (проверка до разыменования), в сборку ядра добавили флаг GCC **`-fno-delete-null-pointer-checks`** — он запрещает компилятору выкидывать проверки на NULL на основании более раннего разыменования (в коде ядра «разыменование NULL → oops», а не UB-в-вакууме, поэтому такая оптимизация для ядра вредна). Дополнительный слой защиты со стороны системы — запрет отображения нулевой страницы (`mmap_min_addr > 0`).

### Знаковый offset → беззнаковый адрес (иллюстрация класса бага)

```c
// Упрощённый паттерн обработчика записи с offset:
ssize_t mem_write(const char __user *buf, size_t count, loff_t *ppos) {
    // loff_t — знаковый 64-бит. Если *ppos пришёл отрицательным,
    // а проверка диапазона забыта или сделана в знаковой арифметике...
    unsigned long addr = *ppos;  // знаковый → беззнаковый: -1 даёт 0xFFFF...FFFF
    // ...write по addr — большой беззнаковый адрес вместо «маленького отрицательного»
}
```

После перевода знакового в беззнаковый `(unsigned long)(-1LL) == 0xFFFFFFFFFFFFFFFF` — то есть «отрицательное смещение» превращается в адрес у верхней границы пространства. Класс бага: доверие к offset без проверки `*ppos >= 0` и сравнение в неверной знаковости.

> Реальная уязвимость `/proc/<pid>/mem` (CVE-2012-0056, *mempodipper*) по сути была не про sign-extension, а про **отсутствие проверки прав** при записи после рефакторинга в 2.6.39 — здесь приведён лишь родственный по духу паттерн «знаковый offset → беззнаковый адрес», а не реконструкция того CVE.

**Тип UB/баг:** неявное расширение знака знакового типа в беззнаковый; отсутствие проверки диапазона.
**Что поймало бы:** `-Wsign-conversion`, ручная проверка `*ppos >= 0`.

### Signed integer overflow → произвольное исполнение кода

```c
size_t total = header.count * header.size;  // оба int из заголовка сети
char *buf = malloc(total);
// Если count=0x10000, size=0x10001: int overflow → total = маленькое число
// malloc(маленькое) → write(buf, data, count*size) → heap overflow
memcpy(buf, data, (size_t)header.count * header.size);  // выход за пределы buf
```

**Тип UB:** знаковое переполнение (если типы `int`); умножение перед проверкой.
**Лечение:** `__builtin_mul_overflow`; проверять отдельно каждый операнд.
**Что поймало бы:** UBSan, ASan на `memcpy`.

### Integer truncation в размере аллокации

```c
// Получили size из сети как uint64_t (или size_t на 64-бит)
uint64_t network_len = get_u64();
uint32_t local_len   = (uint32_t)network_len;  // УСЕЧЕНИЕ!
char *buf = malloc(local_len + 1);
memcpy(buf, data, network_len);  // пишем network_len байт в local_len+1 байт → overflow
```

**Тип:** integer truncation (impl-defined поведение при приведении к меньшему беззнаковому типу).
**Лечение:** никогда не урезай размер без явной проверки `if (n > UINT32_MAX) return -EINVAL`.

### Type confusion через void* без тега

```c
typedef enum { OBJ_FOO, OBJ_BAR } obj_type;
typedef struct { obj_type type; /* ... */ } base_t;
typedef struct { obj_type type; int data[100]; } foo_t;
typedef struct { obj_type type; char name[8]; } bar_t;

void process(void *obj) {
    foo_t *f = (foo_t*)obj;  // каст без проверки тега!
    f->data[0] = 42;         // если obj — на самом деле bar_t: пишем за name[8]
}
```

**Лечение:** всегда проверяй тег `((base_t*)obj)->type` перед кастом; или используй tagged union.

### Format string via user input — логгер

```c
// Логгер получает сообщение от сети:
char *user_msg = receive_from_network();
syslog(LOG_INFO, user_msg);  // УЯЗВИМОСТЬ! user_msg используется как формат-строка

// user_msg == "%n%n%n%n%n" → запись по адресам из va_list syslog
// Правильно:
syslog(LOG_INFO, "%s", user_msg);
```

**Что поймало бы:** `gcc -Wformat -Wformat-security` выдаёт предупреждение; ASan не ловит (не OOB по памяти, а нарушение семантики).

### Use-after-free через goto-cleanup с неправильным порядком меток

```c
int bad_cleanup(void) {
    char *a = malloc(64);
    if (!a) goto out;
    char *b = malloc(64);
    if (!b) goto out;   // ОШИБКА: прыгаем на out, не free(a)! Утечка.

    // ... работа ...

out:
    free(b);   // b может быть NULL если второй malloc упал — но это ok: free(NULL)
    free(a);   // a — уже free? НЕТ, но у нас тут порядок меток неправильный
    return 0;
}

// Правильный паттерн:
int good_cleanup(void) {
    int ret = -1;
    char *a = NULL, *b = NULL;
    a = malloc(64); if (!a) goto out;
    b = malloc(64); if (!b) goto out_free_a;
    ret = 0;
out_free_a:
    free(b);
out:
    free(a);
    return ret;
}
```

**Тип:** не UB, а логическая ошибка в порядке меток → утечка памяти. В более сложных случаях — free в неправильном порядке → heap corruption.

### Off-by-one в strncpy — забытый нулевой терминатор

```c
char dst[16];
const char *src = "exactly15chars!";  // ровно 15 символов + '\0' = 16 байт

strncpy(dst, src, sizeof(dst));
// strncpy пишет min(strlen(src)+1, n) символов, НО:
// если strlen(src) == sizeof(dst) — '\0' НЕ записывается!
// dst[15] = '!' — нет терминатора; dst — не корректная C-строка

printf("%s\n", dst);  // читает за пределы dst до ближайшего '\0' — UB/OOB read

// Правильно: всегда добавляй терминатор явно:
strncpy(dst, src, sizeof(dst) - 1);
dst[sizeof(dst) - 1] = '\0';
// или snprintf:
snprintf(dst, sizeof(dst), "%s", src);
```

**Что поймало бы:** ASan на `printf` (чтение за пределами массива).

## 13. Практика модуля (соответствует «Руки» трека)

Делать руками, не читать. Каждое — компилируется с `-std=c17 -Wall -Wextra -Wconversion -Wsign-conversion -fsanitize=undefined,address`.

1. **Overflow-safe arithmetic lib.** `add/sub/mul` для `int`/`size_t` с детектом переполнения через `__builtin_*_overflow`; тесты на граничных (`INT_MIN`, `SIZE_MAX`). *(закрывает §2–3)*
2. **Сравнения без UB.** Функция «безопасно сравнить знаковое и беззнаковое» (аналог C++20 `cmp_less`) на чистом C. Тест на `-1 < 0u`. *(§3.2)*
3. **Integer promotions.** `uint8_sum_overflows`, `bitwise_not_u8`, `safe_lt`. *(§3, §8)*
4. **Type punning тремя способами.** `float↔uint32` через грязный каст, union, memcpy. Сравни ассемблер `-O2`, объясни, какой нарушает strict aliasing. *(§4)*
5. **`restrict`-бенч.** Векторное сложение с/без `restrict`, замерь и покажи разницу в ассемблере. *(§5)*
6. **`read_whole_file` с goto-cleanup** под ASan+valgrind на крайних случаях. *(§6)* — это и есть автотестируемое задание в приложении.
7. **Struct layout.** Переставить поля `bad_layout`, реализовать `container_of`. *(§9)*
8. **Safe strings.** `safe_strcpy`, `safe_strcat`, `parse_int` с полным error-checking. *(§10)*
9. **Function pointers и dispatch table.** Компараторы для `qsort`, dispatch table из 4 операций. *(§8.5)*
10. **Обёртка над сырым syscall с `EINTR`-ретраем** — мостик в Ф2 (`write`, перезапуск при `EINTR`).
11. **Сравнение библиотечной обёртки и `syscall()`** (`getpid` через glibc vs `syscall(SYS_getpid)`) — мостик в Ф2, vDSO.

<details>
<summary><b>Эталонные ответы (раскрыть после попытки)</b></summary>

1. **Impl-defined** — реализация выбирает и документирует (знаковость `char`). **Unspecified** — выбирает из набора, документировать не обязана, может меняться (порядок вычисления аргументов). **UB** — никаких требований, вся программа невалидна (знаковое переполнение). Практическая разница: на impl-defined можно полагаться, прочитав доку компилятора/ABI; на unspecified — нет, но программа остаётся корректной.
2. Наличие UB на пути исполнения снимает с компилятора любые обязательства на этом пути; он выводит факты из «UB не происходит». `x+1<x`: знаковое переполнение UB ⇒ `x+1` «всегда» `>x` ⇒ выражение всегда `0` ⇒ `return 0`. Проверка исчезает на `-O2`.
3. (а) знаковое переполнение — билтины/`-fwrapv`/UBSan; (б) чтение неинициализированного — инициализация при объявлении/MSan; (в) нарушение strict aliasing — `memcpy`/`-fno-strict-aliasing`/`-Wstrict-aliasing`. (Также: NULL-дереф с последующим удалением проверки, сдвиг ≥ ширины.)
4. Промоушен: `char/short/bool/битполя` → `int` до арифметики. UAC: операнды бинарной операции приводятся к общему типу по рангу; при равном ранге беждает беззнаковый. `-1 < 0u`: `-1`→`UINT_MAX` ⇒ ложно. `~(uint8_t)0`: операнд → `int 0`, `~0 == -1` (int).
5. `size_t` беззнаковый, всегда `>= 0`, после `0` идёт `SIZE_MAX` ⇒ условие никогда не ложно. Корректно: `for (size_t i = n; i-- > 0; ) { use(i); }`.
6. Доступ к объекту — только через совместимый тип, кроме char-семейства и union-члена. Легально для битов: `memcpy`; через `unsigned char*` побайтно; union-punning (в C). C vs C++: union-punning определён в C, UB в C++ (там `memcpy`/`bit_cast`). Ядро: массовые касты низкоуровневого кода → отключают оптимизацию флагом, иначе UB.
7. `restrict`: к объекту не обращаются ни через какой иной указатель в течение жизни указателя. Взамен — снятие предположения об алиасинге → переупорядочивание/векторизация. Нарушение — UB (тихая порча данных под оптимизацией).
8. goto-cleanup: единое место освобождения в обратном порядке захвата, без дублирования по веткам, корректный teardown — стиль ядра. `free(NULL)` no-op ⇒ можно инициализировать всё в NULL и иметь общую метку. `close(-1)` даёт `EBADF` ⇒ fd надо инициализировать `-1` и закрывать под `if (fd>=0)`.
9. `volatile` — не атомарность и не барьер памяти; гарантирует лишь «не кэшировать/не выкидывать обращения». Для потоков нужны `_Atomic`/барьеры; `volatile` не упорядочивает доступы к другим объектам и не даёт атомарности RMW.
10. UBSan — логические UB в рантайме (переполнение, сдвиги, неверные приведения, NULL-дереф). ASan — память (OOB, UAF, double-free, утечки). TSan — гонки. Баг из в.2 поймал бы UBSan (`signed-integer-overflow`), если этот вход реально исполнится.

</details>
