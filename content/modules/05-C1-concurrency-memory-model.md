# Модуль C1 — Конкурентность и модель памяти

> Этот модуль помечен в треке как **«закрепить»**: ты уже знаешь C++ atomics,
> `compare_exchange`, `shared_ptr` с CAS. Здесь мы пересобираем эти знания
> строго и снизу вверх — от транзисторов кэша до формальной модели памяти —
> и доводим до уровня, на котором ты можешь **доказать** корректность
> lock-free структуры через memory ordering, а не «проверить, что вроде
> работает».
>
> **Язык изложения — двуязычный.** Основной — **C++** (`std::atomic`,
> `std::thread`, RAII-локи): это твой родной язык и язык userspace-этапа 2A;
> упражнения тоже на C++. Рядом, где это важно, дан **C-эквивалент**
> (`<stdatomic.h>`, `pthread`) — потому что этапы 2B (ядро) и 2C (embedded) —
> это **только C**, и ту же модель памяти там выражают на C11. Ключевой факт:
> **модель памяти C11 и C++11 — одна и та же** (один и тот же стандартный
> текст), меняется только синтаксис: `std::atomic<T>` ↔ `_Atomic T`,
> методы `.load()/.store()` ↔ функции `atomic_load_explicit/atomic_store_explicit`,
> `std::memory_order_acquire` ↔ `memory_order_acquire`. Научившись на C++, ты
> бесплатно умеешь то же на C.

## 1. Почему общая переменная «не работает»

Наивная модель в голове новичка: «память — это большой массив байт, потоки
по очереди читают и пишут в него». Эта модель **неверна на трёх уровнях
сразу**, и каждый уровень имеет право переупорядочивать твои операции.

### 1.1 Демонстрация проблемы

Классический пример. Два потока, две переменные, обе изначально 0:

```c
/* Поток 1 */            /* Поток 2 */
x = 1;                   y = 1;
r1 = y;                  r2 = x;
```

Вопрос: возможен ли исход `r1 == 0 && r2 == 0`?

Наивно — нет: чтобы `r1 == 0`, поток 1 прочитал `y` до записи `y = 1` в потоке
2; чтобы `r2 == 0`, поток 2 прочитал `x` до записи `x = 1`. Но `x = 1` идёт
**раньше** `r1 = y`, а `y = 1` — раньше `r2 = x`. Кажется, что это противоречие.

**На реальном x86 этот исход наблюдается.** Это не баг процессора — это
зафиксированное в спецификации поведение, называемое **store-load
reordering**. Каждый поток видит свои операции по порядку, но другой поток
может увидеть их в другом порядке.

### 1.2 Три уровня переупорядочивания

```
   Исходный код на C
        │
        ▼  (1) КОМПИЛЯТОР переставляет, удаляет, кэширует в регистрах
   Машинные инструкции
        │
        ▼  (2) CPU исполняет out-of-order, спекулятивно
   Поток микроопераций
        │
        ▼  (3) ПОДСИСТЕМА ПАМЯТИ: store buffer, кэши, протокол когерентности
   То, что видят другие ядра
```

**(1) Компилятор.** При `-O2` компилятор считает, что он единственный, кто
трогает память (если ты не сказал иначе). Он спокойно:
- держит переменную в регистре и не перечитывает её из памяти в цикле;
- переставляет независимые записи;
- выкидывает «лишние» чтения.

```c
/* Что написал ты */            /* Что увидит другой поток после -O2 */
while (!flag) { }               if (!flag) for(;;) { }   /* flag прочитан ОДИН раз */
do_work();                      /* do_work недостижим, если flag был 0 */
```

**(2) Процессор.** Современное суперскалярное ядро исполняет десятки инструкций
«в полёте», вне порядка программы, переименовывает регистры, спекулирует по
веткам. Порядок завершения (retire) не совпадает с порядком, в котором
результаты становятся видны памяти.

**(3) Подсистема памяти.** У каждого ядра есть **store buffer** — очередь
записей, ещё не дошедших до кэша. Своё ядро читает из store buffer (видит свои
записи сразу), а другие ядра — нет, пока запись не «сольётся» (drain) в
когерентный кэш. Отсюда store-load reordering из §1.1.

### 1.3 Вывод

> Любое межпоточное взаимодействие через память **обязано** использовать
> синхронизацию: мьютексы, атомарные операции с правильным `memory_order`, или
> барьеры. Иначе у тебя нет вообще никаких гарантий о порядке — компилятор и
> железо вправе сделать что угодно. Это не «оптимизация поведения», это
> **неопределённое поведение (UB)** при наличии data race.

## 2. Data race vs race condition

Два термина, которые путают почти все. Разница принципиальная.

### 2.1 Data race (гонка данных) — это про UB

**Определение (C11 §5.1.2.4):** data race возникает, когда два потока
обращаются к одной ячейке памяти, **хотя бы одно из обращений — запись**, эти
обращения **не упорядочены** отношением happens-before, и **ни одно не
атомарно**.

Data race — это **неопределённое поведение**. Не «иногда неправильный
результат», а UB в полном смысле: компилятор имеет право предположить, что его
не бывает, и оптимизировать соответственно. Программа с data race не имеет
смысла вообще.

```c
int counter = 0;
/* Поток A */          /* Поток B */
counter++;             counter++;   /* DATA RACE → UB */
```

`counter++` — это read-modify-write (load, +1, store). Два потока без
синхронизации = data race = UB. Типичный видимый симптом — «потерянные
инкременты» (итог меньше ожидаемого), но формально поведение неопределено.

### 2.2 Race condition (состояние гонки) — это про логику

Race condition — это когда **корректность зависит от относительного времени**
событий. Race condition может существовать **без** data race и наоборот.

Пример race condition без data race (всё атомарно, но логика всё равно гонится):

```c
/* TOCTOU — Time-Of-Check To Time-Of-Use */
if (atomic_load(&balance) >= amount) {   /* проверка */
    /* другой поток успел снять деньги здесь */
    atomic_fetch_sub(&balance, amount);  /* использование — баланс ушёл в минус */
}
```

Здесь **нет** data race (обе операции атомарны), но есть **race condition**:
проверка и действие не атомарны как единое целое. Лечится мьютексом или
CAS-циклом, а не атомиком на каждую операцию.

### 2.3 Матрица

| | Есть data race | Нет data race |
|--|--|--|
| **Есть race condition** | `counter++` из 2 потоков | TOCTOU на атомиках |
| **Нет race condition** | (редко; теоретически «benign», но всё равно UB) | корректный код под мьютексом |

> **Запомни формулировку для собеседования:** data race — это техническое
> свойство (UB по стандарту), race condition — это свойство корректности
> (неправильный результат из-за времени). Устранение data race (атомики)
> **не** устраняет автоматически race condition.

## 3. Атомарность, `volatile` и `_Atomic`

### 3.1 Что такое атомарная операция

**Атомарная операция** — неделимая с точки зрения других потоков: другой поток
видит её либо целиком выполненной, либо ещё не начатой, но никогда —
наполовину. Плюс атомарные операции в C11 несут **memory ordering** (см. §5),
чего у обычных операций нет.

Атомарность нужна по двум причинам:
1. **Неразрывность** read-modify-write (`++`, `+=`, CAS).
2. **Отсутствие «рваного» чтения** (torn read): на 32-битной шине запись
   64-битного значения может быть видна другим ядром «половинками».

### 3.2 Главное заблуждение: `volatile` ≠ атомарность

`volatile` в C/C++ означает ровно одно: **«не кэшируй это в регистре, каждое
обращение — реальное обращение к памяти»**. Он создан для memory-mapped I/O
(регистры устройств), где каждое чтение/запись имеет побочный эффект.

`volatile` **НЕ** даёт:
- атомарности (`volatile int x; x++;` — всё ещё read-modify-write, всё ещё гонка);
- memory ordering (компилятор не переставит два `volatile`-доступа между собой,
  но **процессор** их переставит, и обычные не-`volatile` доступы вокруг — тоже);
- межъядерной видимости.

```cpp
volatile int ready = 0;          // НЕВЕРНО как флаг синхронизации (C++ и C)
std::atomic<int> ready_ok{0};    // ВЕРНО (C++);  C-эквивалент: _Atomic int ready_ok = 0;
```

> **Правило:** в межпоточном коде `volatile` тебе почти никогда не нужен.
> Нужен `_Atomic` (C11) / `std::atomic` (C++). `volatile` — для железа (MMIO)
> и обработчиков сигналов (`volatile sig_atomic_t`), не для потоков.
> Исключение — ядро Linux, где своя модель (`READ_ONCE`/`WRITE_ONCE` поверх
> `volatile` + барьеры компилятора); в userspace на это не опирайся.

### 3.3 `std::atomic` (C++) и `_Atomic` (C)

```cpp
// --- C++ (основной) ---
#include <atomic>
std::atomic<int> counter{0};

counter.fetch_add(1);                                  // атомарный ++, по умолч. seq_cst
int v = counter.load();                                // атомарное чтение
counter.store(42);                                     // атомарная запись
counter++;                                             // тоже атомарен, но всегда seq_cst
counter.fetch_add(1, std::memory_order_relaxed);       // явный, дешёвый порядок
```

```c
// --- C-эквивалент (ядро/embedded) ---
#include <stdatomic.h>
_Atomic int counter = 0;                               // или atomic_int counter;
atomic_fetch_add(&counter, 1);                         // seq_cst
int v = atomic_load(&counter);
atomic_store(&counter, 42);
atomic_fetch_add_explicit(&counter, 1, memory_order_relaxed);
```

**Lock-free или нет?** Не всякий атомарный тип реализован без блокировки — для
больших структур компилятор может спрятать мьютекс внутри. Проверяй:

```cpp
std::atomic<Big> s;
static_assert(std::atomic<Big>::is_always_lock_free);  // C++17, на этапе компиляции
if (s.is_lock_free()) { /* честный lock-free */ }      // C: atomic_is_lock_free(&s)
// Для фундаментальных типов на x86-64 (int, long, указатели) — всегда lock-free.
```

## 4. Железо: кэши, когерентность, store buffer

Чтобы понимать memory ordering, надо понимать, **почему** оно слабое. Источник —
архитектура памяти. (Подробнее — Drepper, «What Every Programmer Should Know
About Memory».)

### 4.1 Иерархия кэшей и линия кэша

```
Ядро 0            Ядро 1
┌──────┐          ┌──────┐
│ L1d  │ 32 KB    │ L1d  │   ~4 такта,  64-байтная линия
│ L2   │ 256KB-1M │ L2   │   ~12 тактов
└───┬──┘          └───┬──┘
    └──────┬──────────┘
        ┌──┴──┐
        │ L3  │  общий, ~40 тактов
        └──┬──┘
        ┌──┴──┐
        │ RAM │  ~200+ тактов (≈100 нс)
        └─────┘
```

Ключевой факт: память кэшируется **линиями по 64 байта**, не по байтам. Это
порождает false sharing (§10). Промах в RAM стоит ~сотни тактов — порядок,
который определяет, lock-free твоя структура или нет на практике.

### 4.2 MESI — протокол когерентности кэша

Чтобы ядра не видели разные значения одной ячейки, кэши «договариваются» по
протоколу MESI. Каждая линия в каждом кэше имеет одно из состояний:

| Состояние | Смысл |
|-----------|-------|
| **M** (Modified) | линия только у меня, изменена, в RAM устарела |
| **E** (Exclusive) | линия только у меня, чистая (== RAM) |
| **S** (Shared) | линия может быть в нескольких кэшах, чистая |
| **I** (Invalid) | линии нет / устарела |

Чтобы **записать** в линию, ядро должно получить её в состоянии M, для чего
рассылает **Read-For-Ownership (RFO)** и **инвалидирует** копии в других кэшах.
Это межъядерный трафик. Когда два ядра пишут в одну линию по очереди, она
«пинг-понгует» между их кэшами (cache line bouncing) — каждая запись стоит как
промах. Это физическая причина, по которой конкурентная запись в общую ячейку
не масштабируется (§10, §13).

### 4.3 Store buffer и почему x86 — это TSO

Запись в когерентный кэш медленная (нужен RFO). Чтобы не ждать, ядро кладёт
запись в **store buffer** и продолжает исполнение. Последствия:

- своё ядро читает из store buffer → видит свои записи сразу (**store
  forwarding**);
- другие ядра увидят запись только когда она сольётся в кэш;
- **load может «обогнать» более ранний store** к другому адресу — это и есть
  store-load reordering из §1.1.

**x86/x86-64 = TSO (Total Store Order):** единственное переупорядочивание,
которое допускает железо, — store→load. Store→store, load→load, load→store
сохраняют порядок. Поэтому x86 — «сильная» модель, и многие гонки на нём
**не воспроизводятся** (но воспроизведутся на ARM!).

### 4.4 ARM/POWER — слабая модель

На ARM и POWER store buffer и подсистема памяти допускают **любые**
переупорядочивания (store→store, load→load и т.д.), если нет зависимостей и
барьеров. Поэтому:

> Код, «работающий» на x86 без правильного ordering, регулярно ломается на ARM.
> Сегодня это не экзотика: серверы Graviton/Ampere, все телефоны, Apple
> Silicon. **Пиши по модели C11, а не по тому, что прощает x86.**

Это прямой ответ на вопрос ЭКСПЕРТ-1 трека: `seq_cst` дорог на ARM, потому что
для него нужны полные барьеры (`dmb ish`), тогда как на x86 TSO даёт почти всё
бесплатно, и `seq_cst`-store требует лишь `xchg`/`mfence`.

### 4.5 NUMA — память не одинаково «далеко»

На многосокетных серверах память **локальна сокету**: ядро сокета 0 читает
«свою» RAM быстрее, чем RAM сокета 1 (доступ идёт через межсокетный
интерконнект). Это **NUMA** (Non-Uniform Memory Access). Следствия для
конкурентного кода:

- линия в состоянии M на «чужом» сокете стоит ещё дороже (когерентность через
  интерконнект) — false sharing и общий счётчик на NUMA бьют сильнее;
- аллоцируй и трогай данные на том ядре, которое будет с ними работать
  (**first-touch policy**: страница физически ложится к тому, кто первым в неё
  записал); привязывай потоки (`pthread_setaffinity_np`, `numactl`);
- per-CPU/per-node данные — не роскошь, а условие масштабирования на 2+ сокетах.

### 4.6 Цифры, которые надо помнить

Порядок величин (для интуиции, конкретика зависит от CPU):

| Операция | ~Такты | ~Время |
|----------|-------:|-------:|
| Регистр / L1-попадание | 1–4 | <1 нс |
| L2-попадание | ~12 | ~3 нс |
| L3-попадание (свой сокет) | ~40 | ~12 нс |
| Промах в локальную RAM | ~200 | ~60–100 нс |
| Доступ к RAM чужого NUMA-узла | ~300+ | ~120 нс |
| Атомарный RMW без контеншна | ~15–30 | единицы нс |
| CAS под контеншном (bouncing линии) | сотни | как промах |
| `futex` усыпление/пробуждение (syscall) | тысячи | ~1–3 мкс |
| Переключение контекста | — | ~1–5 мкс |

Главный вывод: **межъядерная синхронизация на изменяемой общей линии стоит как
промах в память**. Поэтому масштабируемость = минимум общей записи, а спор
«спинлок vs мьютекс» решается тем, короче ли секция, чем пара переключений
контекста (§8.3).

## 5. Модель памяти C11: шесть `memory_order`

Это сердце модуля. Модель памяти — это **контракт**: набор правил, которые
говорят, какие записи обязан увидеть поток при чтении. Она строится вокруг
одного отношения — **happens-before**.

### 5.1 Базовые отношения

- **sequenced-before** — порядок внутри одного потока (то, что задаёт исходный
  код, с точностью до точек следования).
- **synchronizes-with** — межпоточная связь, которую создаёт пара
  release-операция → acquire-операция (см. ниже).
- **happens-before** — транзитивное замыкание sequenced-before и
  synchronizes-with. Если A happens-before B, то эффекты A видны в B.

> **Главное правило:** если запись W и чтение R **не** связаны happens-before, и
> хотя бы одна неатомарна — это **data race (UB)**. Если связаны — R обязан
> увидеть W (или более позднюю запись).

### 5.2 Шесть порядков

```cpp
// C++: std::memory_order_relaxed и т.д. (или enum class std::memory_order::relaxed)
// C:   memory_order_relaxed и т.д. — те же имена без префикса std::
std::memory_order_relaxed   // только атомарность, без ordering
std::memory_order_consume   // «облегчённый acquire» по зависимостям (см. 5.7)
std::memory_order_acquire   // для загрузок: ничто ниже не уедет выше
std::memory_order_release   // для записей: ничто выше не уедет ниже
std::memory_order_acq_rel   // для RMW: и acquire, и release сразу
std::memory_order_seq_cst   // + единый глобальный порядок (по умолчанию)
```

#### relaxed — только атомарность

Гарантирует **неделимость** операции и **единый modification order** для данной
переменной (все потоки видят записи в одну атомарную переменную в одном
порядке). Но **не** упорядочивает её относительно других переменных.

Применение: счётчики, где важен лишь итог, а не порядок относительно других
данных.

```cpp
// Корректный счётчик статистики: важна только финальная сумма
packets.fetch_add(1, std::memory_order_relaxed);            // C++
// atomic_fetch_add_explicit(&packets, 1, memory_order_relaxed);  // C
```

#### release / acquire — рабочая лошадка

- **release-запись** (store): все операции, которые в этом потоке
  **sequenced-before** неё, становятся видимы любому потоку, который сделает
  **acquire-чтение** этого же значения.
- **acquire-чтение** (load): всё, что было до соответствующего release в том
  потоке, становится видимо после этого чтения.

Пара release→acquire на **одной** атомарной переменной создаёт
**synchronizes-with**, и через него — happens-before. Это **механизм
публикации данных**:

```cpp
int data;                          // обычные, НЕ атомарные данные
std::atomic<bool> ready{false};

// Поток-писатель
data = 42;                                              // (1)
ready.store(true, std::memory_order_release);           // (2) публикуем

// Поток-читатель
while (!ready.load(std::memory_order_acquire))          // (3) ждём
    ;
assert(data == 42);                // (4) ГАРАНТИРОВАНО: (1) happens-before (4)
```

```c
/* C-эквивалент: тот же смысл, иной синтаксис */
int data;  _Atomic _Bool ready = false;
data = 42;  atomic_store_explicit(&ready, true, memory_order_release);
while (!atomic_load_explicit(&ready, memory_order_acquire)) ;
assert(data == 42);
```

Почему `data == 42` гарантировано? (1) sequenced-before (2); (2)
synchronizes-with (3) (release→acquire на `ready`, и (3) прочитал значение,
записанное (2)); (3) sequenced-before (4). По транзитивности (1)
happens-before (4). Запись `data` обязана быть видна. **И при этом `data`
остаётся обычной переменной — гонки нет, потому что happens-before её
упорядочил.**

> Это и есть «доказательство через memory ordering», которого требует трек.
> Запомни шаблон рассуждения: *sequenced-before → synchronizes-with →
> sequenced-before ⇒ happens-before*.

#### acq_rel — для read-modify-write

RMW-операция (`fetch_add`, `exchange`, `compare_exchange`) и читает, и пишет.
`acq_rel` делает её одновременно acquire (по отношению к прочитанному значению)
и release (по отношению к записанному). Нужна для узлов, которые и забирают
чужую публикацию, и публикуют свою (мьютексы, lock-free стек).

#### seq_cst — самый сильный (и дефолтный)

Кроме acquire/release добавляет **единый глобальный тотальный порядок** всех
seq_cst-операций, который видят все потоки одинаково. Это закрывает дыру
store-load из §1.1: если в примере Деккера обе записи seq_cst, исход
`r1==0 && r2==0` невозможен.

`seq_cst` — поведение по умолчанию для `++`, `atomic_store`, `atomic_load` без
явного порядка. Самый простой для рассуждения, но **самый дорогой**: на x86 —
`mfence`/`xchg`, на ARM — полные барьеры. Используй его как безопасный дефолт,
а ослабляй до acquire/release **только** доказав корректность.

### 5.3 Сводная таблица

| `memory_order` | Атомарность | Запрещает reorder | synchronizes-with | Глобальный порядок | Цена x86 / ARM |
|----------------|:-:|--|:-:|:-:|--|
| relaxed | ✓ | ничего | ✗ | ✗ | дёшево / дёшево |
| acquire (load) | ✓ | downward (вниз нельзя поднять) | ✓ (с release) | ✗ | дёшево / `dmb` |
| release (store) | ✓ | upward (вверх нельзя опустить) | ✓ (с acquire) | ✗ | дёшево / `dmb` |
| acq_rel (RMW) | ✓ | оба | ✓ | ✗ | средне / `dmb` |
| seq_cst | ✓ | оба | ✓ | **✓** | `mfence` / полный барьер |

### 5.4 Release sequence — тонкость

Если после release-записи **тот же** или другие потоки делают RMW-операции над
этой переменной (не простые store), они образуют **release sequence**, и
acquire, прочитавший любое значение из этой последовательности, всё равно
синхронизируется с исходным release. Это важно для корректности счётчиков
ссылок и семафоров.

### 5.5 Modification order и когерентность

Для **каждой** атомарной переменной существует единый **modification order** —
тотальный порядок всех записей в неё, согласованный для всех потоков (даже при
relaxed). Гарантии когерентности (всегда, независимо от order):
- read-read: если ты прочитал значение, последующие чтения не вернут «более
  старое»;
- write-write: твои записи в одну переменную видны другим в том же порядке.

relaxed теряет порядок **между разными** переменными, но не внутри одной.

### 5.6 Типичная ошибка: ordering — это про ПАРУ

Самая частая ошибка — повесить `release` на запись и забыть `acquire` на
чтение (или наоборот). **Ordering работает только парами.** Release без acquire
на другой стороне не создаёт synchronizes-with — публикации не происходит.

### 5.7 `consume` — почему его не используют

`memory_order_consume` задумывался как дешёвый acquire, переносящий порядок
только по **цепочке зависимостей данных** (dependency ordering) — выгодно на
POWER/ARM, где зависимость бесплатна, а полный acquire-барьер дорог. На практике
**ни один компилятor не реализует его честно** — все повышают до acquire. C++17
официально не рекомендует его до пересмотра. **Используй acquire.** Знать про
consume нужно для собеседований и чтения чужого кода.

## 6. CAS: compare-and-swap, weak vs strong, ABA

### 6.1 Что такое CAS

Compare-and-swap — атомарный примитив, на котором строится почти весь lock-free
код: «если переменная равна ожидаемому — заменить на новое, иначе — сказать,
что было на самом деле».

```cpp
// C++: метод на std::atomic<T>
bool std::atomic<T>::compare_exchange_strong(T& expected, T desired);
// Атомарно:
//   if (value == expected) { value = desired; return true; }
//   else                   { expected = value; return false; }
// C-эквивалент: atomic_compare_exchange_strong(_Atomic T* obj, T* expected, T desired)
```

Канонический CAS-цикл (атомарно применить функцию `f` к атомику):

```cpp
T old = obj.load(std::memory_order_relaxed);
T neu;
do {
    neu = f(old);                      // вычисляем новое из старого
} while (!obj.compare_exchange_weak(
            old, neu,
            std::memory_order_acq_rel,    // при успехе
            std::memory_order_relaxed));  // при провале old уже перечитан
```

```c
/* C-эквивалент того же цикла */
T old = atomic_load_explicit(obj, memory_order_relaxed), neu;
do { neu = f(old); }
while (!atomic_compare_exchange_weak_explicit(
           obj, &old, neu, memory_order_acq_rel, memory_order_relaxed));
```

Если CAS провалился, `old` уже обновлён реальным значением — цикл повторяется.
Так реализуются lock-free инкременты, стеки, очереди.

### 6.2 weak vs strong

| | `compare_exchange_strong` | `compare_exchange_weak` |
|--|--|--|
| Ложный провал | никогда | **может** провалиться, даже если `*obj == *expected` |
| Цена на LL/SC | внутренний цикл (дороже) | одна попытка (дешевле) |
| Когда брать | если CAS **не** в цикле | если CAS **в** цикле |

**Почему weak вообще существует?** На LL/SC-архитектурах (ARM, POWER, RISC-V)
CAS реализуется парой **Load-Linked / Store-Conditional**: LL читает и ставит
«монитор» на адрес, SC записывает только если с момента LL никто адрес не
трогал. SC может провалиться не только из-за реальной записи, но и из-за
прерывания, переключения контекста, ложного срабатывания монитора, промаха
кэша. `strong` обязан скрыть эти ложные провалы внутренним циклом — лишние
инструкции. `weak` отдаёт ложный провал наружу — а ты и так в цикле, тебе
дешевле.

> **Правило:** в цикле — всегда `weak` (ложный провал безвреден, ты повторишь
> итерацию). Одиночный CAS вне цикла (например, «один раз попытаться занять
> слот») — `strong`, чтобы не городить цикл вокруг ложного провала. На x86
> разницы в цене почти нет (`lock cmpxchg` не имеет ложных провалов), но писать
> надо переносимо.

### 6.3 ABA-проблема

Самая коварная ловушка lock-free. CAS проверяет **значение**, а не «менялось
ли». Сценарий на lock-free стеке:

```
Поток 1 хочет снять вершину A (next = B):
  читает head = A, готовит CAS(head, A → B)
  ... ВЫТЕСНЕН ...
Поток 2: pop A, pop B, освобождает B, push A обратно.
  Теперь head = A снова, но A->next уже не B (B освобождён!).
Поток 1 просыпается:
  CAS(head, A → B) УСПЕШЕН (head всё ещё == A!)
  head стал B — указывает на освобождённую/чужую память. Катастрофа.
```

CAS «не заметил», что значение успело уйти и вернуться (A→…→A). Решения:
1. **Tagged pointer / версионный счётчик:** упаковать в одно атомарное слово
   указатель + счётчик версии, инкрементируемый при каждом изменении. ABA не
   проходит, т.к. тег различается. Нужен двойной-широкий CAS (`cmpxchg16b` на
   x86-64). Ограничение: тег может переполниться (ABA-A очень редко, но
   теоретически).
2. **Hazard pointers** (§12) — не освобождать узел, пока на него кто-то
   ссылается.
3. **RCU / epoch-based reclamation** (§12) — отложенное освобождение.

ABA — это прямой ответ на вопрос МЕХАНИЗМЫ-5 трека.

### 6.4 Типичные ошибки CAS-кода

1. **Забыли, что `expected` меняется при провале.** `compare_exchange` при
   неудаче **пишет** в `expected` актуальное значение. Это удобно (не надо
   перечитывать), но если ты в провальной ветке снова сам делаешь `load`, то
   затрёшь свежее значение устаревшим и получишь лишние итерации или баг.
2. **`strong` в цикле.** Работает, но медленнее на ARM/POWER (внутренний цикл
   против ложных провалов поверх твоего цикла). В цикле — `weak`.
3. **`weak` вне цикла.** Одиночный `weak`-CAS может «ложно» провалиться при
   равенстве — и ты ошибочно решишь, что значение изменилось. Вне цикла — `strong`.
4. **Слишком сильный порядок «на всякий случай».** `seq_cst` в CAS-цикле, где
   хватает `acq_rel`/`relaxed`, — потеря производительности на ровном месте,
   особенно на ARM.
5. **Слишком слабый порядок на успехе.** Если CAS публикует данные (как `push` в
   Treiber stack) — успех должен быть `release` (или `acq_rel`), иначе записанные
   поля узла не видны тому, кто его потом прочитает.
6. **ABA не учтён.** CAS по указателю, который может уйти-и-вернуться, без
   tag/reclamation — мина (§6.3, §15.2).
7. **Знаковое переполнение в индексах.** В кольцевых очередях разница
   беззнаковых индексов (`tail - head`) корректна по модулю; не приводи к
   знаковому типу неаккуратно — UB переполнения поймает UBSan.

```cpp
// Канонически правильно: weak в цикле, expected не трогаем в провальной ветке.
T old = a.load(std::memory_order_relaxed);
while (!a.compare_exchange_weak(old, f(old),
        std::memory_order_acq_rel,      // успех (если публикуем — минимум release)
        std::memory_order_relaxed))     // провал: old уже обновлён, просто крутимся
    ;
```

## 7. Барьеры: `atomic_thread_fence`

Иногда ordering нужно отделить от конкретной операции — тогда используют
**отдельный барьер**:

```c
atomic_thread_fence(memory_order_release);  /* «забор» в коде */
atomic_thread_fence(memory_order_acquire);
```

`atomic_thread_fence(release)` перед relaxed-store даёт примерно тот же эффект,
что release-store, но порядок применяется ко **всем** предшествующим
операциям, а не только к этой записи. Используется, когда одна release-точка
публикует много данных, записанных relaxed, или в коде с массовыми атомиками,
где навешивать порядок на каждую операцию дорого.

Не путай с `atomic_signal_fence` (только барьер компилятора, для обработчиков
сигналов в **том же** потоке — без межъядерной синхронизации) и с барьерами
компилятора (`asm volatile("" ::: "memory")`), которые запрещают переупорядочивание
**только компилятору**, но не процессору.

```
asm volatile("" ::: "memory")   — барьер КОМПИЛЯТОРА (CPU свободен)
atomic_thread_fence(seq_cst)    — барьер компилятора И процессора
mfence / dmb ish                — машинный барьер процессора
```

## 8. Спинлок vs мьютекс, и futex под капотом

### 8.1 Спинлок

Спинлок — «занятое ожидание»: крутимся в цикле, пока не захватим. Никаких
системных вызовов, но жжём CPU.

```cpp
// C++ (основной). atomic_flag — единственный гарантированно lock-free тип.
struct Spinlock {
    std::atomic_flag f = ATOMIC_FLAG_INIT;
    void lock() {
        while (f.test_and_set(std::memory_order_acquire))
            __builtin_ia32_pause();            // PAUSE: гасит spin-wait (x86)
    }
    void unlock() { f.clear(std::memory_order_release); }
};
// C: atomic_flag f; atomic_flag_test_and_set_explicit(&f, memory_order_acquire);
//    atomic_flag_clear_explicit(&f, memory_order_release);
```

**TTAS (Test-and-Test-And-Set)** — улучшение: сначала читаем (relaxed) в цикле,
и только увидев «свободно», пробуем дорогой `exchange`. Это снижает RFO-трафик
(см. §4.2): пока крутимся на чтении, линия в состоянии S и не пинг-понгует.
(Именно это ты реализуешь в упражнении `02-ttas-spinlock`.)

```cpp
struct Spinlock {
    std::atomic<bool> locked{false};
    void lock() {
        for (;;) {
            while (locked.load(std::memory_order_relaxed))
                __builtin_ia32_pause();                       // крутимся на дешёвом чтении
            if (!locked.exchange(true, std::memory_order_acquire))
                return;                                        // успели занять
        }
    }
    void unlock() { locked.store(false, std::memory_order_release); }
};
```

### 8.2 Мьютекс и futex

Мьютекс при контеншне **усыпляет** поток через ядро, не жжёт CPU. В Linux это
строится на **futex** (fast userspace mutex):

- **fast path (нет контеншна):** захват/освобождение — это атомарный CAS в
  userspace, **без** системного вызова. Поэтому неконтендный
  `pthread_mutex_lock` почти так же дёшев, как спинлок.
- **slow path (есть контеншн):** только тогда вызывается `futex(2)` —
  `FUTEX_WAIT` усыпляет поток в ядре до `FUTEX_WAKE`.

```c
/* идея glibc-мьютекса (упрощённо) */
lock:  if (CAS(&m, 0, 1)) return;            /* свободен → заняли, без syscall */
       while (atomic_exchange(&m, 2) != 0)   /* помечаем «есть ждущие» */
           futex_wait(&m, 2);                /* спим в ядре */
unlock: if (atomic_exchange(&m, 0) == 2)     /* были ждущие? */
           futex_wake(&m, 1);                /* разбудить одного */
```

### 8.3 Когда что

| | Спинлок | Мьютекс |
|--|--|--|
| Критическая секция | **очень короткая** (десятки нс) | любая |
| Контеншн | низкий | любой |
| Стоимость ожидания | жжёт CPU | спит (0% CPU) |
| Syscall | нет | только при контеншне |
| На 1 ядре под нагрузкой | **катастрофа** (крутится, пока вытеснят) | ок |

> **Правило:** в userspace по умолчанию — мьютекс (его fast path уже почти
> бесплатен). Спинлок оправдан только когда секция короче, чем стоимость двух
> переключений контекста, и ты знаешь, что держатель не будет вытеснен.
> В ядре всё иначе: там спинлоки повсеместны, потому что нельзя спать в
> атомарном контексте (см. модуль K2).

## 9. Deadlock и три его соседа

### 9.1 Четыре условия Коффмана

Deadlock возможен **тогда и только тогда**, когда выполнены **все четыре**
условия одновременно:

1. **Mutual exclusion** — ресурс держится эксклюзивно.
2. **Hold and wait** — поток держит один ресурс и ждёт другой.
3. **No preemption** — ресурс нельзя отнять, только владелец освобождает.
4. **Circular wait** — есть цикл в графе ожидания (A ждёт B, B ждёт A).

Убери **любое одно** — deadlock невозможен. На практике проще всего убирать (4).

### 9.2 Классический deadlock и лечение через lock ordering

```c
/* DEADLOCK: поток 1 берёт A→B, поток 2 берёт B→A */
/* Поток 1 */                 /* Поток 2 */
lock(A); lock(B);             lock(B); lock(A);   /* цикл → завис */
```

**Лечение — глобальный порядок блокировок (lock ordering):** всегда захватывай
мьютексы в одном и том же порядке (например, по возрастанию адреса):

```cpp
// C++ (основной): std::scoped_lock берёт оба мьютекса БЕЗ deadlock
// (внутри — алгоритм с trylock и откатом), да ещё и RAII-разблокировка.
std::mutex a, b;
{
    std::scoped_lock lk(a, b);     // порядок аргументов не важен — deadlock невозможен
    /* критическая секция */
}                                  // оба отпущены автоматически
```

```c
/* C-эквивалент: фиксированный порядок захвата вручную (по адресу) */
void lock_two(pthread_mutex_t *a, pthread_mutex_t *b) {
    if (a > b) { pthread_mutex_t *t = a; a = b; b = t; }   /* всегда один порядок */
    pthread_mutex_lock(a);
    pthread_mutex_lock(b);
}
```

В C++ всегда предпочитай RAII-обёртки (`std::scoped_lock`, `std::unique_lock`):
они освобождают мьютекс при любом выходе из области, включая исключение, —
руками `unlock()` на каждой ветке ты рано или поздно забудешь.

### 9.3 Соседи

- **Livelock** — потоки не заблокированы, но и не прогрессируют (бесконечно
  уступают друг другу). Лечится рандомизированным backoff.
- **Starvation** — поток никогда не получает ресурс (несправедливый планировщик
  / lock). Лечится fair-локами, очередями.
- **Priority inversion** — низкоприоритетный держит lock, нужный
  высокоприоритетному; средний приоритет вытесняет низкого → высокий ждёт
  бесконечно. Лечится priority inheritance (`PTHREAD_PRIO_INHERIT`). Это
  убило Mars Pathfinder в 1997 — классический кейс.

## 10. False sharing и выравнивание по линии кэша

### 10.1 Проблема

Память когерентна по **линиям 64 байта**, не по переменным (§4.1). Если две
независимые переменные, которые пишут **разные** потоки, попадают в **одну**
линию — линия пинг-понгует между ядрами, как будто это одна переменная. Логически
конфликта нет (false), но железо его создаёт (sharing).

```c
struct counters {
    atomic_long a;   /* пишет поток 0 */
    atomic_long b;   /* пишет поток 1 — в той же 64-байтной линии! */
};
/* Каждая запись b инвалидирует линию у ядра 0 и наоборот.
   Производительность падает в РАЗЫ при том, что данные независимы. */
```

### 10.2 Лечение — выравнивание и padding

Развести враждующие переменные по разным линиям:

```cpp
// C++ (основной). hardware_destructive_interference_size (C++17) — «правильный»
// размер линии кэша для платформы, обычно 64.
struct Counters {
    alignas(std::hardware_destructive_interference_size) std::atomic<long> a{0};
    alignas(std::hardware_destructive_interference_size) std::atomic<long> b{0};
};                                   // a и b гарантированно в разных линиях
// C-эквивалент: _Alignas(64) _Atomic long a;  _Alignas(64) _Atomic long b;
```

Или padding вручную (так часто делают в C и в ядре — там есть `____cacheline_aligned`):

```cpp
struct PaddedCounter {
    std::atomic<long> value{0};
    char pad[64 - sizeof(std::atomic<long>)];
};
```

Это и есть упражнение `04-false-sharing`: развести два счётчика по линиям и
**замерить** разницу (`perf c2c`, `perf stat`).

### 10.3 Как это увидеть и измерить

- `perf c2c record/report` — прямо показывает строки с false sharing и какие
  ядра конфликтуют.
- Микробенчмарк: один и тот же код с `alignas(64)` и без — разница в разы (см.
  упражнение 04). Это «замер эффекта false sharing» из трека.

> **Где встречается в реальности:** массив на-поток счётчиков/локов; head и tail
> одной очереди (их трогают разные потоки — разводи по линиям!); поля
> структуры, обновляемые разными ядрами. Узнать в коде по симптому: «алгоритмически
> распараллелили, а ускорения нет/стало хуже».

## 11. Lock-free SPSC и MPMC очереди — кульминация

Здесь мы собираем всё и **доказываем** корректность через ordering.

### 11.1 SPSC — single-producer single-consumer ring buffer

Одно ограничение (ровно один писатель и ровно один читатель) убирает почти всю
сложность: не нужен CAS, достаточно двух атомарных индексов с acquire/release.

```cpp
// C++ (основной). Это ровно то, что ты пишешь в упражнении 01-spsc-queue.
template <std::size_t CAP>            // CAP — степень двойки
class SpscQueue {
    int buf_[CAP];
    alignas(64) std::atomic<std::size_t> head_{0};  // пишет ТОЛЬКО consumer; своя линия
    alignas(64) std::atomic<std::size_t> tail_{0};  // пишет ТОЛЬКО producer; своя линия
public:
    bool push(int v) {                                            // PRODUCER
        auto t = tail_.load(std::memory_order_relaxed);           // свой — relaxed
        auto h = head_.load(std::memory_order_acquire);           // чужой — acquire
        if (t - h == CAP) return false;                           // полна
        buf_[t & (CAP-1)] = v;                                    // (A) пишем данные
        tail_.store(t + 1, std::memory_order_release);            // (B) публикуем
        return true;
    }
    bool pop(int& out) {                                          // CONSUMER
        auto h = head_.load(std::memory_order_relaxed);           // свой — relaxed
        auto t = tail_.load(std::memory_order_acquire);           // чужой — acquire
        if (h == t) return false;                                 // пуста
        out = buf_[h & (CAP-1)];                                  // (C) читаем данные
        head_.store(h + 1, std::memory_order_release);           // (D) освобождаем
        return true;
    }
};
```

> **C-эквивалент** дословно тот же: `_Atomic size_t head, tail;` и
> `atomic_load_explicit/atomic_store_explicit(..., memory_order_acquire/release)`.
> Именно так это и живёт в ядре — кольца SQ/CQ io_uring (модуль C2) суть SPSC
> между userspace и ядром.

**Доказательство корректности (это то, что требует трек):**

- *Видимость данных producer→consumer.* Запись данных (A) sequenced-before
  release-store tail (B). Consumer читает tail acquire (в `spsc_pop`); если он
  увидел `t+1`, то его acquire-чтение синхронизировалось с release-записью (B).
  Значит (A) happens-before чтение (C). Данные гарантированно видны — **гонки
  нет**, хотя `buf` не атомарный.
- *Производитель не перезапишет непрочитанное.* `push` сравнивает с `head`,
  прочитанным **acquire**; consumer двигает `head` только **после** чтения
  данных (D — release после C). Producer не догонит непрочитанный слот.
- *Свои индексы — relaxed.* Producer единственный пишет `tail`, поэтому читать
  его себе можно relaxed (когерентность одной переменной гарантирована). Только
  **чужой** индекс читается acquire.
- *false sharing.* `head` и `tail` разведены `alignas(64)` — иначе два потока
  колотят одну линию (§10).

Это эталон рассуждения «release публикует — acquire забирает».

### 11.2 MPMC — почему всё резко сложнее

Много писателей и много читателей: два producer'а могут одновременно захотеть
один слот `tail`. Простой store больше не годится — нужен **CAS** на индексах,
а вместе с CAS приходит **ABA** (§6.3) и проблема **reclamation** (§12).

Промышленный подход — **ограниченная очередь Вьюкова (bounded MPMC)** с
посекционными последовательностями (per-slot sequence numbers), которая
обходится без освобождения памяти (узлы — слоты массива) и потому свободна от
ABA:

```cpp
// C++ (основной). CAP — степень двойки; в конструкторе cells_[i].seq = i.
template <std::size_t CAP>
class MpmcQueue {
    struct Cell { std::atomic<std::size_t> seq; int val; };
    Cell cells_[CAP];
    alignas(64) std::atomic<std::size_t> enq_{0};   // позиция постановки
    alignas(64) std::atomic<std::size_t> deq_{0};   // позиция снятия
public:
    MpmcQueue() { for (std::size_t i = 0; i < CAP; ++i) cells_[i].seq.store(i); }

    bool push(int v) {
        auto pos = enq_.load(std::memory_order_relaxed);
        for (;;) {
            Cell& c = cells_[pos & (CAP-1)];
            auto seq = c.seq.load(std::memory_order_acquire);
            auto diff = (std::intptr_t)seq - (std::intptr_t)pos;
            if (diff == 0) {                                  // слот свободен ровно для нас
                if (enq_.compare_exchange_weak(pos, pos + 1,
                        std::memory_order_relaxed)) {
                    c.val = v;
                    c.seq.store(pos + 1, std::memory_order_release);   // опубликовали
                    return true;
                }
            } else if (diff < 0) {
                return false;                                 // очередь полна
            } else {
                pos = enq_.load(std::memory_order_relaxed);   // перечитали
            }
        }
    }
};
```

Симметричный `mpmc_pop`. Гениальность схемы — **нет malloc/free на горячем
пути**, поэтому нет ABA и нет hazard pointers. За это платим фиксированной
ёмкостью. Эту структуру ты доводишь в упражнениях/капстоуне.

### 11.3 lock-free ≠ wait-free

- **obstruction-free:** поток прогрессирует, если остальные «замёрзнут».
- **lock-free:** хотя бы **один** поток системы всегда прогрессирует (нет
  глобального зависания), но отдельный поток может «голодать».
- **wait-free:** **каждый** поток завершает операцию за ограниченное число
  шагов. Самая сильная и самая дорогая гарантия.

CAS-циклы дают lock-free, не wait-free: твой конкретный CAS может проваливаться
бесконечно при невезении, хотя система в целом движется.

## 12. Reclamation: как безопасно освобождать память в lock-free

Главная боль lock-free со **связанными** структурами (стек, очередь на узлах):
как освободить узел, если другой поток, возможно, прямо сейчас его читает?
`free` под ним = use-after-free.

### 12.1 Tagged pointers (версионирование)

Упаковать указатель + счётчик в двойное слово, менять CAS'ом `cmpxchg16b`.
Решает ABA дёшево, но **не** решает «когда физически освободить» (а ещё тег
конечен). Подходит для пулов фиксированного размера.

### 12.2 Hazard pointers (Майкл)

Идея: перед разыменованием узла поток **публикует** его адрес в свой
«hazard-слот» (атомарная запись, видимая всем). Тот, кто хочет освободить узел,
**сканирует** все hazard-слоты: если адрес там есть — откладывает `free` в
retire-список и попробует позже.

```
Reader:  hp[me] = node; (release/seq_cst) перечитать, что node ещё в структуре
Reclaimer: retire(node); если node нет ни в одном hp[] → free(node), иначе позже
```

- **Плюс:** ограниченная память (на поток — O(K) отложенных узлов), нет ABA.
- **Минус:** дорогая запись hazard-указателя и скан на каждое освобождение;
  сложно. Отвечает на вопрос ЭКСПЕРТ-2 трека: hazard pointers решают именно
  **safe memory reclamation** — «когда можно физически освободить узел в
  lock-free структуре».

### 12.3 RCU и epoch-based reclamation

- **RCU (Read-Copy-Update):** читатели почти бесплатны (вход/выход из read-side
  критической секции — это просто отметка), писатель публикует новую версию и
  ждёт **grace period** — момента, когда гарантированно ни один читатель не
  держит старую. Тогда освобождает старую. Доминирует в ядре Linux (см. модуль
  K2). В userspace — `liburcu`.
- **Epoch-based:** глобальный счётчик эпох; узел освобождается, когда все потоки
  прошли две эпохи. Проще hazard pointers, но «застрявший» поток задерживает
  освобождение у всех.

> Вывод: **reclamation — настоящая причина, по которой lock-free сложен.** Сам
> алгоритм очереди часто проще, чем безопасное освобождение его узлов. Поэтому
> на практике выбирают bounded-структуры без free (§11.2) или RCU.

## 13. Когда lock-free ХУЖЕ мьютекса

Зрелость инженера — не любить lock-free, а знать, когда он вреден (вопрос
ЭКСПЕРТ-4 трека).

1. **Высокий контеншн.** При сильной конкуренции CAS-цикл крутится впустую,
   линия пинг-понгует (§4.2). Хороший мьютекс с futex усыпит лишние потоки и
   суммарно даст больший throughput, чем стая ядер, жгущих CAS.
2. **Сложность = баги.** Lock-free код невероятно сложно сделать корректным
   (ABA, reclamation, ordering). Цена ошибки — невоспроизводимый прод-креш раз в
   неделю. Мьютекс на 5 строк, который очевидно правильный, почти всегда лучше
   «гениальной» lock-free структуры, которую никто в команде не может
   проверить.
3. **Латентность хвоста (tail latency).** Lock-free даёт прогресс системы, но
   **отдельный** поток может голодать → плохой p99/p999. Где важен именно хвост,
   честный fair-мьютекс предсказуемее.
4. **Нет реальных замеров.** Если ты не измерил `perf stat`/бенчмарком, что
   мьютекс — бутылочное горлышко, lock-free — преждевременная оптимизация.

> **Правильный порядок действий:** (1) сначала мьютекс; (2) измерил, что он
> узкое место; (3) уменьшил критическую секцию / шардировал локи; (4) и только
> если всё ещё горлышко и контеншн умеренный — рассматривай lock-free, с TSan и
> бенчмарком в руках.

## 14. Атомарный сборник рецептов

Прежде чем строить lock-free структуры, нужно довести до автоматизма десяток
типовых атомарных паттернов. Это «гаммы» конкурентного программиста: каждый —
короткий, но с одной неочевидной деталью ordering, на которой все спотыкаются.

### 14.1 One-shot флаг готовности (publish/subscribe)

Уже видели в §5.2 — это базовый рецепт публикации. Производственный вариант
добавляет **spin-then-block** вместо чистого busy-wait, чтобы не жечь CPU:

```cpp
std::atomic<bool> ready{false};
Data data;                                   // обычные данные

// publisher
data = compute();
ready.store(true, std::memory_order_release);
ready.notify_all();                          // C++20: разбудить ждущих (futex)

// subscriber
ready.wait(false, std::memory_order_acquire);// C++20: спать, пока ready==false
use(data);                                   // data гарантированно видна
```

`ready.wait(false)` усыпляет поток в ядре (через futex), пока значение равно
`false`, — это правильнее, чем `while(!ready.load()) ;`, который крутит CPU.
До C++20 это писали на `std::condition_variable` (§17) или futex напрямую.

### 14.2 Счётчики: relaxed для инкремента, осторожно с чтением

Для чистой статистики (важен лишь итог) — `relaxed`:

```cpp
std::atomic<uint64_t> packets{0};
packets.fetch_add(1, std::memory_order_relaxed);          // горячий путь
// ...
uint64_t total = packets.load(std::memory_order_relaxed); // когда всё стихло
```

Тонкость: если по достижении счётчиком значения ты **публикуешь** что-то ещё
(«набралось N — буди обработчик»), то операция, принимающая решение, уже не
`relaxed` — нужен `acq_rel`, иначе данные, защищённые этим условием, не
синхронизируются. И помни (§4.2): один общий счётчик под нагрузкой — точка
сериализации; для масштабирования **шардируй** (счётчик на поток/ядро) и
суммируй при чтении:

```cpp
struct alignas(64) Shard { std::atomic<uint64_t> v{0}; };  // своя линия кэша
Shard shards[NCPU];
shards[my_cpu()].v.fetch_add(1, std::memory_order_relaxed);// почти без контеншна
uint64_t sum = 0; for (auto& s : shards) sum += s.v.load(std::memory_order_relaxed);
```

### 14.3 Double-Checked Locking (DCL) — классика «как НЕ надо» и как надо

Ленивая инициализация синглтона. **Историческая ловушка:** до модели памяти
(C++03/старая Java) этот код был **сломан** и приводил к выдаче недостроенного
объекта:

```cpp
// СЛОМАНО без атомиков: другой поток видит ненулевой instance,
// но поля объекта ещё не записаны (reorder записи указателя и полей).
if (!instance) { lock(m); if (!instance) instance = new Singleton(); unlock(m); }
return instance;
```

Почему ломалось: `instance = new Singleton()` — это «выделить, сконструировать,
записать указатель». Компилятор/CPU могли записать указатель **до** конструктора
→ второй поток видит не-null `instance` и читает мусорные поля. Корректный
вариант — через `std::atomic` с release/acquire:

```cpp
std::atomic<Singleton*> instance{nullptr};
std::mutex m;
Singleton* get() {
    Singleton* p = instance.load(std::memory_order_acquire);   // 1-я проверка
    if (!p) {
        std::scoped_lock lk(m);
        p = instance.load(std::memory_order_relaxed);          // 2-я (под локом)
        if (!p) {
            p = new Singleton();                               // полностью построен
            instance.store(p, std::memory_order_release);      // публикуем ПОСЛЕ
        }
    }
    return p;
}
```

release-store гарантирует: поток, увидевший указатель через acquire-load,
увидит и полностью сконструированный объект. Но в современном C++ DCL писать
руками не нужно — есть готовые средства (14.4).

### 14.4 Ленивая инициализация правильно: `call_once` и magic statics

```cpp
// (а) Function-local static — потокобезопасна с C++11 («magic statics»):
Singleton& get() { static Singleton s; return s; }   // компилятор сам ставит барьеры

// (б) std::call_once — когда инициализация сложнее одной переменной:
std::once_flag flag;
void init_subsystem() { std::call_once(flag, [] { /* ровно один раз */ }); }
```

Оба варианта решают DCL без единого явного атомика и без гонок. **Правило:** в
C++ предпочитай (а)/(б) ручному DCL — меньше шансов ошибиться в ordering.
C-эквивалент `std::call_once` — `pthread_once(&once, init_fn)`.

### 14.5 Подсчёт ссылок: внутренности `shared_ptr` (refcount)

Это самый поучительный рецепт — он объясняет, **зачем** существует release
sequence (§5.4) и acquire-fence. Счётчик ссылок:

```cpp
// Увеличение ссылки — relaxed: новый владелец появляется из уже существующего,
// happens-before уже обеспечен самим фактом, что у тебя есть указатель.
ctrl->refs.fetch_add(1, std::memory_order_relaxed);

// Уменьшение — release: «я отпустил, все мои записи в объект завершены».
if (ctrl->refs.fetch_sub(1, std::memory_order_release) == 1) {
    // Я последний. Нужно acquire, чтобы УВИДЕТЬ записи всех других потоков,
    // сделанные ДО их release-decrement, прежде чем звать деструктор:
    std::atomic_thread_fence(std::memory_order_acquire);
    delete ctrl->object;        // безопасно: видны эффекты всех бывших владельцев
}
```

Разбор: инкремент `relaxed`, потому что добавление владельца ничего не
публикует. Декремент `release` — публикует все изменения объекта, которые делал
этот владелец. Тот, кто доводит счётчик до нуля, обязан `acquire` (через fence)
**синхронизироваться** со всеми release-декрементами остальных потоков — иначе
деструктор может не увидеть их записи и почистить объект в несогласованном
состоянии. Серия `fetch_sub` образует **release sequence**, поэтому финальный
acquire ловит их все. Именно так устроен `std::shared_ptr` в libstdc++/libc++.

## 15. Lock-free структуры на узлах: Treiber stack и Michael–Scott

SPSC/MPMC из §11 хранили данные в **массиве** (слоты — не освобождаются). Как
только узлы становятся **отдельными аллокациями** в связном списке, появляются
две новые боли: **ABA** (§6.3) и **reclamation** (§12). Разберём это на двух
канонических структурах.

### 15.1 Treiber stack — простейший lock-free стек

```cpp
template <class T>
class TreiberStack {
    struct Node { T value; Node* next; };
    std::atomic<Node*> head_{nullptr};
public:
    void push(T v) {
        Node* n = new Node{std::move(v), nullptr};
        n->next = head_.load(std::memory_order_relaxed);
        // CAS публикует n вместе с его полями → release; при провале перечитали next
        while (!head_.compare_exchange_weak(
                   n->next, n,
                   std::memory_order_release,
                   std::memory_order_relaxed))
            ; // n->next обновлён актуальным head_, пробуем снова
    }
    bool pop(T& out) {
        Node* old = head_.load(std::memory_order_acquire);  // видеть поля узла
        while (old &&
               !head_.compare_exchange_weak(
                   old, old->next,
                   std::memory_order_acquire,
                   std::memory_order_acquire))
            ; // old обновлён; перечитываем
        if (!old) return false;
        out = std::move(old->value);
        delete old;                 // ⚠ ВОТ ЗДЕСЬ живёт ABA и use-after-free!
        return true;
    }
};
```

Ordering: `push` делает release-CAS — публикует узел и его `value`/`next`. `pop`
читает `head_` с acquire — видит опубликованные поля. Логика стека верна.
**Но `delete old` в `pop` некорректен** в общем случае — см. 15.2.

### 15.2 ABA вживую на Treiber stack

```
head -> A -> B -> C
Поток 1: pop(): прочитал old=A, готовит CAS(head: A -> A->next == B). Вытеснен.
Поток 2: pop A (head=B), pop B (head=C), delete B.
         push нового узла, которому malloc вернул ТОТ ЖЕ адрес, что был у A.
         Теперь head -> A' (адрес A), а A'->next = C.
Поток 1 просыпается: CAS(head: A -> B) — УСПЕШЕН, т.к. head снова == адрес A!
         head стал B — но B уже delete'нут. Стек разрушен (use-after-free).
```

CAS сравнил **адрес**, а не «менялся ли узел». Лечение (см. §12):
- **tagged pointer:** `head_` = `{Node*, uint64 tag}`, tag++ на каждом изменении,
  CAS через `cmpxchg16b` (нужен `__int128`/`std::atomic<struct>` с DWCAS). ABA не
  пройдёт, т.к. tag различается.
- **hazard pointers / RCU / epoch** — не освобождать узел, пока кто-то на него
  смотрит; тогда `delete old` откладывается до безопасного момента.

Это упражнение `05-treiber-stack`: реализуешь стек и убеждаешься через TSan, что
без reclamation он ломается под нагрузкой, а с tagged-pointer/epoch — нет.

### 15.3 Michael–Scott queue — lock-free FIFO на узлах

Каноническая неограниченная MPMC-очередь (в основе `ConcurrentLinkedQueue` Java,
многих C++ библиотек). Идея — связный список с раздельными `head` и `tail`,
каждый двигается своим CAS, плюс «dummy»-узел, чтобы очередь никогда не была
по-настоящему пустой:

```cpp
// Скелет (без reclamation — её добавляешь hazard pointers/epoch):
void enqueue(T v) {
    Node* n = new Node{std::move(v), nullptr};
    for (;;) {
        Node* last = tail_.load(std::memory_order_acquire);
        Node* next = last->next.load(std::memory_order_acquire);
        if (last == tail_.load(std::memory_order_acquire)) {
            if (!next) {
                // tail реально последний — пытаемся прицепить n
                if (last->next.compare_exchange_weak(
                        next, n, std::memory_order_release))
                {
                    tail_.compare_exchange_strong(last, n,    // «доводим» tail
                        std::memory_order_release);
                    return;
                }
            } else {
                // tail отстал — помогаем продвинуть (lock-free helping!)
                tail_.compare_exchange_strong(last, next,
                    std::memory_order_release);
            }
        }
    }
}
```

Два урока: (1) **helping** — поток, увидевший «отставший» `tail`, двигает его за
другого, чтобы система не зависала (это и делает структуру lock-free, а не
просто obstruction-free); (2) узлы освобождаются — значит снова нужны hazard
pointers/RCU. Полная реализация с reclamation — материал капстоуна; здесь важно
понимать схему и почему она сложнее массивной MPMC из §11.2.

> **Вывод раздела:** структуры на узлах концептуально элегантны, но
> «бесплатный» `delete` в них — ловушка. На практике в проде чаще берут
> **ограниченные** структуры без освобождения (§11.2) или прячут reclamation за
> RCU. Treiber stack — отличный учебный пример именно потому, что обнажает ABA.

## 16. Масштабируемые блокировки: ticket, MCS, CLH

TTAS-спинлок (§8.1) чинит трафик **ожидания**, но не трафик **захвата**: в
момент освобождения все ждущие ядра разом делают `exchange` на одной линии —
**thundering herd** на уровне кэша, и линия пинг-понгует. Плюс TTAS
**несправедлив**: поток может голодать произвольно долго. Решение — локи, где
каждый ждущий крутится на **своей** ячейке.

### 16.1 Ticket lock — справедливый (FIFO), но всё ещё общая линия

```cpp
struct TicketLock {
    std::atomic<unsigned> next{0};   // выдаём номерки
    std::atomic<unsigned> serving{0};// какой номерок обслуживаем
    void lock() {
        unsigned my = next.fetch_add(1, std::memory_order_relaxed);  // взял номерок
        while (serving.load(std::memory_order_acquire) != my)
            cpu_pause();             // жду свой номер (FIFO-справедливость)
    }
    void unlock() {
        serving.fetch_add(1, std::memory_order_release);             // следующий
    }
};
```

Плюс: строгая очередь (FIFO), нет голодания. Минус: все крутятся на одной
переменной `serving` — при освобождении инвалидируется линия у **всех** ждущих
(хоть просыпается лишь один). На десятках ядер это всё ещё бутылочное горлышко.

### 16.2 MCS lock — каждый крутится на своей линии (масштабируемый)

MCS (Mellor-Crummey & Scott) выстраивает ждущих в **очередь узлов**: каждый
поток крутится на **своём** флаге, который освободит предшественник. Никакого
общего spin-адреса → нет cache-line bouncing, идеальная масштабируемость.

```cpp
struct McsLock {
    struct Node { std::atomic<Node*> next{nullptr}; std::atomic<bool> locked{false}; };
    std::atomic<Node*> tail{nullptr};

    void lock(Node* self) {
        self->next.store(nullptr, std::memory_order_relaxed);
        Node* pred = tail.exchange(self, std::memory_order_acq_rel); // встаю в хвост
        if (pred) {                              // очередь не пуста
            self->locked.store(true, std::memory_order_relaxed);
            pred->next.store(self, std::memory_order_release);       // пусть меня разбудят
            while (self->locked.load(std::memory_order_acquire))     // кручусь на СВОЁМ флаге
                cpu_pause();
        }
    }
    void unlock(Node* self) {
        Node* succ = self->next.load(std::memory_order_acquire);
        if (!succ) {                             // вроде я последний
            Node* expected = self;
            if (tail.compare_exchange_strong(expected, nullptr,
                    std::memory_order_release))
                return;                          // действительно последний
            while (!(succ = self->next.load(std::memory_order_acquire)))
                cpu_pause();                     // ждём, пока преемник допишет next
        }
        succ->locked.store(false, std::memory_order_release);        // бужу преемника
    }
};
```

Ключевая идея: `tail.exchange` атомарно ставит меня в конец и отдаёт
предшественника; я прошу его записать `pred->next = self` и кручусь на
**локальном** `self->locked`. Освобождающий будит ровно одного преемника, меняя
**его** флаг. Линия `locked` каждого узла приватна — масштабируется почти
линейно. **Цена:** нужен узел `Node` на каждый захват (обычно на стеке вызвавшего).

> Ядро Linux использует **qspinlock** — гибрид MCS для случая высокого
> контеншна (быстрый путь — простой TAS, медленный — MCS-очередь). CLH-lock —
> близкий родственник MCS, где крутятся на флаге *предшественника*; чуть проще,
> но хуже на NUMA. Это упражнение `06-mcs-lock`.

### 16.3 Backoff — дешёвое смягчение контеншна

Если не хочется полноценного MCS, помогает **exponential backoff**: при провале
CAS подождать растущее (рандомизированное) время, прежде чем повторить, — это
разреживает попытки и снижает bouncing.

```cpp
unsigned delay = 1;
while (!try_acquire()) {
    for (unsigned i = 0; i < delay; ++i) cpu_pause();
    if (delay < 1024) delay <<= 1;            // экспоненциальный рост
}
```

Backoff лечит **livelock** (§9.3) и сглаживает пики, но не даёт
справедливости — для неё нужен ticket/MCS.

## 17. Condition variables и futex вглубь

Атомики хороши для коротких ожиданий. Для «жди, пока выполнится сложное
условие» нужен примитив, который **усыпляет** поток, — condition variable, а под
ней в Linux — futex.

### 17.1 Правильное использование condition variable

```cpp
std::mutex m;
std::condition_variable cv;
std::queue<Task> q;

// producer
{ std::scoped_lock lk(m); q.push(task); }
cv.notify_one();

// consumer
std::unique_lock lk(m);
cv.wait(lk, [&]{ return !q.empty(); });   // ПРЕДИКАТ в цикле — критично!
Task t = q.front(); q.pop();
```

Три железных правила CV:
1. **Предикат проверяется в цикле** (`cv.wait(lk, pred)` — это `while(!pred()) wait`).
   Причина — **spurious wakeup**: `wait` может вернуться без всякого `notify`.
   Без предиката получишь обработку несуществующего события.
2. **Мьютекс держится при изменении условия и при `wait`.** `wait` атомарно
   отпускает мьютекс и засыпает, при пробуждении снова берёт. Если менять
   условие без мьютекса — **lost wakeup** (см. 17.2).
3. `notify` можно звать и без мьютекса, но обычно проще держать тот же лок.

### 17.2 Lost wakeup и spurious wakeup

- **Lost wakeup:** producer выставил условие и сделал `notify` в окно, когда
  consumer уже проверил условие (ещё ложное), но **ещё не уснул**. Сигнал
  потерян, consumer спит навсегда. Защита — менять условие **под тем же
  мьютексом**, что используется в `wait`: тогда «проверка→сон» консьюмера и
  «изменение→notify» продюсера сериализуются, окна нет.
- **Spurious wakeup:** `wait` вернулся «просто так». Это не баг, а допускаемое
  поведение (так дешевле реализовать на futex). Поэтому — всегда предикат-цикл.

### 17.3 `notify_one` vs `notify_all` и thundering herd

`notify_all` будит **всех** ждущих; они просыпаются, дерутся за мьютекс, все
кроме одного снова засыпают — **thundering herd**. Если событие позволяет
прогресс только одному потоку — буди `notify_one`. `notify_all` нужен, когда
условие могло стать истинным для нескольких (или поменялся «тип» условия).

### 17.4 Futex под капотом: машина состояний 0/1/2

`std::mutex`/`condition_variable` в glibc стоят на **futex** (§8.2). Мьютекс —
атомарное слово с тремя состояниями:

```
0 = свободен
1 = занят, ждущих нет
2 = занят, есть ждущие (нужно будить через ядро)

lock():
    if CAS(word, 0->1) success: return                 # fast path, без syscall
    while (val = exchange(word, 2)) != 0:              # помечаем «есть ждущие»
        futex(word, FUTEX_WAIT, 2)                     # спим, пока word==2
unlock():
    if exchange(word, 0) == 2:                         # были ждущие?
        futex(word, FUTEX_WAKE, 1)                     # будим одного
```

Важные детали `futex(2)`:
- `FUTEX_WAIT(addr, expected)` засыпает, **только если** `*addr == expected` в
  момент входа в ядро (атомарная проверка) — это закрывает гонку «значение
  изменилось между моей проверкой и засыпанием» (тот же lost-wakeup, но на
  уровне ядра).
- `FUTEX_WAIT` может вернуться **спонтанно** (сигнал, `EINTR`) — снова цикл.
- `FUTEX_WAKE(addr, n)` будит до `n` ждущих на этом адресе.
- futex работает с **физической** страницей, поэтому годится и для futex'ов в
  разделяемой памяти между процессами (`FUTEX_*_PRIVATE` — оптимизация для
  одного процесса). Это мост к IPC в модуле C3.

### 17.5 Priority inversion и PI-futex

Вспомни §9.3: низкоприоритетный держит лок, нужный высокоприоритетному, а
средний вытесняет низкого — высокий ждёт бесконечно. Решение — **priority
inheritance**: пока высокий ждёт лок, держатель временно получает приоритет
высокого. В Linux это `FUTEX_LOCK_PI`/`PTHREAD_PRIO_INHERIT`. В realtime-системах
(включая ядро PREEMPT_RT) PI-мьютексы обязательны. Это снова про Mars Pathfinder.

### 17.6 Когда CV не нужна

Если ждёшь готовности **дескриптора** (сокет, таймер), правильнее не CV, а
интеграция в event loop: `eventfd`, `timerfd`, `signalfd` + epoll (модули
C2/C4). CV — для ожидания **условия в памяти** внутри процесса; для I/O-событий —
поллинг дескрипторов.

## 18. Инструментарий конкурентности C++20

C++20 принёс в стандарт то, что раньше писали на futex/pthread руками. Знать
обязательно — это твой современный тулкит на этапе 2A.

### 18.1 `atomic::wait` / `notify` — futex в стандарте

```cpp
std::atomic<bool> flag{false};
flag.wait(false, std::memory_order_acquire);  // спать, пока flag==false
// ...
flag.store(true, std::memory_order_release);
flag.notify_all();                            // разбудить ждущих
```

Это futex без ручного `futex(2)`: эффективное ожидание без busy-wait. Заменяет
большинство ad-hoc спинов из §14.1.

### 18.2 `std::latch` — одноразовый обратный отсчёт

```cpp
std::latch start{N};               // N участников
// в каждом потоке:
start.count_down();                // «я готов»
start.wait();                      // ждать, пока все N не отметятся
// дальше все стартуют синхронно. Повторно использовать нельзя (one-shot).
```

Идеально для «дождаться, пока все воркеры инициализировались» или для замеров
(старт по сигналу).

### 18.3 `std::barrier` — многоразовый барьер с фазами

```cpp
std::barrier sync{N, []{ /* completion: вызовется ОДИН раз в конце каждой фазы */ }};
// в потоке, в цикле фаз:
do_phase_work();
sync.arrive_and_wait();            // ждать всех; затем completion; затем новая фаза
```

В отличие от `latch`, переиспользуется по фазам (итеративные алгоритмы:
симуляции, параллельные стадии).

### 18.4 Семафоры: `counting_semaphore` / `binary_semaphore`

```cpp
std::counting_semaphore<MAX> sem{0};
sem.release();                     // +1 (отдать «разрешение»)
sem.acquire();                     // -1, спать если 0
std::binary_semaphore sig{0};      // == counting_semaphore<1>: сигнал/событие
```

Семафор — для ограничения параллелизма (пул из K разрешений) и для сигналинга
между потоками без мьютекса. C-эквивалент — POSIX `sem_t` (`sem_wait`/`sem_post`).

### 18.5 `std::jthread` и кооперативная отмена (`stop_token`)

```cpp
std::jthread worker([](std::stop_token st) {
    while (!st.stop_requested()) {        // кооперативно проверяем отмену
        do_chunk();
    }
});                                       // деструктор jthread: request_stop()+join()
// worker.request_stop(); — попросить остановиться
```

`jthread` (C++20) сам делает `join()` в деструкторе (RAII против забытого join и
`std::terminate`) и несёт `stop_token` для **кооперативной** отмены — правильной
замены небезопасному `pthread_cancel`. Поток сам выбирает безопасные точки
выхода.

### 18.6 `std::atomic_ref` — атомарный доступ к обычной памяти

```cpp
int buffer[N];                     // обычный массив
std::atomic_ref<int> a(buffer[i]); // временно трактуем элемент как атомик
a.fetch_add(1, std::memory_order_relaxed);
```

Позволяет применять атомарные операции к данным, которые **не** объявлены
`std::atomic` (например, элемент чужого массива). Аналог `READ_ONCE`/`WRITE_ONCE`
из ядра в типобезопасной обёртке.

### 18.7 `std::atomic<std::shared_ptr>` — lock-free указатель с refcount

C++20 даёт атомарные специализации для `shared_ptr`/`weak_ptr` — корректную
публикацию и подмену умного указателя без мьютекса (то, что в C++11 делали через
свободные функции `atomic_load(&sp)`). Под капотом — ровно §14.5 + DWCAS/hazard.
Это твой готовый «RCU для одного указателя» в userspace.

## 19. Модели памяти процессоров и litmus-тесты

Финальный уровень понимания: увидеть **разницу архитектур** на минимальных
примерах — litmus-тестах. Litmus-тест — крошечная программа на 2+ потока с
вопросом «возможен ли такой исход?». Инструменты `herd7`/`litmus7` (проект
diy/herdtools) дают точный ответ по формальной модели каждого CPU.

### 19.1 SB (Store Buffer) — store→load reordering

```
x = y = 0
Поток 1:           Поток 2:
  x = 1;             y = 1;
  r1 = y;            r2 = x;
Вопрос: r1 == 0 && r2 == 0 ?
```

- **x86-TSO:** ✅ **разрешён** (store попадает в store buffer, load обгоняет).
  Это §1.1. Чтобы запретить — `seq_cst` на обе пары (или `mfence` между store и
  load).
- **ARM/POWER:** ✅ тоже разрешён, и причин ещё больше.
Это единственный reorder, который вообще допускает x86, — поэтому x86 «сильный».

### 19.2 MP (Message Passing) — основной тест release/acquire

```
x = flag = 0
Поток 1 (writer):       Поток 2 (reader):
  x = 42;                 while (flag == 0) {}
  flag = 1;               r = x;
Вопрос: r == 0 возможно ? (т.е. флаг увидели, а данные — нет)
```

- **x86-TSO:** ❌ **невозможно** даже с обычными (`relaxed`) доступами: x86 не
  переставляет store→store и load→load. Поэтому «кривой» код без acquire/release
  на x86 **работает** — и тем опаснее.
- **ARM/POWER:** ✅ **возможно**! `r == 0` наблюдается: либо writer переставил
  два store, либо reader — два load. **Лечение:** `flag` писать `release`,
  читать `acquire` (или барьеры `dmb`). Ровно это мы делали в §5.2 и в SPSC.

> Вот почему «у меня на ноуте (x86) работает» — ложное успокоение. MP-тест
> ломается на любом ARM-сервере/телефоне. **Пиши по модели, а не по x86.**

### 19.3 LB и IRIW — зачем вообще нужен `seq_cst`

**IRIW (Independent Reads of Independent Writes):**

```
x = y = 0
П1: x=1   П2: y=1   П3: r1=x; r2=y;   П4: r3=y; r4=x;
Вопрос: r1=1,r2=0  И  r3=1,r4=0  одновременно?
(т.е. П3 видит x раньше y, а П4 — y раньше x: НЕТ единого порядка записей)
```

- С `acquire/release` этот «несогласованный» исход возможен на
  **не-multi-copy-atomic** архитектурах — прежде всего **POWER**: release/acquire
  дают лишь парную синхронизацию, но **не** единый глобальный порядок.
- **Тонкость (важна для herd7):** ARMv8 и RISC-V RVWMO — *multi-copy-atomic*
  (формально с 2018), поэтому именно IRIW на них **запрещён** даже простыми/
  acquire-загрузками. Главный реальный пример, где IRIW наблюдается, — POWER.
- С `seq_cst` — **запрещён** везде: seq_cst требует единого тотального порядка
  всех seq_cst-операций, который видят все потоки одинаково.

Итого `seq_cst` нужен сверх acquire/release по двум разным причинам: (1) для
запрета IRIW на non-MCA железе (POWER); (2) для запрета store→load (SB/Деккер,
§19.1) — и **это** требует `seq_cst` даже на x86 и ARMv8. Дороже всего seq_cst на
POWER/ARM, где для глобального порядка нужны самые сильные барьеры. Это формальный
ответ на вопрос ЭКСПЕРТ-1.

### 19.4 Зависимостный порядок (consume) на ARM

```
П1: data = 42;  p.store(&data, release);
П2: q = p.load(consume);  r = *q;   // r == 42 гарантировано ПО ЗАВИСИМОСТИ
```

На ARM/POWER чтение `*q` **зависит по адресу** от `q`, и железо не может его
переупорядочить раньше загрузки `q` — синхронизация по данным «бесплатна», без
барьера. Это и хотел использовать `memory_order_consume`. Но компиляторы его не
реализуют (повышают до acquire), потому что отследить цепочку зависимостей через
оптимизатор крайне сложно. Знать — для RCU (ядро использует именно
dependency ordering через `rcu_dereference`) и для собеседований.

### 19.5 Сводная таблица моделей

| Reorder / свойство | x86-64 (TSO) | ARMv8 | POWER | RISC-V (RVWMO) |
|--------------------|:---:|:---:|:---:|:---:|
| store→store | нет | да | да | да |
| load→load | нет | да | да | да |
| load→store | нет | да | да | да |
| **store→load** | **да** | да | да | да |
| MP без барьеров ломается | нет | **да** | **да** | **да** |
| Барьер acquire/release | почти бесплатно | `ldar`/`stlr` или `dmb` | `lwsync` | `fence r,rw`/`fence rw,w` |
| Полный барьер (seq_cst) | `mfence`/`xchg` | `dmb ish` | `sync` (hwsync) | `fence rw,rw` |

Вывод: x86 прощает многое, слабые модели — почти ничего. Код, проверенный только
на x86, **обязательно** гоняй на ARM (Graviton/Apple Silicon) и под TSan, а
спорные места — через `herd7`/`litmus7`.

### 19.6 Инструменты верификации модели

- `herd7` — символьно перечисляет **все** допустимые исходы litmus-теста по
  модели выбранного CPU. «Да/нет» без догадок.
- `litmus7` — реально **прогоняет** тест миллионы раз на железе и считает, как
  часто исход наблюдается (полезно увидеть, что на ARM он действительно
  всплывает).
- CDSChecker / GenMC — модель-чекеры для C/C++11-атомиков: проверяют твою
  lock-free структуру по всем переплетениям и ordering. Это «TSan на стероидах»
  для маленьких структур — то, чем доказывают корректность всерьёз.

## 20. Инструменты

### 20.1 ThreadSanitizer (TSan) — обязателен

Динамический детектор гонок. Инструментирует обращения к памяти и отношения
happens-before, ловит data race даже если он «не выстрелил» в этом запуске.

```sh
g++ -std=c++20 -fsanitize=thread -g -O1 -pthread prog.cpp -o prog && ./prog
# (C-эквивалент: gcc -fsanitize=thread -g -O1 prog.c -o prog)
# Печатает: точную пару конфликтующих доступов, оба стека, оба потока.
```

- **Несовместим с ASan** (нельзя `-fsanitize=thread,address` вместе).
- Замедляет в 5–15×, память ×5–10. Гоняй в CI на отдельной сборке.
- Ловит **только то, что исполнилось**: гоняй с нагрузкой/стрессом. Чистый TSan
  — необходимое, но не достаточное условие корректности.

### 20.2 perf — измерять, а не гадать

```sh
perf stat -e cache-misses,cache-references ./prog   # промахи кэша
perf c2c record ./prog && perf c2c report           # false sharing: какие линии/ядра
perf stat -e cs ./prog                               # context switches (контеншн локов)
```

### 20.3 Helgrind/DRD (Valgrind)

Альтернатива TSan: детектируют гонки и нарушения порядка блокировок (lock
ordering → потенциальный deadlock). Медленнее TSan, но ловят deadlock-потенциал
до того, как он случился.

## 23. Чеклист ревью конкурентного кода

Прогоняй каждый кусок многопоточного кода по этому списку — он ловит 90%
реальных багов из этого модуля.

**Доступ к данным**
- [ ] Каждая переменная, к которой обращаются ≥2 потока и хотя бы один пишет, —
      либо `std::atomic`, либо под мьютексом, либо упорядочена happens-before.
      Иначе — data race (UB).
- [ ] Нет `volatile` в роли межпоточной синхронизации (он её не даёт, §3.2).
- [ ] Нет «benign race»: формально любой data race — UB, даже если «по логике
      безвредно». Сделай поле атомарным (можно `relaxed`).

**Ordering**
- [ ] У каждой публикации (release-store) есть парная подписка (acquire-load) на
      **той же** переменной (§5.6). Release без acquire ничего не публикует.
- [ ] `seq_cst` стоит только там, где реально нужен единый глобальный порядок
      (паттерн IRIW/Деккера, §19.3); иначе — acquire/release/relaxed.
- [ ] `relaxed` используется только для данных, чей порядок относительно других
      переменных не важен (счётчики, флаги без публикации данных).
- [ ] CAS в цикле — `compare_exchange_weak`; одиночный CAS — `strong` (§6.2).

**Структура и время жизни**
- [ ] В lock-free структуре на узлах решён вопрос reclamation (hazard
      pointers/RCU/epoch/tagged ptr) — нет «голого» `delete` снятого узла (§12, §15.2).
- [ ] Нет ABA: если CAS по указателю, который может «уйти и вернуться», — есть
      tag/версия или защита reclamation.
- [ ] `shared_ptr`/refcount: инкремент `relaxed`, декремент `release`, последний
      владелец — `acquire`-fence перед уничтожением (§14.5).

**Блокировки**
- [ ] Все мьютексы захватываются в одном глобальном порядке (lock ordering),
      либо взяты разом `std::scoped_lock` — нет шанса circular wait (§9.2).
- [ ] Мьютексы под RAII (`scoped_lock`/`unique_lock`) — освобождаются при
      исключении и на всех ветках.
- [ ] `condition_variable`: условие меняется под тем же мьютексом; `wait` — с
      предикат-циклом; `notify_one` где можно, `notify_all` где нужно (§17).
- [ ] Спинлок — только для очень коротких секций без вытеснения держателя;
      по умолчанию мьютекс (§8.3).

**Производительность**
- [ ] Враждующие по записи поля (head/tail очереди, на-поток счётчики) разнесены
      по линиям кэша `alignas(64)` — нет false sharing (§10).
- [ ] Под высоким контеншном измерено, что lock-free реально выигрывает у
      мьютекса (а не просто «звучит круто») (§13).

**Проверка**
- [ ] Код прогнан под **TSan** с нагрузкой/стрессом (не только «happy path»).
- [ ] Спорные lock-free места проверены модель-чекером (CDSChecker/GenMC) и/или
      на реальном **ARM** (не только x86) (§19).

## 24. Глоссарий ключевых терминов

- **Data race** — несинхронизированный доступ ≥2 потоков к одной ячейке, хотя бы
  один — запись, не упорядоченный happens-before и не атомарный. Это **UB**.
- **Race condition** — зависимость корректности от относительного времени
  событий. Может быть без data race (TOCTOU на атомиках) и наоборот.
- **Атомарная операция** — неделимая для других потоков; в C11/C++ ещё и несёт
  memory ordering.
- **`memory_order`** — параметр атомарной операции, задающий гарантии
  упорядочивания (relaxed / acquire / release / acq_rel / seq_cst / consume).
- **happens-before** — частичный порядок: если A happens-before B, эффекты A
  видны в B. Строится из sequenced-before (внутри потока) и synchronizes-with
  (между потоками).
- **synchronizes-with** — межпоточная связь release-операции с acquire-операцией,
  прочитавшей её значение.
- **Modification order** — единый тотальный порядок всех записей в **одну**
  атомарную переменную; согласован для всех потоков даже при `relaxed`.
- **Release sequence** — цепочка RMW после release-store; acquire, прочитавший
  любой её элемент, синхронизируется с исходным release (нужно для refcount).
- **CAS (compare-and-swap)** — атомарный примитив «сравни и поменяй»; основа
  lock-free. weak допускает ложный провал (дёшев на LL/SC), strong — нет.
- **LL/SC** — Load-Linked/Store-Conditional, реализация CAS на ARM/POWER/RISC-V;
  источник ложных провалов weak-CAS.
- **ABA** — значение указателя ушло и вернулось (A→…→A); CAS «не заметил».
  Лечится tag/версией, hazard pointers, RCU.
- **TSO (Total Store Order)** — «сильная» модель x86: единственный аппаратный
  reorder — store→load.
- **Слабая модель** — ARM/POWER/RISC-V: допускают почти все переупорядочивания
  без барьеров/зависимостей.
- **Store buffer** — очередь записей ядра, ещё не дошедших до когерентного кэша;
  причина store→load reorder и store forwarding.
- **MESI** — протокол когерентности кэша (Modified/Exclusive/Shared/Invalid);
  запись требует RFO и инвалидации копий.
- **RFO (Read-For-Ownership)** — запрос эксклюзивного владения линией кэша перед
  записью; межъядерный трафик.
- **False sharing** — независимые переменные в одной линии кэша → линия
  «пинг-понгует» между ядрами. Лечится `alignas(64)`.
- **Spinlock** — занятое ожидание (жжёт CPU); хорош для очень коротких секций.
- **Futex** — fast userspace mutex: атомик в userspace + `futex(2)` для сна в
  ядре только при контеншне. Основа `std::mutex`/`condition_variable` в Linux.
- **MCS lock** — масштабируемый спинлок: очередь узлов, каждый крутится на своём
  флаге; нет общего spin-адреса, FIFO.
- **Lost wakeup** — `notify` пришёл между проверкой условия и засыпанием
  потребителя → сигнал потерян. Лечится изменением условия под тем же мьютексом.
- **Spurious wakeup** — `wait`/`FUTEX_WAIT` вернулся без `notify`; поэтому всегда
  предикат-цикл.
- **Lock-free** — гарантирован прогресс системы (кто-то всегда движется), но не
  отдельного потока. **Wait-free** — каждый поток за ограниченное число шагов.
- **Reclamation** — безопасное освобождение памяти в lock-free структуре;
  настоящая причина их сложности. Методы: hazard pointers, RCU, epoch, tagged ptr.
- **Hazard pointer** — публикуемый указатель «я смотрю на этот узел»; reclaimer
  не освобождает узел, пока он в чьём-то hazard-слоте.
- **RCU (Read-Copy-Update)** — читатели почти бесплатны; писатель ждёт grace
  period, прежде чем освободить старую версию. Доминирует в ядре Linux.
- **Priority inversion** — низкоприоритетный держит лок, нужный высокому;
  лечится priority inheritance (PI-мьютекс).
- **Litmus-тест** — минимальная многопоточная программа с вопросом «возможен ли
  исход?»; проверяется `herd7`/`litmus7` по модели CPU.

## 25. Что дальше

C1 закрепил конкурентность и модель памяти — фундамент, на котором стоит весь
userspace-этап. Дальше:

- **C2 (epoll/io_uring):** event loop, где эти атомики связывают потоки с
  очередями готовности; SQ/CQ io_uring — это, по сути, SPSC-кольца из §11.1
  между тобой и ядром.
- **C3 (IPC, shm):** та же модель памяти, но через границу процессов —
  futex и атомики в разделяемой памяти.
- **K2 (синхронизация в ядре):** спинлоки, RCU, per-CPU — те же идеи, но где
  «спать нельзя» и спинлок становится основным инструментом.

### Мини-проект для закрепления (по желанию, но очень полезно)

Собери из кирпичей модуля **thread pool** и доведи его до приличного уровня —
это прямой выход на задачу трека «task scheduler до lock-free»:

1. Базовая версия: `N` воркеров, **одна** очередь задач под `std::mutex` +
   `std::condition_variable` (§17). Корректная остановка через `std::jthread` +
   `stop_token` (§18.5). Это эталон корректности.
2. Замени глобальную очередь на **per-worker** очереди + **work-stealing**:
   каждый воркер берёт из своей, а опустев — «крадёт» из чужой. Очередь
   воркера — это ограниченная SPSC/«deque Chase–Lev»; кража — со стороны
   `steal` с CAS. Здесь оживают §11, §6, §15.
3. Замерь: throughput и tail latency базовой версии против work-stealing на
   1/2/4/8 ядрах. Прогони `perf stat` (context switches, cache-misses) и
   `perf c2c` (нет ли false sharing на очередях — разнеси `top`/`bottom`
   индексы по линиям, §10).
4. Прогони **TSan** на стресс-нагрузке. Затем — на **ARM** (если есть доступ к
   Graviton/Apple Silicon): именно там вылезет недостающий acquire/release.
5. Сформулируй вывод: на каком профиле нагрузки work-stealing реально выиграл, а
