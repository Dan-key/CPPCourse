## 1. Syscall как контракт: ядро, glibc, ты

### 1.1 Что физически происходит при вызове

На x86-64 Linux системный вызов — это инструкция `syscall` (на 32-битных системах исторически `int 0x80`). Соглашение о регистрах **отличается** от обычного соглашения вызова функций C (System V AMD64 ABI):

| | Обычный вызов функции | Syscall |
|---|---|---|
| Аргументы 1–6 | `rdi, rsi, rdx, rcx, r8, r9` | `rdi, rsi, rdx, r10, r8, r9` |
| Номер/что вызываем | имя символа | число в `rax` |
| Возврат | `rax` | `rax` (см. ниже про ошибки) |

**Почему `r10`, а не `rcx`?** Инструкция `syscall` сама использует `rcx` (сохраняет туда адрес возврата) и `r11` (флаги). Поэтому четвёртый аргумент сдвинут в `r10` — это решение ABI, а не каприз.

**Как сообщается об ошибке.** Ядро возвращает в `rax` **малое отрицательное число = `-errno`** (например, `-EBADF`), а не `-1`. Именно glibc-обёртка делает следующий шаг: если результат в диапазоне `[-4095, -1]`, она записывает `errno = -результат` и возвращает вызывающему `-1` (или `NULL` для функций, возвращающих указатель). **Это поведение обёртки, а не ядра** — вот первая трещина между «что делает Linux» и «что делает libc».

```c
// Грубо то, что делает glibc-обёртка write():
ssize_t write(int fd, const void *buf, size_t n) {
    long ret = raw_syscall(SYS_write, fd, buf, n);
    if (ret < 0 && ret > -4096) {       // -1..-4095 — это -errno
        errno = (int)-ret;
        return -1;
    }
    return ret;
}
```

### 1.2 `syscall(2)` — обход именованной обёртки

`syscall(2)` — функция glibc, позволяющая вызвать *произвольный* номер syscall напрямую, минуя именованную обёртку (`write`, `open` и т.п.). Зачем это нужно в реальном коде:

- **Новый syscall ещё не имеет обёртки** в установленной версии glibc (новые ядра добавляют syscalls быстрее, чем glibc их оборачивает — классика для `io_uring_setup`/`io_uring_enter` на старых дистрибутивах, см. C2);
- **Нужно обойти поведение обёртки** — например, glibc может кэшировать/перехватывать вызов (так делают `fork`, `exit` — обёртки запускают доп. логику вокруг сырого syscall: atfork-хэндлеры, флаш буферов и т.д.);
- учебная цель — увидеть голый интерфейс ядра.

```c
#define _GNU_SOURCE
#include <unistd.h>
#include <sys/syscall.h>

ssize_t raw_write(int fd, const void *buf, size_t n) {
    return syscall(SYS_write, fd, buf, n);   // тот же errno-контракт, что у write()
}
```

**Важная тонкость (ЭКСПЕРТ-уровень):** `syscall(2)` — это **тоже обёртка glibc**, просто универсальная: она *всё ещё* транслирует `-errno` в `errno=...; return -1`. Чтобы увидеть по-настоящему сырое значение из `rax` (отрицательный `-errno`), нужна инлайн-ассемблерная вставка с инструкцией `syscall` напрямую — то, что делают musl/статически собранные раннеры без libc. Для повседневной работы хватает `syscall(2)`.

> **Практика 1.A:** напиши `my_write`/`my_read`/`my_close` через `syscall(2)`, прогони `strace` на бинарнике, использующем твои обёртки и обычные glibc-функции — убедись, что *номера syscalls и их аргументы идентичны*. Различие будет только в том, что glibc-версии могут переходить через PLT/vDSO (см. §4).

### 1.3 Контракт во времени: как ABI остаётся стабильным и при этом растёт

«Syscall — это контракт» верно не только в момент вызова, но и **на десятилетия**. Главное правило ядра Linux — **«не ломать userspace»** (Linus Torvalds): бинарник, собранный 20 лет назад, обязан работать на современном ядре. Отсюда — дисциплина, по которой системные вызовы **эволюционируют, не меняя** старый контракт. Это не про конкретный вызов, а про **принцип проектирования**, который ты будешь встречать в `man 2` постоянно.

- **Новый вызов вместо изменения старого.** Номер syscall и расположение его аргументов фиксированы **навсегда** — поменять нельзя. Нужна новая возможность → добавляют **новый** вызов рядом: `open` → `openat` → `openat2`, `stat` → `statx`, `clone` → `clone3` (Ф3 §8.2), `dup2` → `dup3`. Старые продолжают работать вечно.
- **Флаги с зарезервированным нулём.** Многие вызовы принимают `flags`, где «лишние» биты заранее зарезервированы под 0, а ядро **обязано** вернуть `EINVAL` на неизвестный бит. Тогда программа, собранная под новое ядро и выставившая новый флаг, на старом ядре получит честную ошибку, а не молча неверное поведение.
- **Size-версионируемые структуры** — ключевой современный приём. Вместо фиксированной (нерасширяемой) структуры новый вызов берёт **указатель + её размер**: `clone3(&args, sizeof args)`, `openat2(..., &how, sizeof how)`, `statx`. По размеру ядро понимает «версию»: пришёл размер меньше — недостающие поля считаются нулём; больше, но хвост нулевой — ок; больше с ненулевым хвостом — `E2BIG`. Так одну структуру расширяют годами, сохраняя совместимость в обе стороны.
- **Forward-compat через `ENOSYS`.** Зовёшь вызов, которого на этом (старом) ядре ещё нет → получаешь **`ENOSYS`** и откатываешься на старый путь (`io_uring_setup` → `poll`; `openat2` → `openat`). Это краеугольный приём переносимости между версиями ядра (подробно — EL1).

**Зачем это в Ф2.** Когда в `man 2` нового вызова ты видишь `size_t size` рядом с указателем на структуру или `flags` с припиской «остальные биты должны быть 0» — теперь понятно **почему**: это механизм, позволяющий ядру не ломать твой код и одновременно расти. И обратное правило **для тебя**: всегда **обнуляй** структуру перед заполнением (`memset`/`= {0}`) и не выставляй неизвестных флагов — иначе «мусор» в зарезервированном поле на будущем ядре включит функцию, которой ты не ждал.

## 2. `errno`: контракт, о котором все забывают на третьей строчке

### 2.1 Где физически живёт `errno`

`errno` — не глобальная переменная в простом смысле. Это макрос, раскрывающийся в вызов `*__errno_location()` — функции, возвращающей указатель на **per-thread** ячейку (через TLS, Ф3). Поэтому `errno` безопасен между потоками «из коробки»: у каждого потока свой `errno`.

### 2.2 Правило валидности (источник 90% багов с errno)

> **`errno` валиден ТОЛЬКО непосредственно после вызова, который явно сигнализировал ошибку** (вернул `-1`, `NULL`, и т.п., согласно описанию конкретной функции). Успешный вызов **не обязан** сбрасывать `errno` в `0`.

Отсюда классический баг:

```c
errno = 0;
int fd = open(path, O_RDONLY);   // допустим, успех, fd >= 0
if (something_else() < 0) {       // success_func, не трогает errno вовсе...
    // ...но если внутри success_func был побочный вызов, упавший и
    // ОБРАБОТАННЫЙ (без забывания errno), errno может быть "грязным"
}
if (errno != 0) {                 // ОШИБКА: errno не сброшен успешным open()
    perror("open");               // ложное сообщение про несвязанную ошибку
}
```

Правильно: **проверяй возвращаемое значение**, и только при индикаторе ошибки читай `errno`:

```c
int fd = open(path, O_RDONLY);
if (fd == -1) {                   // вот точка, где errno гарантированно валиден
    perror("open");
    return -1;
}
```

### 2.3 Печать ошибок

- `perror("open")` → `open: No such file or directory` (использует текущий `errno`);
- `strerror(errno)` — **не thread-safe** в общем случае (статический буфер в старых реализациях; в glibc современный `strerror` thread-safe для известных кодов, но переносимый код использует `strerror_r`);
- `strerror_r(errno, buf, sizeof buf)` — переносимый thread-safe вариант (но осторожно: POSIX- и GNU-версии `strerror_r` имеют разные сигнатуры — зависит от `_GNU_SOURCE`).

> **Практика 2.A:** напиши функцию `void die(const char *what)`, которая печатает `"<what>: <strerror(errno)>"` в stderr и завершает процесс с кодом, отражающим `errno` (например, `exit(errno > 0 && errno < 256 ? errno : 1)`). Используй её во всех практиках модуля вместо ручных `if`-проверок — это закладывает идиому, которая встретится в каждом примере TLPI.

## 3. `EINTR` и сигналы: вызов, который вернулся ни с чем

### 3.1 Откуда берётся `EINTR`

Если процесс **блокируется** в системном вызове (`read` на пустой пайп, `write` в заполненный буфер, `wait`, `select`...) и в этот момент приходит сигнал, для которого установлен обработчик, — ядро может **прервать** вызов: вернуть управление обработчику, а сам syscall завершить с `errno == EINTR`, **не выполнив работу**.

Это не баг ядра — это необходимость: сигнал должен быть доставлен *сейчас*, а не после того, как блокирующий вызов когда-нибудь разблокируется.

### 3.2 `SA_RESTART` — но не везде

При установке обработчика через `sigaction` с флагом `SA_RESTART`, **некоторые** прерванные syscalls ядро **автоматически перезапускает** само (без `EINTR` в userspace). Но не все — TLPI табл. 21-1 (важнейшая таблица модуля):

| Никогда не перезапускаются (даже с `SA_RESTART`) | Перезапускаются с `SA_RESTART` |
|---|---|
| `select`, `pselect`, `poll`, `ppoll`, `epoll_wait`, `epoll_pwait` | `read`, `write`, `ioctl`, `open` на медленных устройствах (пайпы, терминалы, сокеты) |
| `connect` (на сокете — особый случай, см. ниже) | `wait`, `waitpid`, `wait3/4` (на Linux — но не на старых SysV-системах) |
| таймауты вида `sleep`/`nanosleep`/`clock_nanosleep` (возвращают остаток времени) | `ioctl` для большинства устройств |

**Следствие для практики:** даже если ты ставишь `SA_RESTART` на все сигналы, твой event loop (C2) на `epoll_wait`/`select` *обязан* сам обрабатывать `EINTR` — ядро тебе тут не поможет. Это прямая связка с C4 (`signalfd`): event loop на `signalfd` избегает этой проблемы вообще, потому что сигнал превращается в обычное событие fd, а не в прерывание syscall.

**`connect()` — отдельная засада** (ЭКСПЕРТ): если `connect()` на блокирующем сокете прерван `EINTR`, соединение **не отменяется** — оно продолжает устанавливаться в фоне на уровне ядра. Дальше поведение **зависит от системы**: на Linux (это POSIX «liberal behavior») повторный `connect()` с теми же аргументами подхватит уже идущее соединение и доблокирует до его завершения — на Linux так делать **безопасно**. (Важно: `EALREADY` при повторном вызове — это про **неблокирующий** сокет; на блокирующем ядро доблокирует. В `__inet_stream_connect()` повторный вызов на блокирующем сокете снова входит в ожидание и возвращает `0` по завершении рукопожатия.) А вот на BSD/Solaris повторный вызов вернёт ошибку (`EADDRINUSE` на BSD, `EALREADY` на Solaris; `EISCONN` — если соединение уже успело установиться). Поэтому **переносимый** и рекомендованный способ — не звать `connect()` снова, а дождаться готовности на запись через `select`/`poll`/`epoll` и проверить результат через `getsockopt(SO_ERROR)`, ровно как при неблокирующем `connect`.

### 3.3 Partial I/O — отдельная (и более частая) проблема

Даже **без единого сигнала**, `read()`/`write()` **могут вернуть меньше байт, чем запрошено** — это нормальное, документированное поведение для пайпов, сокетов, терминалов (и теоретически для обычных файлов при прерывании сигналом до того, как передан хоть один байт — тогда это `EINTR`/`-1`, а вот если хоть один байт уже передан — возврат частичного количества, **не ошибка**).

Итого для надёжного цикла нужно обрабатывать **три исхода** каждого вызова: полный успех, частичный успех (повторить с остатком), `EINTR` (повторить целиком), другая ошибка (сдаться).

```c
#include <errno.h>
#include <unistd.h>

// Гарантированно читает ровно n байт или возвращает <n при EOF / ошибке (errno установлен)
ssize_t read_full(int fd, void *buf, size_t n) {
    size_t done = 0;
    while (done < n) {
        ssize_t r = read(fd, (char *)buf + done, n - done);
        if (r == 0)
            break;                  // EOF
        if (r < 0) {
            if (errno == EINTR)
                continue;           // ничего не передано — повторяем целиком
            return -1;              // настоящая ошибка
        }
        done += (size_t)r;          // частичная передача — продолжаем с остатком
    }
    return (ssize_t)done;
}

ssize_t write_full(int fd, const void *buf, size_t n) {
    size_t done = 0;
    while (done < n) {
        ssize_t w = write(fd, (const char *)buf + done, n - done);
        if (w < 0) {
            if (errno == EINTR)
                continue;
            return -1;
        }
        done += (size_t)w;
    }
    return (ssize_t)done;
}
```

glibc даёт макрос-помощник для первого случая: `TEMP_FAILURE_RETRY(expr)` (из `<unistd.h>` с `_GNU_SOURCE`) — оборачивает выражение в цикл «пока `EINTR`». Но он **не** решает partial I/O — `read_full`/`write_full` выше нужны в любом случае.

> **Практика 3.A:** возьми `read_full`/`write_full` выше. Напиши тест: дочерний процесс шлёт сигнал родителю каждые несколько миллисекунд (`SIGUSR1`, обработчик пуст), родитель в это время читает большой объём данных из пайпа от третьего процесса/потока. Без `read_full` (с голым `read`) и с `SA_RESTART`/без него — покажи разницу в поведении через `strace -e trace=read,write -f`.

## 4. vDSO: syscall без перехода в ядро

### 4.1 Проблема

`clock_gettime()`, `gettimeofday()`, `time()`, `getcpu()` вызываются **очень часто** (логирование с таймстампами, профилирование, таймауты в горячих циклах). Полноценный syscall — это переход в режим ядра и обратно: затраты порядка ста с лишним наносекунд плюс эффекты на TLB/кэши (особенно заметно после митигаций Meltdown/Spectre, которые сделали переходы дороже).

### 4.2 Решение

Ядро отображает в адресное пространство **каждого** процесса небольшую разделяемую библиотеку — **vDSO** (virtual Dynamic Shared Object) — содержащую userspace-реализации этих функций. Они читают данные о времени из отдельной read-only страницы (`vvar`), которую ядро обновляет, и считают результат **без единого перехода в kernel mode** (используя `rdtsc`/`rdtscp` и калибровочные данные).

Динамический линкер резолвит `__vdso_clock_gettime` и подобные символы из vDSO **прозрачно** — твой вызов `clock_gettime()` из glibc обычно *уже* идёт через vDSO, ты этого не видишь.

### 4.3 Как это пронаблюдать

```sh
$ cat /proc/self/maps | grep vdso
7ffd2b9fe000-7ffd2ba00000 r-xp 00000000 00:00 0  [vdso]

$ ldd /bin/ls | grep vdso
        linux-vdso.so.1 (0x00007ffff7fc4000)
```

```c
#include <sys/auxv.h>
#include <stdio.h>
int main(void) {
    printf("AT_SYSINFO_EHDR = %p\n", (void *)getauxval(AT_SYSINFO_EHDR));  // адрес vDSO
}
```

**Доказательство «бесплатности» через strace:**

```sh
$ strace -c ./prog_using_clock_gettime   # clock_gettime НЕ появится в списке syscalls —
                                          # он выполнился через vDSO, ядро не видело вызова
```

А если принудительно дёрнуть через `syscall(SYS_clock_gettime, ...)` напрямую — ты обойдёшь vDSO-резолвинг glibc и получишь настоящий переход в ядро, который strace увидит. Сравнение этих двух путей по времени (через `rdtsc` до/после в цикле на миллион итераций) — наглядная демонстрация цены syscall.

> **Практика 4.A:** замерь время 10 млн вызовов `clock_gettime(CLOCK_MONOTONIC, ...)` обычным образом (через glibc → vDSO) и через `syscall(SYS_clock_gettime, ...)` (настоящий syscall). Подтверди разницу через `strace -c` (в первом случае `clock_gettime` не появится в трейсе вообще). Это ровно вопрос «что такое vDSO» из самопроверки фундамента — и здесь у тебя будут цифры, а не просто определение.

## 5. Универсальная модель файлового I/O

### 5.1 Три уровня (TLPI рис. 5-2) — must know на всю оставшуюся жизнь с Linux

```
Процесс A                Процесс B           Системная (open file) таблица   Таблица inode
┌──────────────┐         ┌──────────────┐    ┌─────────────────────────┐    ┌──────────┐
│ fd 0 ─┐       │         │ fd 0 ─┐       │    │ запись: флаги, offset,  │    │          │
│ fd 1  │       │         │ fd 1  │       │    │ режим доступа ──────────┼───▶│  inode   │
│ fd 2  │       │         │ fd 2  │       │    └─────────────────────────┘    │  файла   │
│ fd 3 ─┼───────┼─────────┼───────┼──────▶├──▶ запись: флаги, offset, ...─────▶│          │
└───────┘       │         └───────┘       │    └─────────────────────────┘    └──────────┘
  fd-таблица                fd-таблица            open file descriptions          inodes
  (per-process)             (per-process)         (могут шариться между fd)      (глобально)
```

Три независимых уровня:

1. **Таблица файловых дескрипторов** процесса — массив, где индекс = fd, значение = указатель на запись в общесистемной таблице + флаг `FD_CLOEXEC` (close-on-exec: дескриптор автоматически закрывается при `execve`, см. §6.2; этот флаг — **на уровне дескриптора**, не шарится).
2. **Открытое файловое описание** (open file description, OFD) — общесистемная структура: текущий offset, флаги доступа (режим, в котором файл открыт: `O_RDONLY` — только чтение, `O_APPEND` — дозапись в конец и т.д.; полная таблица флагов — §5.4), счётчик ссылок. **Шарится** между fd, ссылающимися на неё — это происходит после `dup`/`dup2` (копируют дескриптор, §8.4) и `fork` (копирует всю fd-таблицу, §6.1).
3. **inode** — сам файл на диске/в памяти: метаданные, содержимое.

### 5.2 Прямые следствия (то, что обычно узнают через баги)

**`dup`/`dup2`/`fork` шарят offset и флаги доступа, но НЕ `FD_CLOEXEC`:**

`dup(oldfd)` создаёт **новый** дескриптор (наименьший свободный номер), указывающий на то же открытое файловое описание, что и `oldfd`; `dup2(oldfd, newfd)` — то же самое, но в **конкретный** номер `newfd` (предварительно закрыв его, если был занят). Оба разбираются подробно в §8.4 (основа shell-пайплайнов). Поскольку описание у них общее, общими становятся offset и флаги доступа — но не `FD_CLOEXEC`, который живёт в самой fd-таблице:

```c
int fd1 = open("file", O_RDWR);
int fd2 = dup(fd1);
write(fd1, "AAAA", 4);
write(fd2, "BBBB", 4);
// файл содержит "AAAABBBB" — offset ОБЩИЙ через open file description,
// хотя fd1 и fd2 — разные дескрипторы
```

```c
int fd = open("log", O_WRONLY | O_CREAT, 0644);
fcntl(fd, F_SETFD, FD_CLOEXEC);   // fcntl(F_SETFD): задать флаги дескриптора → fd НЕ унаследуется после execve
// но O_WRONLY/O_APPEND и offset — будут (это флаги *описания*, не дескриптора)
```

**После `fork()`** — родитель и потомок получают **разные fd**, указывающие на **одно** открытое файловое описание (offset общий!). Если оба пишут в один и тот же файл без `O_APPEND` — гонка за offset.

### 5.3 Атомарность через флаги `open()`

Два флага решают классические гонки именно на уровне ядра (атомарно), а не «лучше стараться в userspace»:

- **`O_APPEND`** — без него «перейти в конец + записать» — **две** операции (`lseek` + `write`), между которыми другой процесс может вклиниться и тоже перейти в конец. С `O_APPEND` ядро делает seek-to-end+write **одной атомарной операцией**. Это база любого общего лог-файла, к которому пишут несколько процессов.
- **`O_CREAT | O_EXCL`** — атомарная проверка-и-создание: если файл существует, `open` вернёт `EEXIST`, и **гарантированно** именно один из конкурирующих вызовов создаст файл. Классический способ сделать файл-лок между процессами (мостик в C5 — PID-файлы, хотя там есть нюансы про "протухшие" PID-файлы).

```c
// Атомарный "file lock" между процессами:
int fd = open("/var/run/myapp.lock", O_CREAT | O_EXCL | O_WRONLY, 0644);
if (fd == -1 && errno == EEXIST) {
    // кто-то уже держит лок (или "протух" — отдельная проблема, см. C5)
}
```

### 5.4 Главные флаги `open()` — справочно

| Флаг | Смысл |
|---|---|
| `O_RDONLY/O_WRONLY/O_RDWR` | режим доступа (взаимоисключающие, не битовая маска между собой) |
| `O_CREAT`, `O_EXCL` | создать; в комбинации — атомарный test-and-create |
| `O_TRUNC` | обрезать существующий файл до 0 при открытии |
| `O_APPEND` | атомарная запись в конец (см. выше) |
| `O_NONBLOCK` | неблокирующий режим (критично для C2) |
| `O_CLOEXEC` | установить `FD_CLOEXEC` атомарно при открытии (избегает гонки fork+fcntl между потоками) |
| `O_DIRECT` | минуя page cache (для своих буферов — Этап 3d, БД) |
| `O_SYNC`/`O_DSYNC` | синхронная запись (durability — Этап 3d) |

> **`umask` и режим создания.** Аргумент режима (`0644`) при создании файла **маскируется** `umask` процесса: фактические права = `mode & ~umask`. Поэтому `open(..., 0666)` обычно даёт `0644` (при `umask 022`). Сам `umask` подробно — в Ф3 (наследуется через `fork`/`exec`) и в C5 (демоны выставляют свой явно); здесь достаточно помнить, что mode в `open()` — это **запрос**, а не гарантия.

> **Практика 5.A:** напиши программу, демонстрирующую общий offset через `dup`: открой файл, `dup`, пиши попеременно через оба fd, проверь результат. Затем повтори через `fork()` без `O_APPEND` и с `O_APPEND` — покажи разницу в итоговом файле при одновременной записи родителя и потомка (используй большие записи и `usleep`, чтобы увеличить шанс гонки без `O_APPEND`).

### 5.5 `lseek` и разрежённые файлы

`lseek(fd, offset, whence)` двигает **offset** открытого файлового описания (уровень 2 из §5.1):

- `SEEK_SET` — абсолютная позиция, `SEEK_CUR` — относительно текущей, `SEEK_END` — от конца.
- Возвращает **новый** offset; `lseek(fd, 0, SEEK_CUR)` — «узнать текущую позицию», ничего не сдвигая.

**`lseek` за конец + запись создаёт ДЫРУ — разрежённый (sparse) файл.** Промежуток не занимает места на диске: ФС хранит только реально записанные блоки, дыры читаются как нули.

```c
int fd = open("sparse.bin", O_RDWR | O_CREAT | O_TRUNC, 0644);
lseek(fd, 1 << 30, SEEK_SET);   // на 1 GiB вперёд, за текущий конец
write(fd, "x", 1);              // логический размер ~1 GiB, на диске — один блок
```

`ls -l` покажет **логический** размер (1 GiB), а `du` — **реально занятое** (несколько КБ). Так устроены образы дисков, core dump'ы (§далее по треку), файлы БД.

**Найти дыры/данные:** `lseek(fd, off, SEEK_HOLE)` и `SEEK_DATA` (Linux 3.1+) перепрыгивают к следующей дыре/следующим данным — так `cp --sparse`, `tar`, `rsync` копируют разрежённые файлы, не «раздувая» их нулями.

> Гонка `lseek`+`write` против `O_APPEND` (§5.3): несколько писателей, делающих `lseek(fd, 0, SEEK_END)` + `write`, могут «наехать» друг на друга между двумя вызовами. Для дозаписи в общий файл — всегда `O_APPEND` (seek-to-end+write **одной** атомарной операцией), а не lseek-в-конец.

### 5.6 Позиционный и векторный I/O: `pread`/`pwrite`, `readv`/`writev`

**Позиционный I/O — `pread`/`pwrite`:** читают/пишут по **явному** offset, **не трогая** offset открытого описания.

```c
pread(fd, buf, n, offset);    // = lseek(offset)+read, но АТОМАРНО и без сдвига общего offset
pwrite(fd, buf, n, offset);
```

Зачем: (1) **потокобезопасность** — несколько потоков читают один и тот же fd с разных позиций без гонки за **общий** offset (он шарится через open file description, §5.1); пара `lseek+read` здесь — классическая гонка. (2) На один системный вызов меньше. Так работают БД, параллельные загрузчики, серверы файлов.

> `pwrite` на дескрипторе с `O_APPEND` **игнорирует** offset и всё равно пишет в конец (особенность Linux) — не комбинируй их.

**Векторный I/O (scatter-gather) — `readv`/`writev`:** одна операция над **несколькими** буферами.

```c
struct iovec iov[2] = {
    { hdr,  hdr_len  },        // буфер 1 — заголовок
    { body, body_len },        // буфер 2 — тело
};
writev(fd, iov, 2);           // оба буфера записаны ОДНИМ вызовом, без склейки и без промежуточного состояния
```

Зачем: записать «заголовок + тело» **без** копирования их в один общий буфер (gather) и **без** двух `write` (нет лишнего syscall и нет момента, когда записан только заголовок). `readv` симметрично разбирает входной поток по нескольким буферам (scatter). Есть `preadv`/`pwritev` (+ offset) и `preadv2`/`pwritev2` (+ флаги, напр. `RWF_NOWAIT`).

> **Связи:** `writev` как **zero-copy gather** в разрезе производительности — C6 §11.5; собственно zero-copy *между дескрипторами* (`sendfile`, `splice`, `copy_file_range`) — C6 §11 и C3 §2.5. Здесь — базовая семантика вызовов, там — выжимание скорости из неё.

### 5.7 Сколько дескрипторов можно открыть: `getrlimit`/`setrlimit`

Число одновременно открытых fd ограничено. Превысишь — `open`/`socket`/`pipe` вернут **`EMFILE`** (лимит **процесса**) или **`ENFILE`** (лимит **системы**).

```c
struct rlimit rl;
getrlimit(RLIMIT_NOFILE, &rl);     // rl.rlim_cur — мягкий лимит, rl.rlim_max — жёсткий
rl.rlim_cur = rl.rlim_max;         // поднять мягкий до жёсткого — можно без root
setrlimit(RLIMIT_NOFILE, &rl);
```

- **Мягкий** лимит (`rlim_cur`) — действующий; процесс поднимает его себе вплоть до **жёсткого** (`rlim_max`) без привилегий, выше жёсткого — только root. Именно это делает `ulimit -n` в shell.
- Текущие значения процесса: `cat /proc/self/limits`.

> **Связи:** в [Ф4](04-F4-debugging-tools.md) лимиты разбираются со стороны отладки (размер core dump через `RLIMIT_CORE` / `ulimit -c`); в C2 `RLIMIT_NOFILE` критичен для epoll-сервера на тысячи соединений (иначе `EMFILE` под нагрузкой). `getrlimit`/`setrlimit` — общий механизм (`RLIMIT_STACK`, `RLIMIT_AS`, `RLIMIT_CPU`, ...); здесь — самый частый в файловом I/O случай.

## 6. Процессы и потоки на уровне syscall — что здесь, а что дальше

Этот модуль **не повторяет** Ф3 (карта памяти, ELF — там «в основном есть») и **не повторяет** C1 (модель памяти потоков — «закрепить»). Здесь — только контракт самих вызовов, который нужен, чтобы читать `man 2` бегло.

### 6.1 `fork()` возвращает дважды

Один вызов, два возврата: `0` в потомке, PID потомка в родителе, `-1` при ошибке (например, превышен `RLIMIT_NPROC` — таблица процессов полна). Всё, что было открыто (fd, мэппинги), **копируется по семантике** (copy-on-write для памяти — Ф3), но fd-таблицы становятся **независимыми копиями**, указывающими на **общие** открытые файловые описания (см. §5.3).

### 6.2 `execve()` заменяет образ процесса целиком

PID сохраняется, но код/данные/куча/стек заменяются. **fd-таблица переживает `exec`**, кроме дескрипторов с `FD_CLOEXEC` — вот почему этот флаг важен: библиотека, открывающая внутренние fd (логи, локи), должна ставить `O_CLOEXEC`/`FD_CLOEXEC`, иначе эти fd «утекут» в любую программу, которую пользователь запустит через `system()`/`popen()`.

### 6.3 `_exit()` vs `exit()` — классический fork-баг

`exit()` (функция **libc**) делает: запускает обработчики `atexit`, **флашит буферы stdio**, затем зовёт `_exit()` (тонкая обёртка над syscall `exit_group`). `_exit()` — немедленное завершение, никакого флаша.

```c
printf("hello");           // осталось в буфере stdio (нет '\n', не выведено)
if (fork() == 0) {
    exit(0);               // ПЛОХО: потомок флашит ТОТ ЖЕ буфер "hello" — напечатается ДВАЖДЫ
}
// ХОРОШО — потомок завершается без флаша унаследованных буферов:
if (fork() == 0) {
    _exit(0);              // буфер "hello" НЕ флашится в потомке
}
```

Правило: **в дочернем процессе после `fork()`, если не делаешь `exec`, используй `_exit()`**, а не `exit()` — иначе унаследованные нефлашенные буферы stdio могут продублироваться.

### 6.4 `wait`/`waitpid` и зомби

Завершившийся процесс становится **зомби** — запись в таблице процессов остаётся, пока родитель не вызовет `wait`/`waitpid` (тогда статус считывается и запись удаляется). Не вызвал `wait` — утечка записей таблицы процессов (видна как `<defunct>` в `ps`). Макросы `WIFEXITED`/`WEXITSTATUS`/`WIFSIGNALED`/`WTERMSIG` разбирают `status`.

> Если родитель сам завершается раньше потомка — потомок «осиротевает» и усыновляется `init`/PID 1 (или ближайшим subreaper), который его и reap'ает. Зомби возникают именно когда родитель **жив**, но не вызывает `wait`.

## 7. `man(2)` как справочник: как читать страницу за 30 секунд

Структура любой man-страницы раздела 2/3 одинакова — научись находить нужное сразу:

1. **SYNOPSIS** — заголовки и feature test macros (`#define _GNU_SOURCE` и т.п.) **до** `#include`. Если функция не находится при компиляции — почти всегда не хватает нужного `_XXX_SOURCE` (`_GNU_SOURCE`, `_POSIX_C_SOURCE >= 200809L`, `_DEFAULT_SOURCE`). Это управляет видимостью прототипов из заголовков glibc — не магия, а простой набор `#ifdef` внутри `<unistd.h>` и др.
2. **RETURN VALUE** — что означает успех (`0`? неотрицательное число? указатель?) и что означает ошибка (`-1`? `NULL`?). **Без этого ты не знаешь, когда смотреть в `errno`.**
3. **ERRORS** — список `errno`-кодов с условиями. Читай выборочно: `EINTR`, `EAGAIN`/`EWOULDBLOCK`, `ENOMEM` почти всегда стоит проверять отдельно от «общей» ошибки.
4. **CONFORMING TO** — POSIX vs Linux-специфика. Важно для переносимости (не критично для тебя — целевая платформа Linux, но помогает понять, что «гарантировано», а что «так делает Linux»).
5. **NOTES** — здесь обычно прячутся реальные грабли (race conditions, устаревшие альтернативы).

> **Практика 7.A:** открой `man 2 open`, `man 2 read`, `man 2 fork`, `man 2 execve`, `man 7 signal` — для каждого найди и выпиши: (a) что означает успех/ошибка в RETURN VALUE, (b) хотя бы 3 кода из ERRORS и условие их возникновения, (c) одну деталь из NOTES, которая удивила. Это не «решение задачи» — это тренировка навыка «читаю man как справочник», которого требует трек.

## 8. Каналы (pipes) и FIFO — базовый IPC

### 8.1 `pipe()` — пара дескрипторов, однонаправленный поток

```c
#include <unistd.h>
int pipe(int pipefd[2]);   // pipefd[0] = read-end, pipefd[1] = write-end
```

`pipe()` создаёт **буфер в ядре** и возвращает два fd: `pipefd[0]` (чтение) и `pipefd[1]` (запись). Данные текут только в одном направлении: пишешь в `[1]`, читаешь из `[0]`.

**Буфер ядра** на Linux — обычно 65 536 байт (64 KiB, начиная с ядра 2.6.11). Если пишешь быстрее, чем читают — `write()` блокируется. Если pipe пуст — `read()` блокируется.

**`PIPE_BUF` — гарантия атомарности.** По POSIX запись ≤ `PIPE_BUF` байт (минимум 512, на Linux обычно 4096) **атомарна**: если несколько процессов пишут в один pipe, записи ≤ `PIPE_BUF` не перемежаются (или записаны целиком, или не записаны вовсе). Для записей > `PIPE_BUF` гарантий нет — данные могут перемешаться.

```c
#include <limits.h>
printf("PIPE_BUF = %d\n", PIPE_BUF);   // обычно 4096
```

### 8.2 Паттерн: родитель → потомок через pipe

Критически важный порядок действий:

```c
int pfd[2];
pipe(pfd);       // СНАЧАЛА pipe — ДО fork

pid_t pid = fork();
if (pid == 0) {
    /* Потомок: читает. Закрыть write-конец — ОБЯЗАТЕЛЬНО */
    close(pfd[1]);
    char buf[256];
    ssize_t n = read(pfd[0], buf, sizeof(buf));
    /* ... обработка ... */
    close(pfd[0]);
    _exit(0);
} else {
    /* Родитель: пишет. Закрыть read-конец */
    close(pfd[0]);
    write(pfd[1], "hello", 5);
    close(pfd[1]);   // сигнализируем EOF
    waitpid(pid, NULL, 0);
}
```

**Почему `pipe()` до `fork()`?** `fork()` копирует fd-таблицу: оба конца pipe будут у обоих процессов. После fork каждый процесс закрывает тот конец, который ему не нужен.

### 8.3 EOF через pipe: когда `read()` вернёт 0

`read()` вернёт 0 (EOF) только тогда, когда **все** write-концы данного pipe закрыты. Это самая частая ошибка при работе с pipe:

```c
// ОШИБКА: потомок не закрыл write-конец
if (fork() == 0) {
    // close(pfd[1]);  ← забыли
    while (read(pfd[0], buf, sz) > 0) { /* ... */ }
    // Зависнет навсегда: родитель закрыл pfd[1], но копия
    // pfd[1] у потомка всё ещё открыта — EOF никогда не придёт
    _exit(0);
}
```

Правило: после `fork()` каждый процесс должен немедленно закрыть те концы pipe, которые он не использует.

### 8.4 Перенаправление через `dup2()`: основа shell-pipeline

`dup2(oldfd, newfd)` делает `newfd` копией `oldfd`, закрывая `newfd` если он был открыт. Этим пользуется shell для `cmd1 | cmd2`:

```c
/* cmd1: перенаправить stdout в write-конец pipe */
dup2(pfd[1], STDOUT_FILENO);
close(pfd[0]);
close(pfd[1]);   // оригинал — дублирован, закрыть
execvp("cmd1", argv1);

/* cmd2: перенаправить stdin из read-конца pipe */
dup2(pfd[0], STDIN_FILENO);
close(pfd[1]);
close(pfd[0]);
execvp("cmd2", argv2);
```

После `dup2` + `close` + `exec` нет «лишних» копий концов pipe — EOF придёт корректно.

**Полный пример `cmd1 | cmd2`:**

```c
#define _GNU_SOURCE
#include <unistd.h>
#include <sys/wait.h>
#include <stdio.h>

void run_pipeline(char *cmd1[], char *cmd2[]) {
    int pfd[2];
    if (pipe(pfd) == -1) { perror("pipe"); return; }

    pid_t p1 = fork();
    if (p1 == 0) {
        dup2(pfd[1], STDOUT_FILENO);
        close(pfd[0]); close(pfd[1]);
        execvp(cmd1[0], cmd1);
        perror("exec cmd1"); _exit(1);
    }

    pid_t p2 = fork();
    if (p2 == 0) {
        dup2(pfd[0], STDIN_FILENO);
        close(pfd[1]); close(pfd[0]);
        execvp(cmd2[0], cmd2);
        perror("exec cmd2"); _exit(1);
    }

    /* Родитель: закрыть оба конца (у детей свои копии через dup2) */
    close(pfd[0]); close(pfd[1]);
    waitpid(p1, NULL, 0);
    waitpid(p2, NULL, 0);
}
```

### 8.5 FIFO — именованный канал

FIFO (named pipe) — как pipe, но имеет путь в файловой системе, поэтому пригоден для IPC между **несвязанными** процессами (не обязательно родитель-потомок).

```c
#include <sys/stat.h>
mkfifo("/tmp/myfifo", 0600);   // создать FIFO

/* Процесс A: писатель */
int fd = open("/tmp/myfifo", O_WRONLY);   // блокирует до появления читателя
write(fd, data, n);
close(fd);

/* Процесс B: читатель */
int fd = open("/tmp/myfifo", O_RDONLY);   // блокирует до появления писателя
read(fd, buf, sz);
close(fd);
```

**Поведение блокировки при `open()`:** без `O_NONBLOCK` открытие FIFO на чтение блокирует процесс до тех пор, пока кто-то не откроет тот же FIFO на запись, и наоборот. С `O_NONBLOCK` открытие на чтение без писателя возвращает fd немедленно; открытие на запись без читателя немедленно возвращает `ENXIO`.

### 8.6 Ограничения pipes

| Ограничение | Деталь |
|---|---|
| Только байтовый поток | Нет границ сообщений — читатель не знает, где заканчивается одна «запись» и начинается другая. Для сообщений — UNIX-сокет (SOCK_SEQPACKET/SOCK_DGRAM) или протокол поверх pipe |
| Только родственные процессы | Обычный pipe виден только через наследование fd. Для несвязанных — FIFO или socket |
| Ограниченный буфер | 64 KiB на Linux — при медленном читателе писатель заблокируется |
| Нет seekа | pipe не seekable (это очередь, не файл) |

> **Практика 8.A:** запусти дочерний процесс через `fork()` + `exec()` (`/bin/cat /etc/os-release` или аналог), читай его `stdout` через pipe в родителе и считай строки (количество символов `'\n'`). Выведи счётчик. Это базовый `popen()`-паттерн вручную.

## 9. `mmap` — отображение файлов и анонимная память

### 9.1 Что такое `mmap`

```c
#include <sys/mman.h>
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
int   munmap(void *addr, size_t length);
```

`mmap()` отображает файл (или анонимную память) в виртуальное адресное пространство процесса. После этого читаешь/пишешь данные через обычные указатели — ядро сáмо загружает страницы из файла при обращении (page fault) и записывает изменения обратно при необходимости (при `MAP_SHARED`).

Ядро подгружает страницы **lazily**: `mmap()` сам по себе не читает данные из файла. Первое обращение к странице вызывает page fault — ядро загружает её, выполнение продолжается прозрачно.

### 9.2 `MAP_SHARED` vs `MAP_PRIVATE`

| Флаг | Семантика |
|---|---|
| `MAP_SHARED` | Изменения видны другим процессам, отобразившим тот же файл; записываются в файл (при условии `msync()` или при завершении) |
| `MAP_PRIVATE` | Copy-on-write: изменения видны только текущему процессу, в файл не попадают. Используется загрузчиком ELF для маппинга кода/данных исполняемого файла |

```c
int fd = open("data.bin", O_RDWR);
struct stat st;
fstat(fd, &st);

/* MAP_SHARED: изменения идут в файл */
char *p = mmap(NULL, st.st_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
p[0] = 'X';   // изменит файл на диске (после msync/munmap)

/* MAP_PRIVATE: copy-on-write, файл не тронут */
char *q = mmap(NULL, st.st_size, PROT_READ | PROT_WRITE, MAP_PRIVATE, fd, 0);
q[0] = 'Y';   // видно только нам

munmap(p, st.st_size);
munmap(q, st.st_size);
close(fd);
```

### 9.3 `MAP_ANONYMOUS` — анонимная память

```c
void *buf = mmap(NULL, size, PROT_READ | PROT_WRITE,
                 MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
```

Не привязано к файлу: `fd = -1`, `offset = 0`. Начальное содержимое — нули (ядро даёт zero-filled страницы). Используется для больших аллокаций в glibc `malloc()`: когда запрос > `MMAP_THRESHOLD` (обычно 128 KiB), `malloc` запрашивает память через `mmap(MAP_ANONYMOUS)`, а не через `brk()`. При `free()` эта память немедленно возвращается ядру через `munmap()` — в отличие от `brk()`-памяти, которая не возвращается сразу.

### 9.4 Производительность: `mmap` vs `read()`/`write()`

| Сценарий | Рекомендация |
|---|---|
| Случайный доступ к большому файлу | `mmap` — нет копирования данных в userspace-буфер, ядро управляет страницами |
| Последовательное чтение большого файла | `read()` с `madvise(MADV_SEQUENTIAL)` или просто `read()` — page cache работает не хуже |
| Файл целиком в памяти, многократный доступ | `mmap` + `mlock()` (зафиксировать страницы в RAM) |
| Малые файлы (< 4 KiB) | `read()` — накладные расходы `mmap` не оправданы |
| Конкурентная запись нескольких процессов | Осторожно: `MAP_SHARED` требует внешней синхронизации |

Главное преимущество `mmap` — **zero-copy**: ядро отдаёт данные прямо из page cache в адресное пространство процесса, без копирования в промежуточный буфер `read()`.

### 9.5 `msync()` — синхронизация изменений на диск

```c
int msync(void *addr, size_t length, int flags);
// MS_SYNC — дождаться записи на диск (блокирует)
// MS_ASYNC — запланировать запись (не блокирует)
// MS_INVALIDATE — сбросить кэшированные данные (перечитать с диска)
```

Для `MAP_SHARED`: без `msync()` **не гарантировано**, что изменения попадут на диск при crash (аппаратный сбой). При штатном завершении ядро флашит dirty pages, но для надёжности нужен явный `msync(MS_SYNC)` перед критическими точками.

```c
/* Правильный паттерн для надёжной записи через mmap */
memcpy(mapped_ptr, data, size);
if (msync(mapped_ptr, size, MS_SYNC) == -1) {
    perror("msync");
}
```

### 9.6 `munmap()` — освобождение отображения

```c
munmap(addr, length);   // освобождает виртуальную память
close(fd);              // закрывает файл — ОТДЕЛЬНО от munmap!
```

`munmap()` не закрывает `fd`, который был передан в `mmap()`. Они независимы. Частичное `munmap()` легально (отображение произвольного диапазона), но `addr` должен быть выровнен на страницу.

### 9.7 `mprotect()` — изменение прав на регион

```c
#include <sys/mman.h>
int mprotect(void *addr, size_t len, int prot);
// PROT_READ | PROT_WRITE | PROT_EXEC | PROT_NONE
```

Изменяет права доступа к уже отображённому региону. Базовое применение:
- **JIT-компилятор**: аллоцировать `MAP_ANONYMOUS` с `PROT_READ|PROT_WRITE`, заполнить машинным кодом, затем `mprotect(PROT_READ|PROT_EXEC)` — теперь можно вызывать как функцию.
- **Guard pages**: `mprotect(PROT_NONE)` создаёт страницу-ловушку, доступ к которой вызывает `SIGSEGV` — используется для детекции выхода за стек или буфер.

Адрес `addr` должен быть выровнен на страницу (`PAGE_SIZE`, обычно 4096).

### 9.8 `madvise()` — подсказки ядру

```c
int madvise(void *addr, size_t length, int advice);
```

| Флаг | Действие |
|---|---|
| `MADV_SEQUENTIAL` | Ядро будет читать страницы вперёд (read-ahead) — ожидается последовательный доступ |
| `MADV_RANDOM` | Отключить read-ahead — доступ будет случайным |
| `MADV_WILLNEED` | Запустить prefetch прямо сейчас — страницы скоро понадобятся |
| `MADV_DONTNEED` | Ядро может освободить эти страницы — если они понадобятся снова, будут прочитаны заново из файла |
| `MADV_FREE` | (Linux 4.5+) страницы свободны, но содержимое ещё доступно до нехватки RAM |

```c
/* Подсказка для последовательного чтения большого файла */
madvise(p, file_size, MADV_SEQUENTIAL);
```

Это не обязательства, а подсказки — ядро может игнорировать.

### 9.9 Ловушки и нюансы

**Файл нулевой длины:** нельзя `mmap()` файл с `length = 0` — системный вызов вернёт `EINVAL`. Нужно проверять `st_size > 0` после `fstat()`.

**`MAP_FAILED`, не `NULL`:** при ошибке `mmap()` возвращает `(void *)-1` (`MAP_FAILED`), а **не** `NULL`. Классическая ошибка — сравнивать с `NULL`:

```c
void *p = mmap(NULL, size, PROT_READ, MAP_SHARED, fd, 0);
if (p == MAP_FAILED) {   // ПРАВИЛЬНО
    perror("mmap");
}
// if (p == NULL) — НЕПРАВИЛЬНО: MAP_FAILED == (void*)-1
```

**Выравнивание `offset`:** параметр `offset` должен быть кратен `PAGE_SIZE` (обычно 4096). Для файлов с произвольным смещением используй `offset & ~(PAGE_SIZE-1)` и корректируй указатель.

**`SIGBUS` при truncate под mmap:** если `ftruncate()` уменьшает файл, а mapping всё ещё покрывает «срезанные» страницы — обращение к ним вызовет `SIGBUS` (не `SIGSEGV`). Это серьёзная ловушка при конкурентном изменении размера файла.

### 9.10 `/proc/self/maps` — карта памяти процесса

```
$ cat /proc/self/maps
55a1b4c01000-55a1b4c03000 r-xp 00000000 08:01 1234567   /bin/cat
...
7f9a12000000-7f9a13000000 rw-p 00000000 00:00 0          [heap]
7ffd2b9fe000-7ffd2ba00000 r-xp 00000000 00:00 0          [vdso]
7ffd2ba03000-7ffd2ba04000 rw-p 00000000 00:00 0          [stack]
```

Каждая строка: `адрес-диапазон права offset dev inode путь`. Права: `r` читаемый, `w` записываемый, `x` исполняемый, `p` private (COW), `s` shared.

```c
/* Прочитать карту памяти программно */
FILE *f = fopen("/proc/self/maps", "r");
char line[256];
while (fgets(line, sizeof(line), f)) {
    fputs(line, stdout);
}
fclose(f);
```

> **Практика 9.A:** открой существующий текстовый файл, отобрази через `mmap(MAP_SHARED, PROT_READ|PROT_WRITE)`, измени первый байт через указатель, вызови `msync(MS_SYNC)`, закрой fd и mapping. Проверь содержимое файла через отдельный `open()+read()`. Это докажет, что изменения через указатель реально попали на диск.

## 10. Блокировки файлов

### 10.1 Зачем нужна координация

Несколько процессов могут одновременно открыть один и тот же файл и писать в него. Без координации — гонка: данные перемежаются, файл повреждается. Два механизма в Linux: `flock()` и `fcntl()` POSIX locks.

### 10.2 `flock()` — BSD-стиль блокировки

```c
#include <sys/file.h>
int flock(int fd, int operation);
// LOCK_SH — shared (read lock): несколько процессов одновременно
// LOCK_EX — exclusive (write lock): только один процесс
// LOCK_UN — снять блокировку
// LOCK_NB — или сразу, или EWOULDBLOCK (без ожидания)
```

```c
int fd = open("data.txt", O_RDWR);

/* Получить exclusive lock (или подождать) */
flock(fd, LOCK_EX);

/* ... критическая секция ... */

flock(fd, LOCK_UN);   // или просто close(fd) — тоже снимает
close(fd);
```

**Семантика наследования.** Блокировка `flock()` привязана к **open file description** (уровень 2 трёхуровневой модели, §5.1), а не к конкретному номеру fd:
- После `dup()` или `fork()` два fd ссылаются на одно описание → одна блокировка. Закрытие одного fd не снимает блокировку, пока хотя бы один fd ссылается на то же описание.
- Два отдельных `open()` → два независимых описания → независимые блокировки.

### 10.3 `fcntl()` locks — POSIX byte-range locking

```c
#include <fcntl.h>
struct flock fl = {
    .l_type   = F_WRLCK,      // F_RDLCK / F_WRLCK / F_UNLCK
    .l_whence = SEEK_SET,
    .l_start  = 0,             // начало диапазона
    .l_len    = 0,             // 0 = до конца файла
};
fcntl(fd, F_SETLKW, &fl);     // F_SETLK (неблокирующий) или F_SETLKW (блокирующий)
```

**Отличия от `flock()`:**
- Блокирует **диапазон байт** (не весь файл) — можно одновременно блокировать разные записи одного файла.
- Привязан к **паре (процесс, inode/файл)** — не к конкретному fd и не к open file description. Это критически важно:

```c
/* ЛОВУШКА в многопоточном коде: */
int fd1 = open("x", O_RDWR);
int fd2 = open("x", O_RDWR);   // другое открытие того же файла

struct flock fl = { .l_type = F_WRLCK, .l_whence = SEEK_SET, .l_start=0, .l_len=0 };
fcntl(fd1, F_SETLK, &fl);  // получили lock через fd1

close(fd2);   // ВНИМАНИЕ: закрытие ЛЮБОГО fd к этому файлу снимает ВСЕ fcntl-locks
              // этого процесса на файл, даже если lock взят через fd1!
```

Это задокументированное поведение POSIX. В многопоточном коде, где потоки могут независимо открывать и закрывать файлы, `fcntl()` locks крайне опасны. Современное лечение — **OFD-локи** (`F_OFD_SETLK`/`F_OFD_SETLKW`, Linux 3.15+): они привязаны к open file description, а не к процессу, и лишены этой ловушки с закрытием «чужого» fd.

**Проверить, кто держит lock:**

```c
struct flock fl = { .l_type = F_WRLCK, .l_whence = SEEK_SET, .l_start=0, .l_len=0 };
fcntl(fd, F_GETLK, &fl);
if (fl.l_type == F_UNLCK) {
    printf("файл не заблокирован\n");
} else {
    printf("файл заблокирован процессом PID=%d\n", (int)fl.l_pid);
}
```

### 10.4 Ловушки и ограничения

**`flock()` не работает через NFS.** Исторически `flock()` не передаётся по сети — на NFS-смонтированных ФС блокировка может быть тихо проигнорирована. `fcntl()` locks работают по NFS с оговорками (зависит от версии NFS и настроек nlm/nlockmgr).

**Advisory locks, не mandatory.** На Linux `flock()` и `fcntl()` — **advisory**: ядро не запрещает другим процессам работать с файлом без взятия lock. Все участники должны соблюдать протокол. Mandatory locking (через `MS_MANDLOCK` на mountpoint + специальные биты разрешений) на практике не используется.

**Hardlinks и разные mountpoints:** ни `flock()`, ни `fcntl()` не защищают от доступа через hardlink или другой mountpoint того же устройства.

### 10.5 PID-файл — single-instance guard

Классический паттерн для демонов: гарантировать, что запущен только один экземпляр.

```c
#include <sys/file.h>
#include <fcntl.h>
#include <stdio.h>
#include <unistd.h>
#include <errno.h>

#define PID_FILE "/var/run/myapp.pid"

int acquire_pid_lock(void) {
    int fd = open(PID_FILE, O_CREAT | O_RDWR, 0644);
    if (fd == -1) return -1;

    /* Неблокирующий exclusive lock */
    if (flock(fd, LOCK_EX | LOCK_NB) == -1) {
        if (errno == EWOULDBLOCK) {
            fprintf(stderr, "already running\n");
        }
        close(fd);
        return -1;
    }

    /* Записать свой PID */
    char buf[32];
    int n = snprintf(buf, sizeof(buf), "%d\n", getpid());
    ftruncate(fd, 0);
    write(fd, buf, (size_t)n);
    /* fd оставить открытым — lock держится пока fd открыт */
    return fd;
}
```

**Почему это надёжно:** при завершении процесса (в том числе при crash, SIGKILL) ядро автоматически закрывает все fd → `flock()` lock снимается → следующий экземпляр сможет взять lock. Нет «протухшего» lock-файла, который нужно вручную чистить.

### 10.6 Advisory vs mandatory locking

Linux поддерживает mandatory locking: монтировать с `MS_MANDLOCK`, выставить `chmod g+s, g-x` на файл — тогда ядро будет проверять lock при каждом `open`/`read`/`write`. На практике эта функциональность deprecated (у неё сложная семантика и баги), не используется. Для реального IPC используют POSIX semaphores, futex, или просто `flock()` с advisory семантикой.

> **Практика 10.A:** реализуй функции `acquire_lock(path)` → `int fd` и `release_lock(fd)`. В `acquire_lock`: `open(O_CREAT|O_RDWR)` + `flock(LOCK_EX|LOCK_NB)`, при `EWOULDBLOCK` вернуть `-1`. В `release_lock`: `close(fd)` (flock снимается автоматически). Запусти два экземпляра конкурентно — второй должен получить `-1`.

## 11. Операции с директориями и ФС

### 11.1 `opendir` / `readdir` / `closedir`

```c
#include <dirent.h>
DIR *opendir(const char *path);
struct dirent *readdir(DIR *dirp);   // NULL при ошибке или конце
int closedir(DIR *dirp);
```

`struct dirent` основные поля:

```c
struct dirent {
    ino_t          d_ino;    /* номер inode */
    unsigned char  d_type;   /* тип записи (не всегда доступен) */
    char           d_name[]; /* имя файла, null-terminated */
};
```

**`d_type` не всегда доступен.** На FAT, некоторых сетевых ФС, XFS с определёнными конфигурациями `d_type == DT_UNKNOWN`. Всегда предусматривай fallback через `lstat()`:

```c
DIR *d = opendir(path);
struct dirent *e;
while ((e = readdir(d)) != NULL) {
    if (strcmp(e->d_name, ".") == 0 || strcmp(e->d_name, "..") == 0)
        continue;

    unsigned char type = e->d_type;
    if (type == DT_UNKNOWN) {
        /* Fallback: используй lstat */
        struct stat st;
        char fullpath[PATH_MAX];
        snprintf(fullpath, sizeof(fullpath), "%s/%s", path, e->d_name);
        lstat(fullpath, &st);
        if (S_ISDIR(st.st_mode))  type = DT_DIR;
        else if (S_ISREG(st.st_mode)) type = DT_REG;
        else if (S_ISLNK(st.st_mode)) type = DT_LNK;
    }
    /* ... */
}
closedir(d);
```

### 11.2 `stat()` / `lstat()` / `fstat()` — метаданные файла

```c
#include <sys/stat.h>
int stat(const char *path, struct stat *st);   // следует symlink
int lstat(const char *path, struct stat *st);  // НЕ следует symlink (возвращает данные о самой ссылке)
int fstat(int fd, struct stat *st);            // по fd (атомарнее, нет TOCTOU)
```

Основные поля `struct stat`:

```c
struct stat {
    mode_t st_mode;    /* тип + права: S_ISREG/S_ISDIR/S_ISLNK/S_ISCHR/S_ISBLK */
    off_t  st_size;    /* размер в байтах (для регулярных файлов) */
    time_t st_mtime;   /* время последней модификации */
    nlink_t st_nlink;  /* число жёстких ссылок */
    uid_t  st_uid;     /* владелец */
    ino_t  st_ino;     /* номер inode */
};
```

Макросы проверки типа:

```c
S_ISREG(st.st_mode)   /* обычный файл */
S_ISDIR(st.st_mode)   /* директория */
S_ISLNK(st.st_mode)   /* символическая ссылка (только с lstat!) */
S_ISCHR(st.st_mode)   /* символьное устройство */
S_ISBLK(st.st_mode)   /* блочное устройство */
```

`fstat()` по fd безопаснее: не уязвим к TOCTOU (см. §11.3) и не требует разрешения на поиск в директории.

### 11.3 TOCTOU — Time-Of-Check-Time-Of-Use

Классическая race condition при работе с файлами:

```c
/* НЕБЕЗОПАСНО: race condition */
if (access(path, W_OK) == 0) {     // ← check
    int fd = open(path, O_WRONLY); // ← use: между этими строками
                                   //   атакующий может подменить path
                                   //   на symlink к /etc/passwd!
}
```

Между `access()` и `open()` другой процесс (или атакующий) может изменить что ссылается на `path` — например, подставить symlink на привилегированный файл.

**Лечение:** открывай файл сразу через `open()`, затем проверяй через `fstat()`:

```c
/* БЕЗОПАСНО: */
int fd = open(path, O_WRONLY | O_NOFOLLOW);  // O_NOFOLLOW: не следовать symlink
if (fd == -1) { /* ошибка или symlink */ }
struct stat st;
fstat(fd, &st);
if (!S_ISREG(st.st_mode)) { close(fd); /* не файл */ }
/* Теперь работаем с fd — атомарно, без TOCTOU */
```

`O_NOFOLLOW` — флаг `open()`, который возвращает `ELOOP` если последний компонент пути является symlink. Защищает от symlink-подмен в check-less коде.

### 11.3.1 Глубже: `*at()`-семейство и `O_PATH` — пути без гонок «по дороге»

`open + fstat` выше закрывает гонку на **последнем** компоненте пути. Но в `"/a/b/c"` атакующий может подменить и **промежуточный** каталог — между разрешением `a`, `b` и `c`, — а `O_NOFOLLOW` ловит только финальный symlink. Корень проблемы концептуальный: каждый вызов со **строковым путём** заново проходит его от корня, и **каждый** проход — новый шанс на гонку.

Решение — **привязать путь к уже открытому каталогу-дескриптору**. Поэтому почти у каждого файлового вызова есть `*at()`-версия с аргументом `dirfd`:

```c
int dfd = open("/safe/dir", O_RDONLY | O_DIRECTORY);   // разрешили каталог ОДИН раз
int fd  = openat(dfd, "config", O_RDONLY);             // путь относительно dfd
fstatat(dfd, "config", &st, 0);
unlinkat(dfd, "config", 0);
renameat(dfd, "a", dfd, "b");
// специальное значение AT_FDCWD как dirfd → поведение обычного open (от текущего каталога)
```

`dfd` указывает на **inode** каталога, а не на имя: даже если кто-то переименует `/safe/dir`, твой дескриптор смотрит на тот же физический каталог. Так весь обход дерева становится **race-free** — открываешь каталог за каталогом дескрипторами и работаешь относительно них, а не передаёшь строку «от корня» каждый раз (это ещё и быстрее: путь не резолвится заново).

**`O_PATH`** — особый режим `open()`: возвращает дескриптор, который **ссылается** на объект, но **не открывает** его для чтения/записи. Им нельзя `read`/`write`, зато можно использовать как `dirfd` в `*at()`, делать `fstat`, обращаться через `/proc/self/fd/NN`. Это «безопасный хэндл на путь»: разрешил путь один раз — дальше оперируешь стабильным дескриптором.

**`openat2()`** (Linux 5.6+) доводит идею до полноценной защиты: принимает `struct open_how` с флагами `RESOLVE_*` — `RESOLVE_NO_SYMLINKS`, `RESOLVE_BENEATH` (запретить выход за пределы `dirfd`), `RESOLVE_NO_MAGICLINKS`, — и ядро **само** гарантирует, что разрешение пути не вылезет наружу (защита от path traversal). Заметь: `open_how` + её размер — это ровно тот **size-версионируемый** интерфейс из §1.3.

> **Вывод-концепт:** имя файла — это *слабая* ссылка (его можно переподвесить под тобой), а **дескриптор — сильная** (он держит сам объект). Безопасная работа с путями = «разрешил один раз → удерживай дескриптор → ходи через `*at()`», а не «дёргай строкой `open("/a/b/c")` повторно». Это закрывает TOCTOU целиком, а не только на последнем компоненте.

### 11.4 `mkdir()` / `rmdir()` / `rename()` — атомарность

```c
mkdir("newdir", 0755);
rmdir("emptydir");              // только пустые директории
rename("tmpfile", "target");    // атомарная замена
```

**`rename()` атомарна** — это гарантировано POSIX и реализовано в Linux. `rename("new.tmp", "target")` либо полностью заменит `target` на `new.tmp`, либо ничего не произойдёт. Это основа паттерна «атомарного обновления файла»:

```c
/* Атомарная запись файла без риска частично записанного состояния */
int tmp_fd = open("target.tmp", O_WRONLY | O_CREAT | O_TRUNC, 0644);
write(tmp_fd, data, size);
fsync(tmp_fd);    // сначала данные на диск
close(tmp_fd);
rename("target.tmp", "target");   // потом атомарная замена
```

**Ограничение:** `rename()` вообще не работает между разными файловыми системами — возвращает `EXDEV` (дело не в атомарности: операция просто не выполняется). Чтобы «переместить» файл между ФС, нужен `copy + unlink`. Поэтому для паттерна атомарной замены временный файл создавай **в той же директории**, что и цель, — тогда `rename()` гарантированно остаётся в пределах одной ФС.

**Долговечность (durability) этого паттерна.** `rename()` атомарна для **видимости**, но чтобы замена пережила **сбой питания**, одного `fsync(tmp_fd)` мало — после `rename` нужно ещё `fsync` **каталога**, иначе сама новая запись в каталоге (ссылка на `target`) может потеряться при крахе, хотя данные файла уже на диске:

```c
int dfd = open(".", O_RDONLY | O_DIRECTORY);   // каталог цели
/* ... write(tmp) → fsync(tmp) → close(tmp) → rename("target.tmp", "target") ... */
fsync(dfd);     // зафиксировать саму запись каталога (иначе rename может «откатиться» при сбое)
close(dfd);
```

- **`fsync`** сбрасывает на диск **данные и метаданные** файла; **`fdatasync`** — только данные (+ минимум метаданных, нужных для чтения), дешевле, когда точные размер/время не важны.
- Без `fsync` `write()` лишь кладёт данные в **page cache** (отложенная запись, write-back) — на диск они попадут «когда-нибудь», и при крахе теряются. Глубже про кэш записи, `O_DIRECT` и `O_SYNC`/`O_DSYNC` — Этап 3d (БД и хранилища).

### 11.5 Рекурсивный обход: `nftw()` или вручную

**`nftw()` (nftw = file tree walk):**

```c
#include <ftw.h>
int nftw(const char *path, callback_fn fn, int nopenfd, int flags);
```

```c
int walk_cb(const char *fpath, const struct stat *sb,
            int typeflag, struct FTW *ftwbuf) {
    if (typeflag == FTW_F)   printf("файл: %s (%lld байт)\n", fpath, (long long)sb->st_size);
    if (typeflag == FTW_D)   printf("дир:  %s\n", fpath);
    return 0;   // ненулевое значение останавливает обход
}

nftw("/home/user", walk_cb, 16 /* макс. открытых fd */, FTW_PHYS /* не следовать symlinks */);
```

`FTW_PHYS` — не следовать symlinks (эквивалент `lstat()` вместо `stat()`). `nopenfd` — сколько fd nftw может держать открытыми одновременно.

**Ручной обход** для полного контроля:

```c
void walk(const char *path) {
    DIR *d = opendir(path);
    if (!d) return;
    struct dirent *e;
    char child[PATH_MAX];
    while ((e = readdir(d)) != NULL) {
        if (strcmp(e->d_name,".") == 0 || strcmp(e->d_name,"..") == 0) continue;
        snprintf(child, sizeof(child), "%s/%s", path, e->d_name);
        struct stat st;
        lstat(child, &st);
        if (S_ISDIR(st.st_mode)) walk(child);   // рекурсия
        else if (S_ISREG(st.st_mode)) { /* обработать файл */ }
    }
    closedir(d);
}
```

### 11.6 `inotify` — уведомления об изменениях ФС

```c
#include <sys/inotify.h>
int inotify_init1(int flags);           // IN_NONBLOCK | IN_CLOEXEC
int inotify_add_watch(int fd, const char *path, uint32_t mask);
// mask: IN_CREATE | IN_DELETE | IN_MODIFY | IN_MOVED_FROM | IN_MOVED_TO
```

```c
int ifd = inotify_init1(IN_CLOEXEC);
inotify_add_watch(ifd, "/tmp/watched", IN_CREATE | IN_DELETE | IN_MODIFY);

/* read() блокирует до события */
char buf[4096] __attribute__((aligned(__alignof__(struct inotify_event))));
ssize_t n = read(ifd, buf, sizeof(buf));
struct inotify_event *ev = (struct inotify_event *)buf;
printf("событие: mask=0x%x, name=%s\n", ev->mask, ev->name);
```

`read()` на inotify fd возвращает один или несколько `struct inotify_event` подряд (variable-length из-за поля `name`). Этот fd можно добавить в `epoll`/`poll` — это ключевой мост в C4: сигналы и ФС-события объединяются в одном event loop через fd-интерфейс.

### 11.7 Временные файлы безопасно

```c
/* ПРАВИЛЬНО: mkstemp — атомарно создаёт и открывает уникальный файл */
char tmpl[] = "/tmp/myapp_XXXXXX";
int fd = mkstemp(tmpl);   // tmpl модифицируется — имя файла
/* tmpl теперь содержит реальное имя, fd открыт */
unlink(tmpl);   // удалить после открытия, если имя не нужно

/* ПРАВИЛЬНО: mkdtemp — то же для директорий */
char dirtmpl[] = "/tmp/myapp_XXXXXX";
char *dir = mkdtemp(dirtmpl);

/* НЕПРАВИЛЬНО: tmpnam/tempnam — не использовать */
char *tmp = tmpnam(NULL);   // race condition между tmpnam и open
```

`mkstemp()` возвращает открытый fd атомарно — нет TOCTOU. Паттерн `mkstemp()` + `unlink()` создаёт **анонимный временный файл**: fd открыт, но имя уже удалено. Файл будет удалён автоматически при закрытии fd (или при завершении процесса).

## 12. Сигналы — глубже

> Базовые сведения о сигналах были в §3 (EINTR и обработчики). Этот раздел — механизм изнутри.

### 12.1 `sigaction()` vs `signal()`

```c
#include <signal.h>
int sigaction(int signum, const struct sigaction *act, struct sigaction *oldact);
```

`signal()` — устаревшая функция с **разной исторической семантикой**. В System V после доставки сигнала обработчик сбрасывался в `SIG_DFL` («ненадёжные сигналы»: следующий такой же сигнал мог убить процесс до того, как обработчик переустановит себя). BSD это исправил — обработчик **остаётся** установленным, а прерванные syscalls автоматически перезапускаются (эквивалент `SA_RESTART`). На Linux glibc по умолчанию даёт **BSD-семантику** (обёртка `signal()` вызывает `sigaction()` с нужными флагами), но «голый» системный вызов ядра — System V. Из-за этого разнобоя `signal()` не используют в новом коде — только `sigaction()`, у которого семантика задана явно.

`struct sigaction`:

```c
struct sigaction {
    void (*sa_handler)(int);                      // простой обработчик
    void (*sa_sigaction)(int, siginfo_t *, void*); // расширенный (с SA_SIGINFO)
    sigset_t sa_mask;    // маска сигналов, блокируемых на время обработчика
    int sa_flags;
};
```

> `sa_handler` и `sa_sigaction` на самом деле объединены в **union** (общая память) — задавать оба одновременно нельзя. Выбор делает флаг `SA_SIGINFO`: с ним ядро смотрит на `sa_sigaction`, без него — на `sa_handler`.

Ключевые флаги `sa_flags`:

| Флаг | Действие |
|---|---|
| `SA_RESTART` | Автоматически перезапускать syscalls, прерванные этим сигналом (где возможно) |
| `SA_SIGINFO` | Использовать `sa_sigaction` вместо `sa_handler` — расширенная инфо через `siginfo_t` |
| `SA_NODEFER` | Не блокировать сам сигнал во время обработчика (по умолчанию блокируется) |
| `SA_RESETHAND` | Сбросить обработчик в `SIG_DFL` после первой доставки |
| `SA_NOCLDWAIT` | (для SIGCHLD) не создавать зомби при завершении потомков |

```c
struct sigaction sa = {
    .sa_handler = my_handler,
    .sa_flags   = SA_RESTART,
};
sigemptyset(&sa.sa_mask);
sigaddset(&sa.sa_mask, SIGUSR1);   // заблокировать SIGUSR1 на время обработчика
sigaction(SIGTERM, &sa, NULL);
```

### 12.2 Async-signal-safety — что можно делать в обработчике

Обработчик сигнала может прерваться **в любой момент** выполнения основного кода, в том числе внутри `malloc()`, `printf()`, любого mutex. Поэтому в обработчике можно вызывать **только async-signal-safe** функции — список в `man 7 signal-safety`.

**Нельзя в обработчике:**
- `malloc()` / `free()` — внутри может быть взят mutex
- `printf()`, `fprintf()`, `fwrite()` — буферизация через глобальные структуры
- любой `pthread_mutex_lock()` — deadlock если прерван в момент владения мьютексом
- `exit()` — флашит буферы

**Можно:**
- `write()` (но не `printf()`) — напрямую в fd
- изменить `volatile sig_atomic_t` переменную
- `_exit()`, `kill()`, `sigprocmask()`
- `open()`, `read()`, `close()` — async-signal-safe syscalls

### 12.3 Self-pipe trick

Проблема: event loop использует `select()`/`poll()`/`epoll_wait()` — они не перезапускаются с `SA_RESTART`. Сигнал прерывает их с `EINTR`, но event loop не знает, что именно произошло.

**Решение — self-pipe:**

```c
int selfpipe[2];
pipe(selfpipe);
/* Сделать запись-конец неблокирующим */
fcntl(selfpipe[1], F_SETFL, O_NONBLOCK);

void signal_handler(int sig) {
    /* async-signal-safe: write() и изменение sig_atomic_t — OK */
    char byte = (char)sig;
    write(selfpipe[1], &byte, 1);   // 1 байт в pipe — атомарно, не блокирует
}

/* В event loop: добавить selfpipe[0] в epoll/poll/select */
/* При событии на selfpipe[0]: read() вынимает байт(ы), обрабатываем сигнал */
```

Сигнал превращается в событие fd — корректно объединяется с другими fd в event loop. Это канонический паттерн до появления `signalfd()`.

### 12.4 `signalfd()` — сигналы как fd-события (Linux)

```c
#include <sys/signalfd.h>
int signalfd(int fd, const sigset_t *mask, int flags);
// fd = -1: создать новый fd
// flags: SFD_NONBLOCK | SFD_CLOEXEC
```

```c
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGTERM);
sigaddset(&mask, SIGUSR1);

/* СНАЧАЛА заблокировать сигналы нормальной доставкой */
sigprocmask(SIG_BLOCK, &mask, NULL);

/* Затем создать signalfd */
int sfd = signalfd(-1, &mask, SFD_NONBLOCK | SFD_CLOEXEC);

/* В event loop: read() когда sfd готов */
struct signalfd_siginfo info;
read(sfd, &info, sizeof(info));
printf("получен сигнал %u от PID %u\n", info.ssi_signo, info.ssi_pid);
```

Критически важно: перед созданием `signalfd` нужно **заблокировать** соответствующие сигналы через `sigprocmask()`. Иначе сигнал будет обработан своей обычной диспозицией (для `SIGTERM` по умолчанию — завершит процесс) **раньше**, чем ты успеешь прочитать его из `signalfd`, и через fd он не придёт вовсе. Блокировка как раз и перехватывает сигнал у штатной доставки, оставляя его pending, чтобы `signalfd` мог отдать его как событие fd. (`SIGKILL`/`SIGSTOP` через `signalfd` получить нельзя — они молча игнорируются в маске.)

`signalfd` элегантнее self-pipe trick: не нужен вспомогательный pipe, `siginfo_t` доступна без `SA_SIGINFO`, интегрируется в `epoll` напрямую. Мост в C4.

### 12.5 Маскирование сигналов

```c
sigset_t set, oldset;
sigemptyset(&set);
sigaddset(&set, SIGINT);

/* Заблокировать сигналы */
sigprocmask(SIG_BLOCK, &set, &oldset);   // &oldset — сохранить старую маску
/* ... критическая секция ... */
sigprocmask(SIG_SETMASK, &oldset, NULL); // восстановить
```

В многопоточном коде вместо `sigprocmask()` использовать `pthread_sigmask()` — идентичный интерфейс, но работает на уровне потока, а не процесса. `sigprocmask()` в потоке имеет implementation-defined поведение на уровне POSIX (на Linux работает как `pthread_sigmask`, но это деталь реализации).

**`sa_mask` в `sigaction`:** при установке через `sigaction` поле `sa_mask` блокирует указанные сигналы **атомарно** на время выполнения обработчика. Это надёжнее, чем `sigprocmask()` внутри обработчика.

### 12.6 Pending signals

```c
sigset_t pending;
sigpending(&pending);
if (sigismember(&pending, SIGTERM)) {
    printf("SIGTERM ожидает доставки\n");
}
```

Стандартные (не-realtime) сигналы **не очерёдуются**: если `SIGINT` уже pending и придёт второй `SIGINT` — он потеряется. Realtime-сигналы (`SIGRTMIN..SIGRTMAX`) **очерёдуются** — каждый экземпляр доставляется отдельно, независимо от способа отправки (даже обычным `kill()`). `sigqueue()` нужен не для самого факта очереди, а чтобы вдобавок приложить к сигналу данные (`union sigval`).

```c
/* Отправить realtime-сигнал с данными */
union sigval sv = { .sival_int = 42 };
sigqueue(target_pid, SIGRTMIN + 1, sv);

/* Получить в обработчике через SA_SIGINFO */
void handler(int sig, siginfo_t *info, void *ctx) {
    printf("данные: %d\n", info->si_value.sival_int);
}
```

### 12.7 Отправка сигналов

```c
kill(pid, SIGTERM);         // конкретному процессу
kill(0, SIGTERM);           // всей своей группе процессов
kill(-1, SIGTERM);          // всем процессам (кроме init) — осторожно
kill(-pgid, SIGTERM);       // всей группе pgid

killpg(pgid, SIGTERM);      // то же, что kill(-pgid, ...) но явнее

raise(SIGTERM);             // самому себе (эквивалент kill(getpid(), ...))
```

`kill()` требует прав: можно посылать сигналы только процессам с тем же euid, или если у тебя `CAP_KILL`. Исключение: `SIGCONT` можно посылать в своей сессии.

### 12.8 `SIGCHLD` — reap без race condition

Проблема: несколько потомков могут завершиться почти одновременно, а один `SIGCHLD` может представлять несколько завершений.

```c
void sigchld_handler(int sig) {
    (void)sig;
    int status;
    pid_t pid;
    /* Цикл с WNOHANG — забираем всех завершившихся */
    while ((pid = waitpid(-1, &status, WNOHANG)) > 0) {
        if (WIFEXITED(status))
            /* потомок завершился нормально */;
        else if (WIFSIGNALED(status))
            /* потомок убит сигналом */;
    }
    /* Ошибки: ECHILD = нет завершившихся потомков — нормально */
}

struct sigaction sa = {
    .sa_handler = sigchld_handler,
    .sa_flags   = SA_RESTART | SA_NOCLDSTOP,  // не уведомлять о SIGSTOP/SIGCONT
};
sigemptyset(&sa.sa_mask);
sigaction(SIGCHLD, &sa, NULL);
```

Альтернатива — `SA_NOCLDWAIT`: ядро не создаёт зомби, потомки reap'аются автоматически. Но тогда нельзя получить exit status.

> **Практика 12.A:** реализуй graceful shutdown через SIGTERM. Объяви `volatile sig_atomic_t running = 1;`, обработчик `shutdown_handler` устанавливает `running = 0`. В `main()`: установи обработчик через `sigaction()` с `SA_RESTART`, затем запусти цикл `while (running) { sleep(1); printf("tick\n"); }`. Из другого терминала пошли `SIGTERM` — программа должна завершиться чисто после следующей итерации.

## 13. Практика модуля (сводно)

1. **Сырые syscall-обёртки** (`my_read`/`my_write`/`my_open`/`my_close` через `syscall(2)`) + `strace`-сравнение с glibc-версиями. *(§1)*
2. **`die()`** — единая идиома обработки ошибок через `errno`/`strerror`, используемая во всех следующих практиках. *(§2)*
3. **`read_full`/`write_full`** с корректной обработкой `EINTR` и partial I/O + тест с сигналами под `strace -f`. *(§3)*
4. **vDSO-бенчмарк**: `clock_gettime` через glibc vs через прямой `syscall()`, подтверждённый `strace -c`. *(§4)*
5. **Общий offset через `dup`/`fork`**, демонстрация `O_APPEND` vs без него под конкурентной записью. *(§5)*
6. **Чтение man-страниц** — таблица для 5 syscalls: успех/ошибка/3 errno/1 деталь из NOTES. *(§7)*
7. **pipe + fork + exec**: child stdout → parent через pipe, считаем строки. *(§8)*
8. **mmap round-trip**: отобразить файл, изменить через указатель, msync, проверить. *(§9)*
9. **PID-файл**: acquire/release single-instance guard с `flock(LOCK_EX|LOCK_NB)`. *(§10)*
10. **Рекурсивный dir-walker**: `opendir`/`readdir`/`lstat`, считаем файлы и суммарный размер. *(§11)*
11. **Graceful shutdown**: `volatile sig_atomic_t` + `sigaction()` + SIGTERM из другого терминала. *(§12)*

9. В чём разница `exit()` и `_exit()`, и какой классический баг возникает в дочернем процессе после `fork()`, если использовать неправильный?
10. Что такое зомби-процесс и когда он появляется? Чем «зомби» отличается от «осиротевшего» процесса?

<details>
<summary><b>Эталонные ответы</b></summary>

1. `rdi, rsi, rdx, r10, r8, r9` (плюс номер syscall в `rax`). `rcx` и `r11` использует сама инструкция `syscall` (адрес возврата и флаги), поэтому четвёртый аргумент перенесён в `r10`, чтобы не конфликтовать.
2. Ядро возвращает в `rax` значение `-errno` (малое отрицательное число) при ошибке. glibc-обёртка проверяет диапазон `[-4095,-1]`, записывает `errno = -rax` и возвращает вызывающему `-1`/`NULL` согласно контракту функции.
3. Успешные вызовы не обязаны сбрасывать `errno` в `0` — он может остаться «грязным» от давней ошибки. Валиден `errno` только сразу после вызова, который **сам** сигнализировал ошибку через своё возвращаемое значение (см. RETURN VALUE в man).
4. `select`/`poll`/`epoll_wait` (и родственные) — потому что у них есть аргумент **таймаута**: наивный автоперезапуск отсчитывал бы таймаут заново с нуля, нарушив контракт «ждать не дольше T», поэтому ядро возвращает `EINTR` и отдаёт решение userspace. Также `connect()` на сокете — повторный вызов после `EINTR` ведёт себя по-разному на разных системах (на Linux безопасен, на BSD/Solaris вернёт ошибку), поэтому переносимо ждут готовности через `select`/`poll` + `getsockopt(SO_ERROR)` (см. §3.2).
5. Это нормальное поведение для пайпов/сокетов/терминалов — ядро может передать столько, сколько уместилось в буфер «здесь и сейчас», без ошибки. `EINTR` — это `-1` и **ничего** не передано, сигнал прервал ожидание до начала передачи; partial I/O — успешный возврат `0 < n < запрошено`.
6. vDSO — отображаемая в каждый процесс мини-библиотека с userspace-реализациями `clock_gettime`/`gettimeofday`/`getcpu` и т.п., читающая данные времени из `vvar`-страницы без перехода в ядро. Проверка: `strace -c` не покажет эти вызовы как syscalls, если они ушли через vDSO; `/proc/self/maps` покажет регион `[vdso]`.
7. (1) fd-таблица процесса — индекс=fd → указатель на запись таблицы открытых файлов, плюс `FD_CLOEXEC` (не шарится); (2) открытое файловое описание — offset, флаги доступа, шарится между fd после `dup`/`fork`; (3) inode — сами данные файла, общесистемно. `dup`/`fork` шарят (2), но не флаг `FD_CLOEXEC` из (1).
8. Без `O_APPEND` «перейти в конец + записать» — две раздельные операции, между которыми другой процесс может вклиниться (оба перейдут «в конец», который уже не конец после первой записи) → потеря/перемешивание данных. `O_APPEND` делает это одной атомарной операцией на уровне ядра.
9. `exit()` — флашит stdio-буферы, запускает `atexit`, затем `_exit()`. `_exit()` — немедленно, без флаша. Баг: после `fork()` нефлашенный буфер stdout родителя наследуется потомком; если потомок вызовет `exit()`, содержимое буфера выведется ещё раз → дублирование вывода. Лечится `_exit()` в потомке (если не делает `exec`).
10. Зомби — завершившийся процесс, чья запись в таблице процессов ждёт `wait`/`waitpid` от **живого** родителя (статус ещё не считан). Осиротевший — процесс, чей родитель завершился **раньше** него; его усыновляет init/subreaper, который reap'ает его автоматически — зомби не возникает (или возникает кратковременно до усыновления-и-reap).

</details>
