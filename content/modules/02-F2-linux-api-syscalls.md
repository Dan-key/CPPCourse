## 1. Syscall как контракт: ядро, glibc, ты

### 1.1 Что физически происходит при вызове

На x86-64 Linux системный вызов — это инструкция `syscall` (на 32-битных системах исторически `int 0x80`). Соглашение о регистрах **отличается** от обычного соглашения вызова функций C (System V AMD64 ABI):

| | Обычный вызов функции | Syscall |
|---|---|---|
| Аргументы 1–6 | `rdi, rsi, rdx, rcx, r8, r9` | `rdi, rsi, rdx, r10, r8, r9` |
| Номер/что вызываем | имя символа | число в `rax` |
| Возврат | `rax` | `rax` (см. ниже про ошибки) |

**Почему `r10`, а не `rcx`?** Инструкция `syscall` сама использует `rcx` (сохраняет туда адрес возврата) и `r11` (флаги). Поэтому четвёртый аргумент сдвинут в `r10` — это решение ABI, а не каприз.

**Как сообщается об ошибке.** Ядро возвращает в `rax` **малое отрицательное число = `-errno`** (например, `-EBADF`), а не `-1`. Именно glibc-обёртка делает следующий шаг: если результат в диапазоне `[-4095, -1]`, она записывает `errno = -результат` и возвращает вызывающему `-1` (или `NULL` для функций, возвращающих указатель). **Это поведение обёртки, а не ядра** — вот первая трещина между «что делает Linux» и «что делает libc».

```c
// Грубо то, что делает glibc-обёртка write():
ssize_t write(int fd, const void *buf, size_t n) {
    long ret = raw_syscall(SYS_write, fd, buf, n);
    if (ret < 0 && ret > -4096) {       // -1..-4095 — это -errno
        errno = (int)-ret;
        return -1;
    }
    return ret;
}
```

### 1.2 `syscall(2)` — обход именованной обёртки

`syscall(2)` — функция glibc, позволяющая вызвать *произвольный* номер syscall напрямую, минуя именованную обёртку (`write`, `open` и т.п.). Зачем это нужно в реальном коде:

- **Новый syscall ещё не имеет обёртки** в установленной версии glibc (новые ядра добавляют syscalls быстрее, чем glibc их оборачивает — классика для `io_uring_setup`/`io_uring_enter` на старых дистрибутивах, см. C2);
- **Нужно обойти поведение обёртки** — например, glibc может кэшировать/перехватывать вызов (так делают `fork`, `exit` — обёртки запускают доп. логику вокруг сырого syscall: atfork-хэндлеры, флаш буферов и т.д.);
- учебная цель — увидеть голый интерфейс ядра.

```c
#define _GNU_SOURCE
#include <unistd.h>
#include <sys/syscall.h>

ssize_t raw_write(int fd, const void *buf, size_t n) {
    return syscall(SYS_write, fd, buf, n);   // тот же errno-контракт, что у write()
}
```

**Важная тонкость (ЭКСПЕРТ-уровень):** `syscall(2)` — это **тоже обёртка glibc**, просто универсальная: она *всё ещё* транслирует `-errno` в `errno=...; return -1`. Чтобы увидеть по-настоящему сырое значение из `rax` (отрицательный `-errno`), нужна инлайн-ассемблерная вставка с инструкцией `syscall` напрямую — то, что делают musl/статически собранные раннеры без libc. Для повседневной работы хватает `syscall(2)`.

> **Практика 1.A:** напиши `my_write`/`my_read`/`my_close` через `syscall(2)`, прогони `strace` на бинарнике, использующем твои обёртки и обычные glibc-функции — убедись, что *номера syscalls и их аргументы идентичны*. Различие будет только в том, что glibc-версии могут переходить через PLT/vDSO (см. §4).

## 2. `errno`: контракт, о котором все забывают на третьей строчке

### 2.1 Где физически живёт `errno`

`errno` — не глобальная переменная в простом смысле. Это макрос, раскрывающийся в вызов `*__errno_location()` — функции, возвращающей указатель на **per-thread** ячейку (через TLS, Ф3). Поэтому `errno` безопасен между потоками «из коробки»: у каждого потока свой `errno`.

### 2.2 Правило валидности (источник 90% багов с errno)

> **`errno` валиден ТОЛЬКО непосредственно после вызова, который явно сигнализировал ошибку** (вернул `-1`, `NULL`, и т.п., согласно описанию конкретной функции). Успешный вызов **не обязан** сбрасывать `errno` в `0`.

Отсюда классический баг:

```c
errno = 0;
int fd = open(path, O_RDONLY);   // допустим, успех, fd >= 0
if (something_else() < 0) {       // success_func, не трогает errno вовсе...
    // ...но если внутри success_func был побочный вызов, упавший и
    // ОБРАБОТАННЫЙ (без забывания errno), errno может быть "грязным"
}
if (errno != 0) {                 // ОШИБКА: errno не сброшен успешным open()
    perror("open");               // ложное сообщение про несвязанную ошибку
}
```

Правильно: **проверяй возвращаемое значение**, и только при индикаторе ошибки читай `errno`:

```c
int fd = open(path, O_RDONLY);
if (fd == -1) {                   // вот точка, где errno гарантированно валиден
    perror("open");
    return -1;
}
```

### 2.3 Печать ошибок

- `perror("open")` → `open: No such file or directory` (использует текущий `errno`);
- `strerror(errno)` — **не thread-safe** в общем случае (статический буфер в старых реализациях; в glibc современный `strerror` thread-safe для известных кодов, но переносимый код использует `strerror_r`);
- `strerror_r(errno, buf, sizeof buf)` — переносимый thread-safe вариант (но осторожно: POSIX- и GNU-версии `strerror_r` имеют разные сигнатуры — зависит от `_GNU_SOURCE`).

> **Практика 2.A:** напиши функцию `void die(const char *what)`, которая печатает `"<what>: <strerror(errno)>"` в stderr и завершает процесс с кодом, отражающим `errno` (например, `exit(errno > 0 && errno < 256 ? errno : 1)`). Используй её во всех практиках модуля вместо ручных `if`-проверок — это закладывает идиому, которая встретится в каждом примере TLPI.

## 3. `EINTR` и сигналы: вызов, который вернулся ни с чем

### 3.1 Откуда берётся `EINTR`

Если процесс **блокируется** в системном вызове (`read` на пустой пайп, `write` в заполненный буфер, `wait`, `select`...) и в этот момент приходит сигнал, для которого установлен обработчик, — ядро может **прервать** вызов: вернуть управление обработчику, а сам syscall завершить с `errno == EINTR`, **не выполнив работу**.

Это не баг ядра — это необходимость: сигнал должен быть доставлен *сейчас*, а не после того, как блокирующий вызов когда-нибудь разблокируется.

### 3.2 `SA_RESTART` — но не везде

При установке обработчика через `sigaction` с флагом `SA_RESTART`, **некоторые** прерванные syscalls ядро **автоматически перезапускает** само (без `EINTR` в userspace). Но не все — TLPI табл. 21-1 (важнейшая таблица модуля):

| Никогда не перезапускаются (даже с `SA_RESTART`) | Перезапускаются с `SA_RESTART` |
|---|---|
| `select`, `pselect`, `poll`, `ppoll`, `epoll_wait`, `epoll_pselect` | `read`, `write`, `ioctl`, `open` на медленных устройствах (пайпы, терминалы, сокеты) |
| `connect` (на сокете — особый случай, см. ниже) | `wait`, `waitpid`, `wait3/4` (на Linux — но не на старых SysV-системах) |
| таймауты вида `sleep`/`nanosleep`/`clock_nanosleep` (возвращают остаток времени) | `ioctl` для большинства устройств |

**Следствие для практики:** даже если ты ставишь `SA_RESTART` на все сигналы, твой event loop (C2) на `epoll_wait`/`select` *обязан* сам обрабатывать `EINTR` — ядро тебе тут не поможет. Это прямая связка с C4 (`signalfd`): event loop на `signalfd` избегает этой проблемы вообще, потому что сигнал превращается в обычное событие fd, а не в прерывание syscall.

**`connect()` — отдельная засада** (ЭКСПЕРТ): если `connect()` на блокирующем сокете прерван `EINTR`, **нельзя** просто вызвать `connect()` повторно на том же сокете — соединение уже может быть «в процессе» на уровне ядра, повторный `connect` вернёт `EALREADY`/`EISCONN` непредсказуемо в зависимости от реализации. Правильно — подождать готовности через `select`/`poll`/`epoll` на запись, как при неблокирующем `connect`.

### 3.3 Partial I/O — отдельная (и более частая) проблема

Даже **без единого сигнала**, `read()`/`write()` **могут вернуть меньше байт, чем запрошено** — это нормальное, документированное поведение для пайпов, сокетов, терминалов (и теоретически для обычных файлов при прерывании сигналом до того, как передан хоть один байт — тогда это `EINTR`/`-1`, а вот если хоть один байт уже передан — возврат частичного количества, **не ошибка**).

Итого для надёжного цикла нужно обрабатывать **три исхода** каждого вызова: полный успех, частичный успех (повторить с остатком), `EINTR` (повторить целиком), другая ошибка (сдаться).

```c
#include <errno.h>
#include <unistd.h>

// Гарантированно читает ровно n байт или возвращает <n при EOF / ошибке (errno установлен)
ssize_t read_full(int fd, void *buf, size_t n) {
    size_t done = 0;
    while (done < n) {
        ssize_t r = read(fd, (char *)buf + done, n - done);
        if (r == 0)
            break;                  // EOF
        if (r < 0) {
            if (errno == EINTR)
                continue;           // ничего не передано — повторяем целиком
            return -1;              // настоящая ошибка
        }
        done += (size_t)r;          // частичная передача — продолжаем с остатком
    }
    return (ssize_t)done;
}

ssize_t write_full(int fd, const void *buf, size_t n) {
    size_t done = 0;
    while (done < n) {
        ssize_t w = write(fd, (const char *)buf + done, n - done);
        if (w < 0) {
            if (errno == EINTR)
                continue;
            return -1;
        }
        done += (size_t)w;
    }
    return (ssize_t)done;
}
```

glibc даёт макрос-помощник для первого случая: `TEMP_FAILURE_RETRY(expr)` (из `<unistd.h>` с `_GNU_SOURCE`) — оборачивает выражение в цикл «пока `EINTR`». Но он **не** решает partial I/O — `read_full`/`write_full` выше нужны в любом случае.

> **Практика 3.A:** возьми `read_full`/`write_full` выше. Напиши тест: дочерний процесс шлёт сигнал родителю каждые несколько миллисекунд (`SIGUSR1`, обработчик пуст), родитель в это время читает большой объём данных из пайпа от третьего процесса/потока. Без `read_full` (с голым `read`) и с `SA_RESTART`/без него — покажи разницу в поведении через `strace -e trace=read,write -f`.

## 4. vDSO: syscall без перехода в ядро

### 4.1 Проблема

`clock_gettime()`, `gettimeofday()`, `time()`, `getcpu()` вызываются **очень часто** (логирование с таймстампами, профилирование, таймауты в горячих циклах). Полноценный syscall — это переход в режим ядра и обратно: затраты порядка ста с лишним наносекунд плюс эффекты на TLB/кэши (особенно заметно после митигаций Meltdown/Spectre, которые сделали переходы дороже).

### 4.2 Решение

Ядро отображает в адресное пространство **каждого** процесса небольшую разделяемую библиотеку — **vDSO** (virtual Dynamic Shared Object) — содержащую userspace-реализации этих функций. Они читают данные о времени из отдельной read-only страницы (`vvar`), которую ядро обновляет, и считают результат **без единого перехода в kernel mode** (используя `rdtsc`/`rdtscp` и калибровочные данные).

Динамический линкер резолвит `__vdso_clock_gettime` и подобные символы из vDSO **прозрачно** — твой вызов `clock_gettime()` из glibc обычно *уже* идёт через vDSO, ты этого не видишь.

### 4.3 Как это пронаблюдать

```sh
$ cat /proc/self/maps | grep vdso
7ffd2b9fe000-7ffd2b a00000 r-xp 00000000 00:00 0  [vdso]

$ ldd /bin/ls | grep vdso
        linux-vdso.so.1 (0x00007ffff7fc4000)
```

```c
#include <sys/auxv.h>
#include <stdio.h>
int main(void) {
    printf("AT_SYSINFO_EHDR = %p\n", (void *)getauxval(AT_SYSINFO_EHDR));  // адрес vDSO
}
```

**Доказательство «бесплатности» через strace:**

```sh
$ strace -c ./prog_using_clock_gettime   # clock_gettime НЕ появится в списке syscalls —
                                          # он выполнился через vDSO, ядро не видело вызова
```

А если принудительно дёрнуть через `syscall(SYS_clock_gettime, ...)` напрямую — ты обойдёшь vDSO-резолвинг glibc и получишь настоящий переход в ядро, который strace увидит. Сравнение этих двух путей по времени (через `rdtsc` до/после в цикле на миллион итераций) — наглядная демонстрация цены syscall.

> **Практика 4.A:** замерь время 10 млн вызовов `clock_gettime(CLOCK_MONOTONIC, ...)` обычным образом (через glibc → vDSO) и через `syscall(SYS_clock_gettime, ...)` (настоящий syscall). Подтверди разницу через `strace -c` (в первом случае `clock_gettime` не появится в трейсе вообще). Это ровно вопрос «что такое vDSO» из самопроверки фундамента — и здесь у тебя будут цифры, а не просто определение.

## 5. Универсальная модель файлового I/O

### 5.1 Три уровня (TLPI рис. 5-2) — must know на всю оставшуюся жизнь с Linux

```
Процесс A                Процесс B           Системная (open file) таблица   Таблица inode
┌──────────────┐         ┌──────────────┐    ┌─────────────────────────┐    ┌──────────┐
│ fd 0 ─┐       │         │ fd 0 ─┐       │    │ запись: флаги, offset,  │    │          │
│ fd 1  │       │         │ fd 1  │       │    │ режим доступа ──────────┼───▶│  inode   │
│ fd 2  │       │         │ fd 2  │       │    └─────────────────────────┘    │  файла   │
│ fd 3 ─┼───────┼─────────┼───────┼──────▶├──▶ запись: флаги, offset, ...─────▶│          │
└───────┘       │         └───────┘       │    └─────────────────────────┘    └──────────┘
  fd-таблица                fd-таблица            open file descriptions          inodes
  (per-process)             (per-process)         (могут шариться между fd)      (глобально)
```

Три независимых уровня:

1. **Таблица файловых дескрипторов** процесса — массив, где индекс = fd, значение = указатель на запись в общесистемной таблице + флаг `FD_CLOEXEC` (этот флаг — **на уровне дескриптора**, не шарится).
2. **Открытое файловое описание** (open file description) — общесистемная структура: текущий offset, флаги доступа (`O_RDONLY`/`O_APPEND`/...), счётчик ссылок. **Шарится** между fd, ссылающимися на неё (после `dup`/`dup2`/`fork`).
3. **inode** — сам файл на диске/в памяти: метаданные, содержимое.

### 5.2 Прямые следствия (то, что обычно узнают через баги)

**`dup`/`dup2`/`fork` шарят offset и флаги доступа, но НЕ `FD_CLOEXEC`:**

```c
int fd1 = open("file", O_RDWR);
int fd2 = dup(fd1);
write(fd1, "AAAA", 4);
write(fd2, "BBBB", 4);
// файл содержит "AAAABBBB" — offset ОБЩИЙ через open file description,
// хотя fd1 и fd2 — разные дескрипторы
```

```c
int fd = open("log", O_WRONLY | O_CREAT, 0644);
fcntl(fd, F_SETFD, FD_CLOEXEC);   // НЕ будет унаследован после execve
// но O_RDWR/O_APPEND и offset — будут (это флаги *описания*, не дескриптора)
```

**После `fork()`** — родитель и потомок получают **разные fd**, указывающие на **одно** открытое файловое описание (offset общий!). Если оба пишут в один и тот же файл без `O_APPEND` — гонка за offset.

### 5.3 Атомарность через флаги `open()`

Два флага решают классические гонки именно на уровне ядра (атомарно), а не «лучше стараться в userspace»:

- **`O_APPEND`** — без него «перейти в конец + записать» — **две** операции (`lseek` + `write`), между которыми другой процесс может вклиниться и тоже перейти в конец. С `O_APPEND` ядро делает seek-to-end+write **одной атомарной операцией**. Это база любого общего лог-файла, к которому пишут несколько процессов.
- **`O_CREAT | O_EXCL`** — атомарная проверка-и-создание: если файл существует, `open` вернёт `EEXIST`, и **гарантированно** именно один из конкурирующих вызовов создаст файл. Классический способ сделать файл-лок между процессами (мостик в C5 — PID-файлы, хотя там есть нюансы про "протухшие" PID-файлы).

```c
// Атомарный "file lock" между процессами:
int fd = open("/var/run/myapp.lock", O_CREAT | O_EXCL | O_WRONLY, 0644);
if (fd == -1 && errno == EEXIST) {
    // кто-то уже держит лок (или "протух" — отдельная проблема, см. C5)
}
```

### 5.4 Главные флаги `open()` — справочно

| Флаг | Смысл |
|---|---|
| `O_RDONLY/O_WRONLY/O_RDWR` | режим доступа (взаимоисключающие, не битовая маска между собой) |
| `O_CREAT`, `O_EXCL` | создать; в комбинации — атомарный test-and-create |
| `O_TRUNC` | обрезать существующий файл до 0 при открытии |
| `O_APPEND` | атомарная запись в конец (см. выше) |
| `O_NONBLOCK` | неблокирующий режим (критично для C2) |
| `O_CLOEXEC` | установить `FD_CLOEXEC` атомарно при открытии (избегает гонки fork+fcntl между потоками) |
| `O_DIRECT` | минуя page cache (для своих буферов — Этап 3d, БД) |
| `O_SYNC`/`O_DSYNC` | синхронная запись (durability — Этап 3d) |

> **Практика 5.A:** напиши программу, демонстрирующую общий offset через `dup`: открой файл, `dup`, пиши попеременно через оба fd, проверь результат. Затем повтори через `fork()` без `O_APPEND` и с `O_APPEND` — покажи разницу в итоговом файле при одновременной записи родителя и потомка (используй большие записи и `usleep`, чтобы увеличить шанс гонки без `O_APPEND`).

## 6. Процессы и потоки на уровне syscall — что здесь, а что дальше

Этот модуль **не повторяет** Ф3 (карта памяти, ELF — там «в основном есть») и **не повторяет** C1 (модель памяти потоков — «закрепить»). Здесь — только контракт самих вызовов, который нужен, чтобы читать `man 2` бегло.

### 6.1 `fork()` возвращает дважды

Один вызов, два возврата: `0` в потомке, PID потомка в родителе, `-1` при ошибке (например, превышен `RLIMIT_NPROC` — таблица процессов полна). Всё, что было открыто (fd, мэппинги), **копируется по семантике** (copy-on-write для памяти — Ф3), но fd-таблицы становятся **независимыми копиями**, указывающими на **общие** открытые файловые описания (см. §5.3).

### 6.2 `execve()` заменяет образ процесса целиком

PID сохраняется, но код/данные/куча/стек заменяются. **fd-таблица переживает `exec`**, кроме дескрипторов с `FD_CLOEXEC` — вот почему этот флаг важен: библиотека, открывающая внутренние fd (логи, локи), должна ставить `O_CLOEXEC`/`FD_CLOEXEC`, иначе эти fd «утекут» в любую программу, которую пользователь запустит через `system()`/`popen()`.

### 6.3 `_exit()` vs `exit()` — классический fork-баг

`exit()` (функция **libc**) делает: запускает обработчики `atexit`, **флашит буферы stdio**, затем зовёт `_exit()` (тонкая обёртка над syscall `exit_group`). `_exit()` — немедленное завершение, никакого флаша.

```c
printf("hello");           // осталось в буфере stdio (нет '\n', не выведено)
if (fork() == 0) {
    // ПЛОХО: потомок при exit() флашит ТОТ ЖЕ буфер "hello" — напечатается ДВАЖДЫ
    exit(0);
    // ХОРОШО для потомка перед exec/завершением без флаша родительских буферов:
    _exit(0);
}
```

Правило: **в дочернем процессе после `fork()`, если не делаешь `exec`, используй `_exit()`**, а не `exit()` — иначе унаследованные нефлашенные буферы stdio могут продублироваться.

### 6.4 `wait`/`waitpid` и зомби

Завершившийся процесс становится **зомби** — запись в таблице процессов остаётся, пока родитель не вызовет `wait`/`waitpid` (тогда статус считывается и запись удаляется). Не вызвал `wait` — утечка записей таблицы процессов (видна как `<defunct>` в `ps`). Макросы `WIFEXITED`/`WEXITSTATUS`/`WIFSIGNALED`/`WTERMSIG` разбирают `status`.

> Если родитель сам завершается раньше потомка — потомок «осиротевает» и усыновляется `init`/PID 1 (или ближайшим subreaper), который его и reap'ает. Зомби возникают именно когда родитель **жив**, но не вызывает `wait`.

## 7. `man(2)` как справочник: как читать страницу за 30 секунд

Структура любой man-страницы раздела 2/3 одинакова — научись находить нужное сразу:

1. **SYNOPSIS** — заголовки и feature test macros (`#define _GNU_SOURCE` и т.п.) **до** `#include`. Если функция не находится при компиляции — почти всегда не хватает нужного `_XXX_SOURCE` (`_GNU_SOURCE`, `_POSIX_C_SOURCE >= 200809L`, `_DEFAULT_SOURCE`). Это управляет видимостью прототипов из заголовков glibc — не магия, а простой набор `#ifdef` внутри `<unistd.h>` и др.
2. **RETURN VALUE** — что означает успех (`0`? неотрицательное число? указатель?) и что означает ошибка (`-1`? `NULL`?). **Без этого ты не знаешь, когда смотреть в `errno`.**
3. **ERRORS** — список `errno`-кодов с условиями. Читай выборочно: `EINTR`, `EAGAIN`/`EWOULDBLOCK`, `ENOMEM` почти всегда стоит проверять отдельно от «общей» ошибки.
4. **CONFORMING TO** — POSIX vs Linux-специфика. Важно для переносимости (не критично для тебя — целевая платформа Linux, но помогает понять, что «гарантировано», а что «так делает Linux»).
5. **NOTES** — здесь обычно прячутся реальные грабли (race conditions, устаревшие альтернативы).

> **Практика 7.A:** открой `man 2 open`, `man 2 read`, `man 2 fork`, `man 2 execve`, `man 7 signal` — для каждого найди и выпиши: (a) что означает успех/ошибка в RETURN VALUE, (b) хотя бы 3 кода из ERRORS и условие их возникновения, (c) одну деталь из NOTES, которая удивила. Это не «решение задачи» — это тренировка навыка «читаю man как справочник», которого требует трек.

10. Что такое зомби-процесс и когда он появляется? Чем «зомби» отличается от «осиротевшего» процесса?

<details>
<summary><b>Эталонные ответы</b></summary>

1. `rdi, rsi, rdx, r10, r8, r9` (плюс номер syscall в `rax`). `rcx` и `r11` использует сама инструкция `syscall` (адрес возврата и флаги), поэтому четвёртый аргумент перенесён в `r10`, чтобы не конфликтовать.
2. Ядро возвращает в `rax` значение `-errno` (малое отрицательное число) при ошибке. glibc-обёртка проверяет диапазон `[-4095,-1]`, записывает `errno = -rax` и возвращает вызывающему `-1`/`NULL` согласно контракту функции.
3. Успешные вызовы не обязаны сбрасывать `errno` в `0` — он может остаться «грязным» от давней ошибки. Валиден `errno` только сразу после вызова, который **сам** сигнализировал ошибку через своё возвращаемое значение (см. RETURN VALUE в man).
4. `select`/`poll`/`epoll_wait` (и родственные) — потому что у них есть собственный механизм таймаута/множества fd, и «перезапуск» не имеет однозначного смысла при изменившемся состоянии; также `connect()` на сокете — повторный вызов после `EINTR` опасен (см. п.6 ниже).
5. Это нормальное поведение для пайпов/сокетов/терминалов — ядро может передать столько, сколько уместилось в буфер «здесь и сейчас», без ошибки. `EINTR` — это `-1` и **ничего** не передано, сигнал прервал ожидание до начала передачи; partial I/O — успешный возврат `0 < n < запрошено`.
6. vDSO — отображаемая в каждый процесс мини-библиотека с userspace-реализациями `clock_gettime`/`gettimeofday`/`getcpu` и т.п., читающая данные времени из `vvar`-страницы без перехода в ядро. Проверка: `strace -c` не покажет эти вызовы как syscalls, если они ушли через vDSO; `/proc/self/maps` покажет регион `[vdso]`.
7. (1) fd-таблица процесса — индекс=fd → указатель на запись таблицы открытых файлов, плюс `FD_CLOEXEC` (не шарится); (2) открытое файловое описание — offset, флаги доступа, шарится между fd после `dup`/`fork`; (3) inode — сами данные файла, общесистемно. `dup`/`fork` шарят (2), но не флаг `FD_CLOEXEC` из (1).
8. Без `O_APPEND` «перейти в конец + записать» — две раздельные операции, между которыми другой процесс может вклиниться (оба перейдут «в конец», который уже не конец после первой записи) → потеря/перемешивание данных. `O_APPEND` делает это одной атомарной операцией на уровне ядра.
9. `exit()` — флашит stdio-буферы, запускает `atexit`, затем `_exit()`. `_exit()` — немедленно, без флаша. Баг: после `fork()` нефлашенный буфер stdout родителя наследуется потомком; если потомок вызовет `exit()`, содержимое буфера выведется ещё раз → дублирование вывода. Лечится `_exit()` в потомке (если не делает `exec`).
10. Зомби — завершившийся процесс, чья запись в таблице процессов ждёт `wait`/`waitpid` от **живого** родителя (статус ещё не считан). Осиротевший — процесс, чей родитель завершился **раньше** него; его усыновляет init/subreaper, который reap'ает его автоматически — зомби не возникает (или возникает кратковременно до усыновления-и-reap).

</details>
