# Задание: `eventfd` — разбудить loop из другого потока

`eventfd` — 64-битный счётчик в ядре с дескриптором: запись добавляет к счётчику,
чтение возвращает накопленное и обнуляет. Классическое применение — **разбудить
event loop из другого потока**: внешний поток кладёт задачу в потокобезопасную
очередь (mutex/lock-free SPSC из **C1 §11**) и пишет в `eventfd`, чтобы
`epoll_wait` проснулся и забрал работу.

## Что реализовать

```cpp
int      ev_post(int efd, uint64_t v);     // прибавить v к счётчику (write 8 байт); 0/-1
uint64_t ev_wait(int epfd, int efd);       // дождаться и забрать накопленное
```

### `ev_post`
```cpp
int ev_post(int efd, uint64_t v) {
    return write(efd, &v, sizeof v) == (ssize_t)sizeof v ? 0 : -1;
}
```

### `ev_wait`
Блокируйся на `epoll_wait(epfd, ...)` до готовности `efd`, потом прочитай 8 байт —
ядро вернёт **сумму** всех `ev_post` с прошлого чтения и обнулит счётчик:
```cpp
uint64_t ev_wait(int epfd, int efd) {
    epoll_event evs[4];
    for (;;) {
        int k = epoll_wait(epfd, evs, 4, -1);
        if (k < 0) { if (errno == EINTR) continue; return 0; }
        uint64_t cnt = 0;
        if (read(efd, &cnt, sizeof cnt) == (ssize_t)sizeof cnt)
            return cnt;            // накопленное за период
    }
}
```

## Почему это важно

Так I/O-поток (epoll/io_uring) получает работу от других потоков, **не
просыпаясь впустую**: пока работы нет — спит в `epoll_wait`; появилась —
`ev_post` будит. Счётчик `eventfd` объединяет несколько уведомлений в одно
чтение, поэтому пробуждений не больше, чем реально нужно. Это прямое продолжение
C1: очередь под локом + `eventfd` для пробуждения.

## C-эквивалент

Идентичные сисколлы (`eventfd`, `write`, `read`, `epoll_*`) в C. В ядре похожую
роль играют wait queues (модуль K3).

## Проверка

Автопрогон (ASan/UBSan): продюсер-поток делает N `ev_post(efd, 1)`, консьюмер
через `ev_wait` копит, пока не наберёт ровно N, и число пробуждений не превышает
число `post` (доказывает, что за раз забирается накопленное, а не по одному
вхолостую).
