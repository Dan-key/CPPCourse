# Задание: страница напрямую у buddy-аллокатора

Под всеми аллокаторами лежит **buddy** — он раздаёт **целые страницы** (`PAGE_SIZE`,
обычно 4 КиБ) блоками из 2ⁿ. Когда нужны именно страницы (большой буфер ≥ страницы,
отображение в userspace, DMA-кольцо), их берут напрямую: `__get_free_pages` /
`get_zeroed_page` / `alloc_pages`.

Здесь ты строишь `/dev/cpppage`: при загрузке выделяешь **одну обнулённую страницу** у
buddy, `write` пишет в неё (до `PAGE_SIZE` байт), `read` читает обратно, а при выгрузке
освобождаешь `free_pages` с **тем же order**, что при выделении.

## Что реализовать

- **`init`** — `page_addr = get_zeroed_page(GFP_KERNEL)` (одна обнулённая страница);
  если `0` — `-ENOMEM`; затем `misc_register`.
- **`write(data, n)`** — скопировать в страницу `min(n, PAGE_SIZE)` байт, запомнить
  длину.
- **`read`** — отдать записанные байты из страницы (`simple_read_from_buffer`).
- **`exit`** — `misc_deregister` → `free_pages(page_addr, ORDER)` (ORDER тот же, что при
  выделении — здесь 0).

## Ключевые API

- **`get_zeroed_page(gfp)`** = `__get_free_pages(gfp | __GFP_ZERO, 0)` — одна обнулённая
  страница; возвращает **виртуальный адрес** ядра (0 при неудаче).
- **`__get_free_pages(gfp, order)`** — 2^order страниц подряд (физически непрерывно).
- **`free_pages(addr, order)`** — освободить; **order ДОЛЖЕН совпадать** с выделением,
  иначе порча buddy.
- **`PAGE_SIZE`** — размер страницы (не хардкодь 4096 — на ARM64 бывает 16К/64К).

## Почему именно страницы

Мелкие объекты берут у `kmalloc`/slab (страница на 40-байтный объект — расточительство).
Прямой `alloc_pages` нужен для **целых страниц**: буфер ≥ страницы, `mmap` драйвера
(`remap_pfn_range`, K1), DMA-кольцо. Страница — валюта между MMU, buddy, slab и драйверами.

## Проверка

QEMU: `echo -n "pagedata" > /dev/cpppage`, `cat` → `"pagedata"` (записали в страницу и
прочитали). `dmesg` чист, `rmmod` проходит (страница освобождена тем же order).
