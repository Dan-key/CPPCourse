# Задание: буфер с размером из userspace через kvmalloc

Размер буфера часто **приходит из userspace** и заранее неизвестно — маленький он или
большой. `kmalloc` хорош для маленького, но упрётся в фрагментацию на большом; `vmalloc`
хорош для большого, но расточителен на маленьком. **`kvmalloc`** решает это: пробует
`kmalloc`, на неудаче падает на `vmalloc` — **размер решает за тебя**. `kvfree`
освобождает правильно в обоих случаях.

Здесь ты строишь `/dev/cppkv`: `write` выделяет буфер **размером с записанные данные**
через `kvmalloc` (маленький → kmalloc, большой → vmalloc — автоматически), `read` отдаёт
содержимое. Один и тот же код работает и для 5 байт, и для сотен КБ.

## Что реализовать

- **`write(data, n)`** — `nb = kvmalloc(n, GFP_KERNEL)`; `copy_from_user(nb, ...)`; под
  мьютексом `kvfree(старый)` и заменить указатель/длину на новые.
- **`read`** — отдать текущий буфер (`simple_read_from_buffer` с учётом `*ppos`).
- **`exit`** — `kvfree(buf)` (универсально освобождает и kmalloc-, и vmalloc-память).

## Ключевые API

- **`kvmalloc(size, gfp)`** — kmalloc, если получится; иначе vmalloc. Для данных **CPU**
  (не для DMA — результат может быть физически разбросанным vmalloc).
- **`kvzalloc(size, gfp)`** — то же + обнулить.
- **`kvfree(p)`** — освободить; сам определит, kmalloc это или vmalloc.

## Почему не ветвить вручную

Без `kvmalloc` пришлось бы писать «если size мал — kmalloc, иначе vmalloc» и помнить, чем
освобождать. `kvmalloc`/`kvfree` инкапсулируют это и убирают класс багов «kmalloc упал на
большом размере» и «перепутал kfree/vfree». Это современный дефолт для «большой/
непредсказуемый буфер данных CPU».

## Проверка

QEMU: `echo -n "hello" > /dev/cppkv`, `cat` → `"hello"` (маленький → внутри kmalloc).
Затем один большой `write` (`dd ... bs=262144 count=1`) и чтение той же длины — буфер
масштабируется (внутри может пойти vmalloc). `dmesg` чист.
