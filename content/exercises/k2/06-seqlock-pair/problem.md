# Задание: seqlock — согласованный снимок (kernel-native)

Реализуй защиту пары значений `(a, b)`, которую писатель держит равной (`a == b`),
так чтобы читатель **никогда** не увидел рассогласованный снимок. Это **нативный
примитив ядра** `seqlock_t` — прямой аналог упражнения `03-seqlock` из C1. Им
защищены `jiffies` и монотонные часы: читатели не блокируют писателя, а
**перечитывают** данные, если во время чтения была запись.

## Устройство `/dev/cppseq`

- `write "<N>"` → установить `a = b = N` (атомарно для читателей)
- `read` → согласованный снимок `"a b\n"`; всегда `a == b`

## Что реализовать

### `seq_write` — под write_seqlock
```c
write_seqlock(&sl);       // счётчик последовательности → НЕЧЁТНЫЙ («идёт запись»)
val_a = (int)n;
val_b = (int)n;
write_sequnlock(&sl);     // → ЧЁТНЫЙ («запись завершена»)
```

### `seq_read` — read_seqbegin / read_seqretry
```c
unsigned seq;
do {
    seq = read_seqbegin(&sl);      // запомнить счётчик (ждём, если нечётный)
    a = val_a;
    b = val_b;
} while (read_seqretry(&sl, seq)); // счётчик изменился? была запись → ПЕРЕЧИТАТЬ
```

## Как это работает (и чем отличается от rwlock)

- **Нечётный счётчик** последовательности = «писатель внутри»; читатель, поймав
  изменение счётчика между `seqbegin` и `seqretry`, понимает, что данные могли
  быть рваными, и **повторяет** чтение.
- **Читатели не блокируют писателя** (в отличие от rwlock): писатель всегда
  проходит сразу. Зато читатели могут «крутиться» при частой записи. Поэтому
  seqlock хорош для «**очень** часто читаем, **редко** пишем» и для **маленьких**
  данных (перечитывание дёшево).
- **Барьеры внутри:** `write_seqlock`/`read_seqretry` несут нужные memory
  barriers (release/acquire, C1 §5), поэтому читатель, увидевший чётный
  неизменившийся счётчик, видит согласованные `a` и `b`.
- **Ограничение:** данные под seqlock читатель должен **копировать как есть**
  (никаких разыменований указателей, которые писатель мог поменять, — иначе
  рваный указатель). Для указателей берут RCU (упр. 04), для маленьких значений —
  seqlock.

## Сборка и проверка

`cppmod.ko` в QEMU; тест проверяет `a == b` после записей и **стресс**: писатель в
фоне молотит обновления, а читатель в цикле обязан **всегда** видеть `a == b`
(ни одного рваного снимка). Плюс чистый `dmesg`.
