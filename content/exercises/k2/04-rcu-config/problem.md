# Задание: конфигурация под RCU (Read-Copy-Update)

RCU — **флагманский** примитив синхронизации ядра Linux. Идея: читатели почти
бесплатны (никакой блокировки, никакой записи в общую память), а писатель
**публикует новую версию** и ждёт, пока уйдут читатели старой, прежде чем её
освободить. Это модель из C1 §12, теперь по-настоящему. Применяется повсеместно:
таблицы маршрутизации, namespace'ы, dentry-кэш, списки модулей.

## Устройство `/dev/cppcfg`

- `read` → текущая конфигурация `"<name> <value>\n"`
- `write "<name> <value>"` → опубликовать новую (атомарная замена указателя)

## Что реализовать

### `cfg_read` — read-side (НЕ спать внутри!)
```c
struct cfg *p;
rcu_read_lock();                          // вход в read-side критическую секцию
p = rcu_dereference(current_cfg);         // безопасно разыменовать (с барьером)
strscpy(name, p->name, sizeof name);      // СКОПИРОВАТЬ в локальные, пока под RCU
value = p->value;
rcu_read_unlock();                        // выход; copy_to_user — уже ВНЕ RCU
```

### `cfg_write` — update-side
```c
mutex_lock(&writer_mutex);                            // сериализуем писателей
old = rcu_dereference_protected(current_cfg, lockdep_is_held(&writer_mutex));
rcu_assign_pointer(current_cfg, neu);                 // публикация (барьер!)
mutex_unlock(&writer_mutex);
synchronize_rcu();                                    // ЖДЁМ читателей старой версии
kfree(old);                                           // теперь освободить безопасно
```

## Почему это работает (и почему так быстро для читателей)

- **`rcu_read_lock()`** в классическом RCU — это (почти) просто **запрет
  вытеснения**: ноль атомиков, ноль записи в общую память → читатели
  масштабируются идеально (в отличие от спинлока/rwlock, где общая строка кэша
  пинг-понгует, C1 §10).
- **`rcu_assign_pointer`/`rcu_dereference`** несут нужные **барьеры** (release у
  публикации, dependency-ordering у чтения — ровно §5.7/§19.4 из C1): читатель,
  увидевший новый указатель, увидит и полностью инициализированный объект.
- **`synchronize_rcu()`** блокирует писателя до конца **grace period** — момента,
  когда гарантированно ни один читатель не держит ссылку на старую версию. Только
  после этого `kfree(old)` безопасен (нет use-after-free). Освободить раньше —
  классический баг.
- **Read-side не спит:** под `rcu_read_lock()` (как под спинлоком) нельзя
  блокироваться/спать — поэтому копируем в локальные переменные, а `copy_to_user`
  делаем после `rcu_read_unlock()`.

> Альтернатива `synchronize_rcu()` (блокирующего) — `call_rcu(&old->rcu, free_cb)`:
> отложенное освобождение через колбэк, без ожидания. Для частых обновлений берут
> его. Здесь `synchronize_rcu` нагляднее.

## Сборка и проверка

`cppmod.ko` в QEMU; тест читает дефолт, публикует новые версии, делает серию
быстрых публикаций и проверяет, что `dmesg` чист (нет RCU-стэллов,
use-after-free, порчи slab — что было бы при неверном порядке assign/sync/free).
