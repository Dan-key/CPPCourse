# Задание: per-CPU счётчик — масштабируемая статистика

Реализуй счётчик, у которого **у каждого ядра свой экземпляр**. Инкремент локален
(своя память ядра — никакого общего кэш-лайна), а общая сумма собирается только на
**чтении**. Это прямое решение проблемы масштабируемости из C1 §10/§13: общий
счётчик под нагрузкой пинг-понгует строку кэша между ядрами и сериализуется на
шине когерентности; per-CPU убирает разделяемую запись вообще.

## Устройство `/dev/cpppercpu`

- `write "<N>"` → прибавить `N` к счётчику **текущего** ядра
- `read` → сумма по **всем** ядрам

## Что реализовать

### `pc_write` — локальный инкремент
```c
this_cpu_add(mycount, n);     // прибавить к экземпляру ТЕКУЩЕГО ядра, preempt-safe
```
`this_cpu_add` берёт переменную текущего CPU и меняет её — без блокировки и без
обращения к чужой памяти. Это и есть «нет разделяемой записи».

### `pc_read` — агрегация
```c
for_each_possible_cpu(cpu)
    total += per_cpu(mycount, cpu);   // сложить экземпляры всех ядер
```

## Почему это быстрее блокировки

- **Нет общего кэш-лайна на горячем пути.** Каждое ядро пишет в свою память →
  ноль когерентного трафика между ядрами (C1 §4.2), ноль контеншна. Спинлок/atomic
  на одном общем счётчике, наоборот, сериализует все ядра на одной строке.
- **Цена платится редко** — только на чтении (агрегация O(числа ядер)). Идеально
  для статистики «часто пишем, редко читаем» (сетевые счётчики, vmstat,
  `/proc/stat`).
- **`this_cpu_*` — preempt-safe:** операция атомарна относительно вытеснения на
  своём ядре (использует локальные атомарные инструкции), поэтому отдельный лок не
  нужен. Если бы инкремент состоял из нескольких шагов с возможным вытеснением
  между ними, понадобился бы `get_cpu()`/`put_cpu()` или `local_irq_save`.

> Тонкость точности: при чтении на работающей системе сумма — «почти точная»
> (другое ядро может инкрементить прямо сейчас). Для статистики это норма. Если
> нужна строгая согласованность — `percpu_counter` API с порогами, либо seqlock
> (упр. 06).

## Сборка и проверка

`cppmod.ko` в QEMU; тест прибавляет числа и проверяет агрегат (`read`), и что
`dmesg` чист. На многоядерном QEMU инкременты могут попадать на разные ядра —
сумма всё равно обязана сойтись.
