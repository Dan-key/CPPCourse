# Задание: переполнение и UB-арифметика — найди UBSan'ом, почини

Это **отладочное** упражнение в духе §2.2 и §16 лекции. Код в `starter.c`
компилируется и даже проходит часть проверок, но содержит **три дефекта
арифметики**, которые проявляются только на граничных входах: целочисленное
переполнение и сдвиг на ширину типа. Твоя задача — увидеть их инструментом и
исправить.

## Рабочий процесс

```sh
# UBSan с остановкой на первом UB — главный инструмент здесь:
gcc -std=c17 -fsanitize=undefined -fno-sanitize-recover=all -g starter.c test.c -o prog
./prog
# UBSan напечатает класс: "signed integer overflow", "shift exponent 32 is too large"
# и точную строку — почти буквально показывает, что чинить.
```

По умолчанию UBSan «recover» (печатает и продолжает) — поэтому добавлен
`-fno-sanitize-recover=all`: тогда первый же UB завершает программу, и его не
пропустишь.

## Что реализовать

| Функция | Контракт |
|---|---|
| `int64_t array_sum(const int32_t *a, size_t n)` | сумма `n` элементов **без переполнения** (даже если сумма не влезает в `int32`) |
| `int32_t array_max(const int32_t *a, size_t n)` | максимум `n` элементов (`n >= 1`); должно работать для массива из одних отрицательных |
| `uint32_t low_bits_mask(unsigned k)` | значение с младшими `k` установленными битами, `k` в диапазоне `[0, 32]` |

## Где искать (подсказки)

- `array_sum`: аккумулятор объявлен как `int32_t` — он переполняется раньше, чем
  функция вернёт `int64_t`. Расширь тип аккумулятора.
- `array_max`: максимум инициализирован нулём — для массива `{-5,-3,-9}` функция
  вернёт `0`, которого в массиве нет. Инициализируй первым элементом.
- `low_bits_mask`: `1u << 32` — сдвиг на ширину типа, это **UB** (на x86 счётчик
  маскируется до 5 бит, давая неверный результат). Обработай `k == 32` отдельно.

## Проверка

```sh
gcc -std=c17 -Wall -Wextra -Wconversion -Wsign-conversion \
    -fsanitize=address,undefined -O1 -g starter.c test.c -o prog && ./prog
# Цель: "12/12 пройдено", код выхода 0.
```

Дополнительно прогони версию с `-fno-sanitize-recover=all` (см. выше) и убедись,
что ни одного UBSan-сообщения не осталось.
