# Задание: утечка памяти — найди по балансу аллокаций и почини

Отладочное упражнение по §8 и §13 лекции. `make_table(n)` выделяет массив из `n`
строк, а `free_table` должна освободить **всё**, что было выделено. В стартовом
коде освобождение неполное — есть утечка. Найди её и почини так, чтобы баланс
аллокаций после освобождения был нулевым.

## Как это проверяется

Вместо `malloc`/`free` код использует инструментированный аллокатор из `test.c`:
`dbg_alloc` увеличивает счётчик живых блоков `g_live`, `dbg_free` уменьшает. Это
ровно техника «подсчёта аллокаций» из §13 — так ловят утечки там, где Valgrind/LSan
недоступны. Тест проверяет, что после `free_table` счётчик `g_live` вернулся в `0`.

## Инструменты для самостоятельной диагностики

```sh
# 1) LeakSanitizer (если доступен в твоём окружении):
gcc -std=c17 -fsanitize=address -g starter.c test.c -o prog && ./prog
#    LSan при выходе перечислит "definitely lost" блоки со стеком аллокации.

# 2) mtrace (встроенный трекер glibc, §13.3):
#    обернуть main в mtrace()/muntrace(), запустить с MALLOC_TRACE=/tmp/mt.log,
#    затем: mtrace ./prog /tmp/mt.log  → покажет несбалансированные malloc.

# 3) Просто подсчёт (как в этом тесте): печатать g_live до и после.
```

## Что реализовать

| Функция | Контракт |
|---|---|
| `char **make_table(int n)` | (уже корректна) массив из `n` строк `"row0".."row{n-1}"` |
| `void free_table(char **t, int n)` | освободить **каждую** строку `t[i]` И сам массив `t` через `dbg_free` |

## Где искать

`free_table` освобождает только сам массив `t`, но не строки `t[0..n-1]` —
утекает `n` блоков на каждый вызов. Пройди по `t[i]` и освободи каждый, затем
освободи `t`. Порядок важен: сначала элементы, потом массив (после `dbg_free(t)`
обращаться к `t[i]` уже нельзя — это был бы use-after-free).

## Проверка

```sh
gcc -std=c17 -Wall -Wextra -Wconversion -Wsign-conversion \
    -fsanitize=address,undefined -O1 -g starter.c test.c -o prog && ./prog
# Цель: "5/5 пройдено (живых блоков: 0)", код выхода 0.
```
