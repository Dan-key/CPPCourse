# Задание: false sharing — развести счётчики по кэш-линиям

**False sharing** (ложное разделение, C1 §10) — тихий убийца масштабируемости: два
потока пишут в **разные** переменные, но те лежат в **одной** кэш-линии (64 байта).
Аппаратная когерентность (MESI, C1 §4) инвалидирует **всю** линию при любой записи —
и линия «пинг-понгует» между ядрами, как будто это одна переменная. Логического
конфликта нет (false), но железо его создаёт (sharing), и производительность падает
в разы.

Классический случай — **массив на-поток счётчиков**: `long counters[NCPU]` —
по `long` на поток, но 8 счётчиков влезают в одну 64-байтную линию, и 8 потоков
колотят её наперегонки. Лечение — **выравнивание**: каждый счётчик на **своей**
кэш-линии (`alignas(64)` + padding).

Здесь ты реализуешь массив счётчиков, каждый из которых занимает **отдельную**
кэш-линию.

## Что реализовать

```cpp
struct Counters;                          // непрозрачный массив padded-счётчиков
Counters* counters_create(int n);
void      counters_destroy(Counters*);

void counters_inc(Counters*, int i);      // ++ счётчика i
long counters_get(Counters*, int i);      // значение счётчика i

int  counters_elem_size();                // sizeof одного padded-счётчика (кратно 64, >=64)
int  counters_on_distinct_lines(Counters*, int n); // 1, если никакие два счётчика
                                                    //    НЕ делят 64-байтную линию
```

## Ключ: `alignas(64)` + padding

```cpp
struct alignas(64) PaddedCounter {        // alignas(64) → выравнивание И sizeof кратны 64
    long value = 0;
    // компилятор сам добьёт структуру паддингом до 64 байт (alignas заставляет)
};

struct Counters { PaddedCounter* a; int n; };

Counters* counters_create(int n) {
    Counters* c = new Counters();
    c->a = new PaddedCounter[n];          // aligned-new: массив выровнен на 64
    c->n = n;
    return c;
}
int counters_elem_size() { return (int)sizeof(PaddedCounter); }   // 64

int counters_on_distinct_lines(Counters* c, int n) {
    for (int i = 0; i < n; ++i)
        for (int j = i + 1; j < n; ++j) {
            auto li = (unsigned long)(&c->a[i].value) / 64;
            auto lj = (unsigned long)(&c->a[j].value) / 64;
            if (li == lj) return 0;        // делят линию — false sharing!
        }
    return 1;
}
```

`alignas(64)` делает две вещи: (1) каждый объект начинается на границе 64 байт; (2)
`sizeof` становится **кратным** 64 (чтобы в массиве каждый элемент тоже был выровнен).
Поэтому соседние счётчики в массиве гарантированно на разных линиях.

## Почему это лечит false sharing

Когда `counters[i]` и `counters[j]` на **разных** линиях, запись в один **не**
инвалидирует кэш другого ядра — линии независимы, пинг-понга нет. Цена — память:
8 байт полезных + 56 байт padding на счётчик. Это оправдано **только** для
**горячих, часто пишущихся** разными ядрами данных (на-поток счётчики, `head`/`tail`
очереди, поля, обновляемые разными воркерами). Не выравнивай всё подряд — это раздует
память и убьёт ту самую кэш-локальность.

## Связь с C1

Это практическая часть false sharing из C1 §10: там ты разбирал механизм (MESI, RFO,
cache-line bouncing), здесь — реализуешь **исправление** и проверяешь его
**структурно** (счётчики на разных линиях). Замер эффекта (микробенч с/без
`alignas(64)`, `perf c2c`) — в большом задании модуля.

## Проверка

Автопрогон (ASan/UBSan): `counters_elem_size()` кратен 64 и ≥64; `inc`/`get`
работают независимо для каждого счётчика; `counters_on_distinct_lines` == 1 для
массива (никакие два не делят линию). Стартер (без `alignas`) проваливает проверки
размера и линий — добавь выравнивание.
