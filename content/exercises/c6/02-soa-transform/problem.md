# Задание: SoA vs AoS — data-oriented обработка горячего цикла

В горячем цикле **layout данных важнее алгоритма**. Кэш грузится **линиями по 64
байта** (C1 §4): доступ к памяти платит за **всю** линию, даже если нужно одно поле.
Поэтому как разложены данные — определяет, сколько кэш-линий ты трогаешь.

- **AoS (Array of Structures)** — массив структур: `struct P{ long x,y,vx,vy,mass,...; }
  P arr[N];`. Поля одной сущности рядом. Но если цикл читает **только** `x` и `vx`,
  он всё равно тянет в кэш `y,vy,mass,…` — **впустую** греет линии, мало полезных
  данных на линию.
- **SoA (Structure of Arrays)** — структура массивов: отдельный массив на каждое
  поле: `long x[N], y[N], vx[N], vy[N];`. Цикл по `x` и `vx` читает **только** эти
  два массива — **плотно**, каждая кэш-линия полна нужными данными, и компилятор
  легко **векторизует** (SIMD), т.к. данные одного типа лежат подряд.

Здесь ты реализуешь типовые операции над частицами в **SoA**-раскладке.

## Что реализовать (всё целочисленно — детерминизм под санитайзерами)

```cpp
// Интегрирование: x[i] += vx[i]*dt; y[i] += vy[i]*dt  (для всех i)
void soa_advance(long* x, long* y, const long* vx, const long* vy, int n, long dt);

// Фильтр+подсчёт: сколько частиц внутри прямоугольника [x0,x1]×[y0,y1] (включительно)
int  soa_count_in_box(const long* x, const long* y, int n,
                      long x0, long y0, long x1, long y1);

// Фильтр+редукция по «маске активности»: сумма vx[i] по тем i, где alive[i] != 0
long soa_sum_active_vx(const long* vx, const unsigned char* alive, int n);
```

## Реализация (простые плотные циклы)

```cpp
void soa_advance(long* x, long* y, const long* vx, const long* vy, int n, long dt) {
    for (int i = 0; i < n; ++i) { x[i] += vx[i]*dt; y[i] += vy[i]*dt; }
}
int soa_count_in_box(const long* x, const long* y, int n,
                     long x0, long y0, long x1, long y1) {
    int c = 0;
    for (int i = 0; i < n; ++i)
        c += (x[i] >= x0 && x[i] <= x1 && y[i] >= y0 && y[i] <= y1);  // branchless-накопление
    return c;
}
long soa_sum_active_vx(const long* vx, const unsigned char* alive, int n) {
    long s = 0;
    for (int i = 0; i < n; ++i) s += alive[i] ? vx[i] : 0;   // или (long)(alive[i]!=0)*vx[i]
    return s;
}
```

Эти циклы тривиальны — **в этом и суть**: на правильном layout (SoA) горячий цикл
прост, плотно использует кэш и векторизуется. На AoS тот же алгоритм трогал бы вдвое-
впятеро больше линий.

## Почему layout > алгоритм в hot path

- **Плотность кэш-линий.** SoA: цикл по `x`/`vx` тянет только нужное → ~8 значений
  `long` на 64-байтную линию полезны. AoS: на линии вперемешку `x,y,vx,vy,…` → много
  «балласта», эффективная пропускная способность кэша падает в разы.
- **Векторизация (SIMD).** Подряд лежащие однотипные данные компилятор грузит
  пачками (`movdqa`/AVX) и считает 4–8 элементов за инструкцию. AoS с «дырами» так
  не векторизуется (нужен gather).
- **Предсказуемый префетч.** Последовательный доступ по массиву ядро **предсказывает**
  и подгружает заранее (hardware prefetcher). Прыжки по структурам — хуже.
- **Когда AoS лучше:** если почти всегда нужны **все** поля сущности сразу (трогаешь
  всю структуру) — тогда AoS локальнее. SoA выигрывает на **частичном** доступе в
  массовых циклах (симуляции, ECS-движки, аналитические сканы столбцов в БД).

Это и есть «data-oriented design»: проектируй данные под доступ горячего цикла, а
не под «как удобно описать сущность».

## Проверка

Автопрогон (ASan/UBSan): `advance` сдвигает координаты верно для всех частиц;
`count_in_box` считает попавшие (включая границы); `sum_active_vx` суммирует только
активные. Реализуй плотные циклы — все пройдут.

Сравни на godbolt (`-O2 -ftree-vectorize`) ассемблер SoA-цикла и эквивалентного
AoS — увидишь SIMD на SoA и его отсутствие на AoS.
