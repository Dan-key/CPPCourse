# Задание: syscall batching — амортизировать переходы в ядро

Каждый системный вызов — переход в ядро и обратно (~сотни нс, дороже после
Spectre/Meltdown-митигаций). В readiness-модели (epoll, C2) на N операций — **N**
сисколлов: чтобы прочитать из 100 готовых сокетов, 100 раз `read`. Это линейный
оверхед, доминирующий на высоком темпе мелких операций.

**Syscall batching** убирает его: накопить пачку операций и выполнить её **одним**
сисколлом. Это суть io_uring (C2): кладёшь N запросов в кольцо отправки и одним
`io_uring_enter` говоришь «выполни всё» — N операций за **1** переход вместо N. Здесь
ты моделируешь логику батчера, считая «сисколлы» (флаши).

## Что реализовать

```cpp
struct Batcher;                          // непрозрачный коалесцер операций
Batcher* batch_create(int batch_size);   // флашить пачками по batch_size
void     batch_destroy(Batcher*);

void batch_push(Batcher*, int op);       // поставить операцию; авто-флаш при заполнении
void batch_flush(Batcher*);              // принудительно отправить накопленное (если есть)

int  batch_syscalls(Batcher*);           // сколько флашей было ("io_uring_enter")
int  batch_total_ops(Batcher*);          // сколько операций реально отправлено (флашнуто)
int  batch_pending(Batcher*);            // сколько накоплено, но ещё не флашнуто
```

## Логика

- **`batch_push(op)`**: добавить в буфер (`pending++`). Если `pending == batch_size`
  — **авто-флаш**: `syscalls++`, `total += batch_size`, `pending = 0`. (Одной
  «системной» операцией отправили целую пачку.)
- **`batch_flush()`**: если `pending > 0` — `syscalls++`, `total += pending`,
  `pending = 0`. Если буфер пуст — **ничего** (не делать «пустой сисколл»).

```cpp
void batch_push(Batcher* b, int op) {
    (void)op;                  // в реальном коде op кладётся в кольцо SQE
    b->pending++;
    if (b->pending == b->batch_size) {     // буфер полон → отправить пачку
        b->syscalls++; b->total += b->pending; b->pending = 0;
    }
}
void batch_flush(Batcher* b) {
    if (b->pending > 0) { b->syscalls++; b->total += b->pending; b->pending = 0; }
}
```

## Почему это выигрывает

```
epoll:     [enter] read [exit]  × 100 операций            = 100 переходов в ядро
io_uring:  push×100 (в кольцо, без сисколла) → [enter ×100]= 1 переход
```

- **Амортизация:** фиксированная цена перехода делится на всю пачку. На 100 операциях
  оверхед сисколла падает в ~100 раз.
- **SQPOLL (C2):** ядро само опрашивает кольцо — тогда даже этого одного сисколла
  нет (submission бессисколльный).
- **Когда НЕ помогает:** если время уходит не в переходы, а в саму работу
  (парсинг, копирование, логику) — батчинг бесполезен. Сначала **измерь** (`strace
  -c`: сисколлов на операцию), потом батчируй. На низком темпе оверхед перехода не
  доминирует — и простой epoll достаточен.

Главный принцип C6: батчинг выгоден ровно тогда, когда **профиль** показал, что
доминирует оверхед **перехода в ядро**, а не полезная работа.

## Связь с C2

Это модель того, что io_uring делает кольцами SQ/CQ: `batch_push` ≈ положить SQE
(без сисколла), `batch_flush` ≈ `io_uring_enter` (один сисколл на пачку). Считая
флаши, ты видишь, во сколько раз батчинг сократил переходы в ядро.

## Проверка

Автопрогон (ASan/UBSan): при `batch_size=4` пуш 10 операций даёт 2 авто-флаша
(`syscalls=2`, `total=8`, `pending=2`), затем `flush` → `syscalls=3`, `total=10`,
`pending=0`; флаш пустого буфера не делает сисколл; ровно `batch_size` пушей = 1
флаш. Реализуй логику — все пройдут.
