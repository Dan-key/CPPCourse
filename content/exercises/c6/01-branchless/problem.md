# Задание: branchless-код — убрать непредсказуемые ветви из hot path

Современный CPU исполняет инструкции **конвейером** и **спекулятивно**: он угадывает
исход `if` и начинает выполнять предсказанную ветку заранее. Если предсказатель
ошибся (**branch misprediction**) — конвейер сбрасывается, и это стоит ~15–20 тактов.
Для **предсказуемых** ветвей (почти всегда true) это бесплатно. Но для ветви, чей
исход **зависит от данных** и «50/50» (например, `if (arr[i] >= threshold)` на
случайных данных), предсказатель ошибается в половине случаев — и в горячем цикле
это доминирует над всем остальным.

**Branchless-код** убирает такие ветви, заменяя их арифметикой/битовыми операциями,
которые CPU считает **без предсказания** (через `cmov`, `setcc`, маски). Здесь ты
реализуешь классический набор branchless-примитивов.

## Что реализовать

```cpp
int  bl_min(int a, int b);                 // min без ветви
int  bl_max(int a, int b);                 // max без ветви
int  bl_abs(int x);                        // |x| без ветви (НЕ для INT_MIN)
int  bl_clamp(int x, int lo, int hi);      // зажать в [lo,hi]
int  bl_select(int cond, int a, int b);    // cond ? a : b (cond — 0 или не-0)
int  bl_sign(int x);                       // -1 / 0 / +1
int  bl_count_ge(const int* a, int n, int t); // сколько a[i] >= t (без ветви в цикле)
```

## Подсказки по реализации

```cpp
// min/max через маску знака разности:
int bl_min(int a, int b){ return b ^ ((a ^ b) & -(int)(a < b)); }
//   (a<b) даёт 0/1; -(…) даёт маску 0 или 0xFFFFFFFF; xor-swap выбирает меньшее.
int bl_max(int a, int b){ return a ^ ((a ^ b) & -(int)(a < b)); }

// abs через арифметический сдвиг (знак в маску):
int bl_abs(int x){ int m = x >> (sizeof(int)*8 - 1); return (x ^ m) - m; }
//   x>>31 = 0 (если x>=0) или -1 (если x<0). НЕ вызывай на INT_MIN (переполнение).

int bl_clamp(int x, int lo, int hi){ return bl_min(bl_max(x, lo), hi); }
int bl_select(int c, int a, int b){ return b ^ ((a ^ b) & -(int)(c != 0)); }
int bl_sign(int x){ return (x > 0) - (x < 0); }   // setcc, не ветвь

// count: НАКАПЛИВАЙ булев результат сравнения (без if внутри цикла):
int bl_count_ge(const int* a, int n, int t){
    int c = 0;
    for (int i = 0; i < n; ++i) c += (a[i] >= t);   // (a[i]>=t) это 0/1, без ветви
    return c;
}
```

## Почему это важно

- **Предсказуемая ветвь почти бесплатна** (предсказатель прав) — branchless нужен
  **только** для **непредсказуемых, data-dependent** ветвей в горячем коде. Не
  переписывай всё подряд: ветвь, которая почти всегда идёт одинаково, оставь как
  есть (компилятор и предсказатель справятся, а branchless может быть даже медленнее).
- **Где выгодно:** фильтрация/подсчёт по случайным данным, сравнения в сортировке,
  `min/max/clamp` в тайтовых циклах (графика, DSP, парсеры).
- **`cmov` vs ветвь:** компилятор сам генерирует `cmov` для простых тернарников при
  `-O2`, но иногда «боится» (если ветка дорогая) — тогда явная branchless-формула
  гарантирует отсутствие ветви. Проверяй на godbolt (`-O2`), что вышло `cmov`/`setcc`,
  а не `jne`.

## Замечание про тест

Автотест проверяет **корректность** (branchless-результат совпадает с обычным
`a<b?a:b` и т.д.) — «branchless» не проверяется тестом (это свойство ассемблера,
смотри на godbolt). Смысл задания — реализовать формулы **без ветвей** и понять,
**когда** они нужны, а когда ветвь дешевле.

## Проверка

Автопрогон (ASan/UBSan): min/max/abs/clamp/select/sign/count на наборе входов
совпадают с эталонной «ветвистой» логикой; границы (равные значения, отрицательные,
clamp за обе границы). `INT_MIN` для `abs`/`sign` не подаётся (переполнение —
обсуждается в лекции). Реализуй формулы — все пройдут.
