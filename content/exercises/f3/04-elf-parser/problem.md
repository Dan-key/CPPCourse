# Задание: парсинг ELF64-заголовка

Прочитай и проверь заголовок ELF64-файла (little-endian) и заполни структуру
`elf_info_t`. Это основа понимания того, как загрузчик/ядро распознают
исполняемые файлы.

## Интерфейс

```c
int elf_parse(const char *path, elf_info_t *info);
```

Заполнить `info` и вернуть `0`, если файл — корректный **ELF64 LE**;
вернуть `-1`, если файл не существует, слишком мал, или не ELF64 LE.

```c
typedef struct {
    int      valid;       // 1 если ELF64 LE
    uint16_t e_type;      // ET_REL=1, ET_EXEC=2, ET_DYN=3
    uint16_t e_machine;   // EM_X86_64=62, EM_AARCH64=183
    uint64_t e_entry;     // точка входа
    uint16_t e_phnum;     // число program headers
    uint16_t e_shnum;     // число section headers
} elf_info_t;
```

## Проверки (e_ident)

| Байт         | Значение            | Смысл               |
|--------------|---------------------|---------------------|
| `e_ident[0..3]` | `\x7f` `E` `L` `F` | Magic               |
| `e_ident[4]`    | `2` (ELFCLASS64)   | 64-битный класс     |
| `e_ident[5]`    | `1` (ELFDATA2LSB)  | little-endian       |

Если любая проверка не прошла → вернуть `-1`.

## Алгоритм

```
открыть файл (open/fopen)
прочитать sizeof(Elf64_Ehdr) байт; если меньше — return -1
проверить magic + class + data
скопировать e_type, e_machine, e_entry, e_phnum, e_shnum в info
info->valid = 1
закрыть файл, return 0
```

- Заголовок `Elf64_Ehdr` уже определён в `starter.c`.
- Поскольку требуется именно LE, на x86-64 поля можно читать напрямую —
  но не забудь корректно обрабатывать короткие/битые файлы.
- Всегда закрывай дескриптор/`FILE*` на всех путях выхода.

## Тесты

- Реальный ELF64 (например, сам `./prog`) → `valid==1`, разумные `e_type`/`e_machine`
- Слишком короткий файл → `-1`
- Файл с испорченным magic → `-1`
- Несуществующий путь → `-1`
