# Задание: bounded buffer на POSIX-семафорах

Классический паттерн синхронизации «производитель-потребитель» через **считающие
семафоры**. Семафор — это атомарный счётчик: `sem_wait` уменьшает (блокируясь на
нуле), `sem_post` увеличивает (будя ждущих). Два семафора элегантно решают
ограниченную очередь.

## Идея: два семафора

```
empty — сколько СВОБОДНЫХ слотов (изначально CAP)
full  — сколько ЗАНЯТЫХ слотов  (изначально 0)

put:  sem_wait(empty)  → записать → sem_post(full)     // занял слот
get:  sem_wait(full)   → прочитать → sem_post(empty)   // освободил слот
```

`empty` не даёт производителю писать в полную очередь; `full` не даёт
потребителю читать из пустой. Для **одного** producer и **одного** consumer
мьютекс на индексы не нужен (head трогает только consumer, tail — только
producer, а семафоры упорядочивают видимость).

## Что реализовать

```cpp
int  sq_init(SemQueue* q);        // sem_init(empty,0,CAP); sem_init(full,0,0); 0/-1
void sq_put(SemQueue* q, int v);  // wait(empty) → buf[tail] → post(full)
int  sq_get(SemQueue* q);         // wait(full) → buf[head] → post(empty)
void sq_destroy(SemQueue* q);     // sem_destroy обоих
```

`sem_init(&s, pshared, value)`: `pshared=0` — между потоками; `pshared=1` —
между процессами (тогда сам `sem_t` должен лежать в разделяемой памяти, мост к
упражнениям 02/03).

## Семафор vs мьютекс vs futex

- **Мьютекс/futex** (упр. 04) — взаимное исключение (один в критической секции).
- **Семафор** — счётчик ресурсов (N разрешений): подходит для пулов, ограничения
  параллелизма, ограниченных очередей. Бинарный семафор (value=1) ≈ мьютекс, но
  семафор может «постить» другой поток (сигналинг), а мьютекс владеет один.
- Под капотом POSIX-семафор в Linux тоже стоит на **futex** (быстрый путь без
  сисколла).

## C-эквивалент

Идентичные вызовы в C (`sem_init`/`sem_wait`/`sem_post`/`sem_destroy`). Именованный
вариант — `sem_open("/name", ...)` (как shm — персистентный, для несвязанных
процессов).

## Проверка

Автопрогон (ASan/UBSan): производитель-поток шлёт `0..N-1` через очередь
ёмкостью 16 (заведомо меньше N → реальные блокировки на `empty`/`full`),
потребитель принимает строго по порядку; проверяется FIFO и сумма. В конце —
`sq_destroy` (без утечек).
