# Задание: POSIX разделяемая память (`shm_open` + `mmap`)

Разделяемая память — **самый быстрый** IPC: процессы отображают один объект ядра
в своё адресное пространство и работают с ним как с обычной памятью, **без
копирования через ядро** на каждый обмен (в отличие от pipe/socket).

## Что реализовать

```cpp
int   shm_setup(const char* name, size_t size);  // shm_open + ftruncate; fd/-1
void* shm_map(int fd, size_t size);              // mmap MAP_SHARED RW; ptr/MAP_FAILED
```

### `shm_setup`
```cpp
int fd = shm_open(name, O_CREAT | O_RDWR, 0600);  // имя вида "/myobj", виден в /dev/shm
if (fd < 0) return -1;
if (ftruncate(fd, (off_t)size) != 0) { close(fd); return -1; }  // задать размер
return fd;
```

### `shm_map`
```cpp
return mmap(nullptr, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
```

`MAP_SHARED` критичен: только он делает записи видимыми другим отображениям/
процессам. `MAP_PRIVATE` дал бы copy-on-write приватную копию (для IPC бесполезен).

## Жизненный цикл (важно понимать)

- `shm_open` создаёт **именованный** объект, который **переживает** процесс —
  его надо явно удалять через `shm_unlink(name)` (иначе «утечёт» в /dev/shm).
- `mmap` отображает; `munmap` отменяет отображение (но объект жив, пока не
  `shm_unlink`).
- Это как файл: открытие/размер/отображение/удаление — раздельные операции.

## C-эквивалент

Чистые сисколлы, идентичные в C (на старых glibc нужен `-lrt`; на современных —
нет). Альтернатива POSIX shm — SysV (`shmget`/`shmat`, старый API с ключами) и
анонимная `mmap(MAP_SHARED|MAP_ANONYMOUS)` для родственных процессов (упражнение
03).

## Проверка

Автопрогон (ASan/UBSan): создаём объект, делаем **два** независимых отображения,
пишем структуру через одно — проверяем, что видно через другое (и наоборот). Имя
объекта уникально на процесс; в конце — `munmap` + `close` + `shm_unlink`
(без утечек памяти и объектов).
