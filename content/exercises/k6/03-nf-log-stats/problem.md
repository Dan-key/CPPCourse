# Задание: счётчик пакетов по протоколу через per-CPU + procfs

Новичок, желая «логировать» трафик, ставит `printk` в хук — и кладёт throughput в
разы (хук бежит на **каждый** пакет в softirq, лог сериализуется под локом консоли,
§15). Правильный способ «наблюдать» сетевой hot path — **считать** в per-CPU
счётчики (без контеншна между ядрами) и отдавать агрегат **редко**, через `/proc`.

Этот модуль — синтез **K2** (per-CPU, §9), **K5** (procfs/seq_file) и **K6** (хук):
- хук на `LOCAL_IN` считает пакеты по протоколу (TCP/UDP/ICMP/прочее);
- `/proc/k6_stats` печатает суммарные счётчики по всем ядрам.

## Что реализовать

Procfs и агрегация **уже даны** — изучи их. Допиши **только** инкремент в `count_hook`
(в этом и весь K6-урок: запись на hot path без локов):

```c
switch (iph->protocol) {
case IPPROTO_TCP:  this_cpu_inc(k6_stats.tcp);  break;
case IPPROTO_UDP:  this_cpu_inc(k6_stats.udp);  break;
case IPPROTO_ICMP: this_cpu_inc(k6_stats.icmp); break;
default:           this_cpu_inc(k6_stats.other); break;
}
```

## Почему per-CPU, а не общий счётчик

Из-за **RPS/RFS** (§1.5) твой хук исполняется **параллельно на разных ядрах** для
разных потоков. Общий `counter++` из нескольких softirq на нескольких CPU — гонка
(потерянные инкременты, §16.2). Варианты синхронизации:

- **per-CPU** (`this_cpu_inc`) — у каждого ядра свой экземпляр, инкремент **без
  локов и без межъядерного трафика** когерентности → идеально для hot path;
- агрегация (сумма по `for_each_possible_cpu`) — дорогая, но делается **только** на
  редком чтении `/proc`, не на каждом пакете.

Это прямой аналог `MODULE_AUTHORING`-правила «hot path дешёвый, дорогая работа —
вне его».

## Порядок (раз)регистрации — важен

В `init`: сначала `proc_create`, потом `nf_register_net_hook` (хук — последним,
когда данные готовы). В `exit` — **зеркально**: сначала `nf_unregister_net_hook`
(он дождётся завершения in-flight вызовов хука, §16.1), потом `proc_remove`. Снять
хук **до** освобождения того, что он трогает, — иначе race/UAF при выгрузке.

## Хук в softirq — спать нельзя

`this_cpu_inc` атомарен относительно вытеснения и безопасен в softirq. Никаких
`mutex`/`GFP_KERNEL`/`msleep` в хуке (§1.4).

## Проверка

QEMU: модуль грузится, тест шлёт `ping -c3 127.0.0.1` (3 echo + 3 reply проходят
`LOCAL_IN`), читает `/proc/k6_stats` и проверяет, что `icmp >= 3`. После — `rmmod`,
проверка, что `/proc/k6_stats` удалён, и грep `dmesg` на `BUG:/Oops/WARNING:`.

Для охоты на гонку счётчика в реальном коде — собери ядро с KASAN/lockdep (K2 §15) и
дай параллельную нагрузку на несколько ядер.
