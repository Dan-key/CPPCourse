# Задание: латентность функции ядра через kretprobe

kprobe (упражнение 01) ловит **вход**. Но самый частый вопрос — не «сколько раз», а
«**сколько длится** функция ядра». Для этого нужна **kretprobe**: она перехватывает
**возврат** функции, а пара `entry_handler`/`handler` даёт «время на входе → время на
выходе → дельта» (§6.2, §7.3).

Здесь ты меришь латентность `do_sys_openat2` и отдаёшь `count`/`last_ns`/`max_ns`
через `/proc/k7_lat`. Символ — параметр модуля (нацель на свою функцию драйвера).

## Что реализовать

Регистрация kretprobe, atomic-счётчики и `/proc` уже даны. Допиши два обработчика:

```c
/* ent — на входе: засечь время в ri->data (своё на каждый вызов «в полёте») */
struct lat *d = (struct lat *)ri->data;
d->t0 = ktime_get();

/* ret — на возврате: дельта + накопить */
struct lat *d = (struct lat *)ri->data;
s64 ns = ktime_to_ns(ktime_sub(ktime_get(), d->t0));
atomic64_inc(&count);
atomic64_set(&last_ns, ns);
if (ns > atomic64_read(&max_ns))
	atomic64_set(&max_ns, ns);
```

## Ключевые API и понятия

- **`register_kretprobe(&krp)` / `unregister_kretprobe(&krp)`** — символ в
  `krp.kp.symbol_name`.
- **`entry_handler`** (вход) и **`handler`** (возврат). `ri->data` — буфер на
  **каждый одновременный** вызов (размер = `data_size`); туда кладёшь данные входа,
  забираешь на возврате.
- **`data_size`** — сколько байт `ri->data` резервируется на вызов (здесь
  `sizeof(struct lat)`).
- **`maxactive`** — сколько вызовов функции могут быть «в полёте» одновременно (на
  разных CPU/рекурсивно). Мало → часть возвратов теряется (`krp.nmissed`).
- **`regs_return_value(regs)`** (если нужно) — код возврата функции.

## Почему `ri->data`, а не глобал

Между входом и выходом одной функции может «вклиниться» её же вызов на другом CPU
(или рекурсия). Хранить `t0` в глобале — перепутать времена. `ri->data` — **своё на
каждый экземпляр** вызова, поэтому вход и выход всегда сопоставлены правильно (это та
же проблема, что ключ `tid` в bpftrace, §10.4, и смысл `maxactive`).

## Atomic context — без сна

Оба обработчика — атомарный контекст (§7.4): `ktime_get`/`atomic64_*` можно,
`mutex`/`GFP_KERNEL`/`msleep` — нельзя.

## Проверка

QEMU: модуль грузится, тест делает несколько `cat` (вызовы целевой функции), читает
`/proc/k7_lat` и проверяет, что `count > 0` (латентность измерялась). После — `rmmod`
и грep `dmesg`.

Самостоятельно: сравни с `funclatency do_sys_openat2` (bcc) — должна совпасть форма
гистограммы; добавь в свой модуль per-CPU-гистограмму вместо `last/max`.
