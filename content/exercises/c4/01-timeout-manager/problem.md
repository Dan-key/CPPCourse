# Задание: менеджер таймаутов на min-heap (один timerfd на тысячи таймеров)

Это структура данных, которая стоит за **таймаутами соединений в каждом сервере**.
У тебя тысячи соединений, у каждого свой дедлайн неактивности. Заводить по
`timerfd` на соединение — расточительно. Правильно: один менеджер таймаутов,
который умеет отдать **ближайший** дедлайн (чтобы вооружить **один** `timerfd`) и
забрать все **истёкшие**.

## Что реализовать

```cpp
void   add(uint64_t id, uint64_t deadline_ms);   // запланировать таймер
void   cancel(uint64_t id);                       // отменить (ленивое удаление)
bool   next_deadline(uint64_t* out);              // ближайший дедлайн (живых); false если пусто
size_t expire(uint64_t now_ms, uint64_t* out, size_t max);  // забрать все с deadline<=now
```

Структура — **min-heap** по дедлайну (`std::priority_queue` уже объявлен в стартере)
плюс множество отменённых (`std::unordered_set`) для **ленивого удаления**.

## Алгоритм

- **`add`** — `heap.push({deadline, id})`. O(log n).
- **`cancel`** — НЕ ищем в куче (дорого), а помечаем id отменённым:
  `cancelled.insert(id)`. O(1). Реальное удаление произойдёт при извлечении.
- **`next_deadline`** — снять с вершины кучи все отменённые (lazy cleanup), затем
  вернуть дедлайн вершины. Если куча опустела — `false`.
- **`expire(now)`** — пока вершина имеет `deadline <= now`: снять её; если id
  отменён — пропустить; иначе записать id в `out`. Куча отдаёт по возрастанию
  дедлайна, поэтому результат **уже отсортирован** по дедлайну.

```cpp
size_t expire(uint64_t now, uint64_t* out, size_t max) {
    size_t n = 0;
    while (!heap_.empty() && heap_.top().deadline <= now && n < max) {
        Node top = heap_.top(); heap_.pop();
        auto it = cancelled_.find(top.id);
        if (it != cancelled_.end()) { cancelled_.erase(it); continue; } // отменён — выкинуть
        out[n++] = top.id;
    }
    return n;
}
```

## Почему min-heap, а не отсортированный список

`add` и `expire` — O(log n); `next_deadline` — O(1) (после ленивой чистки).
Отсортированный массив дал бы O(n) на вставку. Хешированное «колесо таймеров»
(timing wheel) даёт амортизированную O(1), но сложнее; min-heap — отличный дефолт
и то, что используют большинство event-loop'ов (libevent, и т.п.).

## Связь с timerfd (C2/C4)

В реальном loop: вооружаешь **один** `timerfd` на `next_deadline() - now`; когда он
сработал — `expire(now)` закрывает все истёкшие таймеры, затем перевооружаешь на
новый ближайший. Так один дескриптор обслуживает десятки тысяч таймеров. Ленивая
отмена нужна, потому что соединения закрываются постоянно, а ковыряться в куче на
каждое закрытие — дорого.

## Проверка

Автопрогон (ASan/UBSan): дедлайны, порядок извлечения по возрастанию, отмена с
пропуском, и стресс на 5000 таймеров с отменой каждого третьего — проверяется
корректность кучи и точный учёт.
