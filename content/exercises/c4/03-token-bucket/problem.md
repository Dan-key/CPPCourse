# Задание: token bucket — rate limiter

Алгоритм, который стоит за каждым «не более N запросов в секунду» — API-лимиты,
throttling, QoS, защита от перегрузки. В «ведре» лежат **токены**; каждый запрос
забирает один; токены **доливаются** с постоянной скоростью до ёмкости. Ключевое
свойство: допускает **всплеск** (burst до `capacity`), но в среднем держит
заданную скорость.

## Что реализовать

```cpp
bool allow(uint64_t now_ms);   // true = пропустить, false = превышен лимит
```

Конструктор задаёт `capacity` (размер всплеска) и `rate_per_sec` (скорость
долива); ведро стартует **полным**. Поля (`tokens_`, `last_ms_`,
`rate_per_ms_`) уже объявлены.

## Алгоритм («ленивый» долив по времени)

```cpp
bool allow(uint64_t now) {
    double elapsed = (double)(now - last_ms_);            // прошло мс
    tokens_ = std::min(cap_, tokens_ + elapsed * rate_per_ms_);  // долить, не выше ёмкости
    last_ms_ = now;
    if (tokens_ >= 1.0) { tokens_ -= 1.0; return true; }  // есть токен → забрать
    return false;                                          // лимит превышен
}
```

«Ленивый» долив (пересчёт при каждом запросе) дешевле, чем будить `timerfd` на
каждый токен: мы просто считаем, сколько токенов «накапало» за прошедшее время.

## Почему именно token bucket, а не «счётчик в окне»

- **Гладкость:** простой счётчик «N в секунду» даёт резкие границы окна (на стыке
  можно пропустить 2N). Token bucket сглаживает — скорость долива постоянна.
- **Всплески:** `capacity > 1` разрешает короткие пачки (реальный трафик
  неравномерен), но среднюю скорость держит. `capacity` = «бюджет всплеска».
- **Дёшево:** O(1) на запрос, одно число состояния.

Родственник — **leaky bucket** (выливается с постоянной скоростью, всплески
сглаживает жёстче). Token bucket популярнее за гибкость.

## Связь с timerfd (C4)

Долив можно привязать к `timerfd` (подливать R токенов каждый тик), но ленивый
расчёт по времени (как здесь) обычно лучше — нет лишних пробуждений. timerfd
пригодится, если надо **будить** ждущих, когда токен станет доступен.

## Проверка

Автопрогон (ASan/UBSan): стартовый всплеск == ёмкости; долив 1 токен за 1 мс при
скорости 1000/с; долив **ограничен ёмкостью** (после долгой паузы доступно не
больше `capacity`, а не бесконечность); средняя скорость держится при равномерном
потоке запросов.
