# Задание: конечный автомат graceful shutdown демона

Это **реальная управляющая логика сервиса** — то, что отличает «убили процесс» от
«корректно остановили, не оборвав запросы». События обычно приходят из `signalfd`
(SIGTERM/SIGHUP, C2 §7.2) и таймера (`timerfd` дедлайна). Здесь ты реализуешь
**ядро** этой логики — конечный автомат.

## Состояния и события

```
Состояния:  RUNNING (0)  — работаем, принимаем запросы
            DRAINING (1) — перестали принимать новые, доделываем активные
            STOPPING (2) — выходим (терминальное)

События:    SIGTERM (0)  — просьба остановиться
            SIGHUP  (1)  — перечитать конфиг на лету
            DRAINED (2)  — все активные соединения завершились
            TIMEOUT (3)  — истёк дедлайн graceful-остановки
```

## Что реализовать

Метод `on(int event)` — таблица переходов:

| Состояние | Событие | → Состояние | Действие |
|-----------|---------|-------------|----------|
| RUNNING | SIGTERM | DRAINING | начать graceful (перестать `accept`) |
| RUNNING | SIGHUP | RUNNING | `reloads_++` (горячая перезагрузка) |
| RUNNING | DRAINED/TIMEOUT | RUNNING | игнор |
| DRAINING | DRAINED | STOPPING | все доделали — штатный выход |
| DRAINING | TIMEOUT | STOPPING | дедлайн — принудительный выход |
| DRAINING | **SIGTERM** | **STOPPING** | **второй SIGTERM = «хватит ждать», форс** |
| DRAINING | SIGHUP | DRAINING | игнор (не перезагружаемся на выходе) |
| STOPPING | * | STOPPING | терминальное |

## Почему это важно

Эта логика — суть **корректного деплоя**: по `SIGTERM` сервис перестаёт брать
новые запросы, доделывает текущие (DRAINING), и только потом выходит — апдейт не
рвёт активные соединения. **Второй** `SIGTERM` (нетерпеливый оператор/оркестратор)
форсирует немедленный выход. **Дедлайн** (`timeout`) не даёт зависнуть навечно,
если соединения не сливаются. `SIGHUP` на лету перечитывает конфиг **без**
перезапуска (классика демонов). Это ровно поведение, которого ждёт systemd
(`TimeoutStopSec`, `ExecReload`) — мост к C5.

## Почему именно автомат, а не куча `if`

Явный FSM делает логику **полной и проверяемой**: каждое сочетание (состояние,
событие) имеет определённый исход, нет «дыр», где сервис застрянет. Это и есть
инженерная разница между «вроде останавливается» и «останавливается корректно
всегда».

## Проверка

Автопрогон (ASan/UBSan) гоняет 5 сценариев: штатный graceful (SIGHUP→reload,
SIGTERM→drain→DRAINED→stop), форс по второму SIGTERM, выход по дедлайну,
множественные SIGHUP, игнор «лишних» событий. Реализуй таблицу — и все пройдут.
