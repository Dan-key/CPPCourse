# Задание: signalfd — дренаж и семантика coalescing vs queueing

`signalfd` превращает доставку сигналов в **читаемые события** — их обрабатывают
синхронно в event-loop, без опасной async-signal-safety (C2 §7.2). Реализуя
корректный «дренаж» signalfd, ты заодно увидишь **фундаментальную семантику Unix**,
которую не покажет ни один учебник лучше, чем тест: стандартные сигналы при
блокировке **схлопываются**, а сигналы реального времени **очередятся**.

## Что реализовать

```cpp
int sig_drain(int sfd, int* counts, int counts_len);
```

Вычитать **все** доступные записи из неблокирующего `signalfd` и посчитать сигналы
по номеру (`counts[ssi_signo]++`). Вернуть общее число записей, `-1` при ошибке.

```cpp
int sig_drain(int sfd, int* counts, int counts_len) {
    int total = 0;
    for (;;) {
        struct signalfd_siginfo si;
        ssize_t n = read(sfd, &si, sizeof si);     // одна запись фикс. размера
        if (n == (ssize_t)sizeof si) {
            if ((int)si.ssi_signo < counts_len) counts[si.ssi_signo]++;
            ++total;
        } else if (n < 0) {
            if (errno == EINTR) continue;
            if (errno == EAGAIN || errno == EWOULDBLOCK) break;  // вычерпали всё
            return -1;
        } else break;
    }
    return total;
}
```

## Семантика, которую раскрывает тест (это и есть «интересное»)

Сигналы заблокированы (`sigprocmask`) и собираются через `signalfd`:

- **СТАНДАРТНЫЙ сигнал** (`SIGUSR1`), посланный **3 раза** подряд при блокировке,
  становится **одним** ожидающим: `counts[SIGUSR1] == 1`. Стандартные сигналы
  **не очередятся** — повторные «теряются» (точнее, схлопываются в флаг «есть
  ожидающий»). Поэтому на счётчик событий стандартными сигналами **полагаться
  нельзя**.
- **Сигнал реального времени** (`SIGRTMIN..SIGRTMAX`), посланный 3 раза,
  **очередится**: `counts[SIGRTMIN] == 3`. RT-сигналы доставляются по одному в
  порядке отправки (до лимита `RLIMIT_SIGPENDING`), и даже несут данные
  (`sigqueue`).

Вывод, который надо унести: если важно **не потерять** уведомления — стандартные
сигналы не годятся (используй RT-сигналы, `eventfd`, или сам факт «что-то
произошло» + проверку состояния). Это объясняет, почему демоны на `SIGCHLD`
**обязаны** в цикле `waitpid(WNOHANG)` собирать ВСЕХ детей: один `SIGCHLD` мог
«схлопнуть» несколько завершений.

## Почему это не обёртка

Смысл задачи — не «вызвать read», а: (1) написать **корректный** неблокирующий
дренаж до `EAGAIN` (как в C2 — частая ошибка остановиться рано), и (2) понять и
продемонстрировать **поведение ядра**, которое ломает наивные предположения о
счёте сигналов.

## Проверка

Автопрогон (ASan/UBSan): блокирует сигналы, шлёт `SIGUSR1`×3, `SIGUSR2`×1,
`SIGRTMIN`×3; твой дренаж должен дать `counts[SIGUSR1]==1` (схлоп),
`counts[SIGRTMIN]==3` (очередь), `total==5`; повторный дренаж пустого → `0` (вышел
по `EAGAIN`, не завис).
