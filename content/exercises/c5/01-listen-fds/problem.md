# Задание: socket activation — разбор переданных дескрипторов (`sd_listen_fds`)

Это **ядро socket activation** systemd. systemd сам открывает слушающий сокет
(`.socket`-юнит), и при первом подключении запускает твой сервис, **передав ему уже
готовый сокет по наследству** — на фиксированных номерах дескрипторов, начиная с
**3** (`SD_LISTEN_FDS_START`). Сервису не нужно `bind`/`listen` самому: он узнаёт,
сколько и каких сокетов ему дали, через **переменные окружения**.

Так решается классическая проблема запуска: сокет открыт **до** старта сервиса,
поэтому клиенты, постучавшиеся в момент рестарта, **не получают `ECONNREFUSED`** —
ядро держит их в backlog, пока сервис поднимается. И права (привязка к порту < 1024)
берёт на себя systemd, а сервис работает без рута.

## Протокол (как договорились systemd и сервис)

- `LISTEN_PID` — PID процесса, которому **предназначены** дескрипторы. Сервис
  обязан **сверить** его со своим `getpid()`: переменные могли «утечь» в потомка,
  и применять их не к тому процессу — баг.
- `LISTEN_FDS` — **число** переданных дескрипторов `N`. Они занимают подряд номера
  `3, 4, …, 3 + N − 1`.
- `LISTEN_FDNAMES` — необязательные **имена** сокетов через `:` (из `FileDescriptorName=`),
  по одному на дескриптор: `"http:metrics"` → fd 3 = http, fd 4 = metrics.

## Что реализовать

```cpp
// Сколько дескрипторов передано ИМЕННО нам. 0, если не для нас или мусор.
int my_listen_fds(int my_pid, const char* listen_pid, const char* listen_fds);

// Номер дескриптора по имени сокета (3 + индекс), или -1 если имени нет.
int my_listen_fd_by_name(int my_pid, const char* listen_pid,
                         const char* listen_fds, const char* listen_fdnames,
                         const char* name);
```

Значения переменных окружения передаются **строками-аргументами** (как `getenv`
вернул бы; `nullptr` = переменная не задана) — чтобы тест мог гонять сценарии без
реального `setenv`.

## Алгоритм

`my_listen_fds`:
1. `listen_pid` пуст/`nullptr` → **0** (нас не активировали).
2. распарсить `listen_pid` как число; если **≠ `my_pid`** → **0** (не нам).
3. `listen_fds` пуст/нечисло/отрицательное → **0**.
4. иначе вернуть это число `N`. Дескрипторы — это `3 .. 3+N-1`.

`my_listen_fd_by_name`:
1. `N = my_listen_fds(...)`; если `0` или `listen_fdnames == nullptr` → **-1**.
2. разбить `listen_fdnames` по `:` максимум на `N` имён;
3. найти индекс `i`, где имя == `name` → вернуть **`3 + i`** (`SD_LISTEN_FDS_START + i`);
4. не нашли → **-1**.

```cpp
constexpr int SD_LISTEN_FDS_START = 3;
```

## Почему это важно (и причём тут zero-downtime)

Socket activation — фундамент **zero-downtime reload**: новый экземпляр сервиса
получает **тот же** слушающий сокет (то же открытое файловое описание), не делая
`bind` заново. Пока старый дренирует in-flight запросы, новый уже принимает — порт
ни на миг не «падает». Это надёжнее самодельной передачи fd через `SCM_RIGHTS`
(C3), потому что сокетом владеет systemd, переживающий рестарты сервиса.

Сверка `LISTEN_PID == getpid()` — не формальность: без неё процесс может ошибочно
принять за «свои» дескрипторы, предназначенные родителю, и читать чужой сокет.

## C-эквивалент

В реальном коде это `sd_listen_fds(0)` из libsystemd (или `sd_listen_fds_with_names`).
Здесь ты пишешь её сам на чистом C/C++ — увидеть, что «магия» systemd — это всего
лишь три переменные окружения и соглашение о номерах fd.

## Проверка

Автопрогон (ASan/UBSan): корректный PID + `LISTEN_FDS="2"` → 2; чужой PID → 0;
пустые/мусорные значения → 0; поиск по имени (`"http:metrics"`, want `"metrics"`)
→ fd 4; отсутствующее имя → -1. Реализуй обе функции — все пройдут.

Запусти и под TSan, если будешь вызывать из нескольких потоков (функции
реентерабельны — не держат глобального состояния):

```sh
g++ -std=c++20 -fsanitize=thread -O1 -g solution.cpp test.cpp -o prog && ./prog
```
