# Задание: graceful drain — не потерять in-flight запросы при остановке

Корректный деплой требует **drain** (слива): по сигналу остановки сервис **перестаёт
принимать новые** запросы, но **доделывает уже принятые** (in-flight), и только когда
их не осталось — выходит. Без этого апдейт рвёт активные запросы: пользователь видит
обрыв, оркестратор — всплеск 5xx. Здесь ты реализуешь **учёт in-flight запросов** —
ядро drain-логики, которое в реальном сервисе кормят `signalfd` (`SIGTERM`) и
`timerfd` (дедлайн), C4.

Это **дополняет** автомат из `04-shutdown-fsm` (C4): там была реакция на события
сигналов; здесь — **подсчёт активных запросов** и переход в «остановлено» ровно
тогда, когда последний запрос завершился (или истёк дедлайн).

## Состояния

```
0 = RUNNING  — принимаем новые запросы
1 = DRAINING — новые НЕ принимаем, доделываем активные (in-flight)
2 = STOPPED  — активных не осталось (или форс по дедлайну); терминальное
```

## Что реализовать

```cpp
struct Drain;                          // непрозрачный
Drain* drain_create();
void   drain_destroy(Drain*);

int    drain_admit(Drain*);            // НАЧАТЬ запрос: 1 принят (active++), 0 отклонён
void   drain_finish(Drain*);           // запрос завершился: active-- (и, может, → STOPPED)
void   drain_begin_shutdown(Drain*);   // RUNNING → DRAINING (перестать принимать)
void   drain_timeout(Drain*);          // дедлайн истёк → форс STOPPED
int    drain_state(Drain*);            // 0/1/2
int    drain_active(Drain*);           // число in-flight запросов
```

## Логика

- **`drain_admit`**: если `state == RUNNING` → `active++`, вернуть `1`. Иначе
  (DRAINING/STOPPED) → вернуть `0` (новые запросы при остановке **отклоняются** —
  это и есть «перестали принимать»).
- **`drain_finish`**: если `active > 0` → `active--`. Если после этого
  `state == DRAINING && active == 0` → `state = STOPPED` (**последний** in-flight
  доделан — можно выходить).
- **`drain_begin_shutdown`**: если `state == RUNNING` → `state = DRAINING`. Если при
  этом `active == 0` (нечего сливать) → сразу `state = STOPPED`.
- **`drain_timeout`**: `state = STOPPED` (форс — не ждём вечно зависшие запросы).
- **STOPPED** терминально: `admit` отклоняет, `begin_shutdown`/`finish` не оживляют.

## Почему именно так

- **`admit`/`finish` оборачивают каждый запрос:** инкремент при приёме, декремент
  при завершении (как RAII-счётчик — в C++ это `struct Guard{ Drain* d; ~Guard(){
  drain_finish(d);} };`). Так сервис **всегда** знает, сколько запросов в полёте.
- **Переход в STOPPED по `active==0`** — гарантия, что мы вышли, **отдав** все
  ответы: ни один in-flight не потерян.
- **Дедлайн (`drain_timeout`)** — страховка от зависших запросов: если клиент не
  дочитывает или обработчик завис, нельзя ждать вечно — выходим принудительно (но
  это уже «грязный» выход, его минимизируют разумным `TimeoutStopSec`).
- **Отказ новых в DRAINING** — то, что в реальном сервере делают, снимая
  listen-сокет с `epoll` (C4 §18) или отвечая `503` с `Connection: close`.

Связь с zero-downtime: при socket activation (упр. 01) **новый** экземпляр уже
принимает на том же сокете, пока **старый** дренирует свои in-flight — порт не
«падает» ни на миг. Этот трекер — про корректный выход **старого**.

## C-эквивалент

Тот же подсчёт активных запросов — обычный `int active` под атомиком/локом (в
многопоточном сервере — `std::atomic<int>`, C1). Здесь однопоточная модель ядра
логики; в реальном коде оборачиваешь `admit`/`finish` вокруг обработки соединения.

## Проверка

Автопрогон (ASan/UBSan): приём 3 запросов; `begin_shutdown` → DRAINING, новые
отклоняются; `finish`×3 → на последнем STOPPED; пустой drain (0 активных) → сразу
STOPPED; `timeout` форсирует STOPPED при активных; после STOPPED `admit` отклоняет.
Реализуй методы — все пройдут.

атомиком/локом (C1):
