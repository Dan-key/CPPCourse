# Задание: watchdog — не зависнуть незаметно

Сервис может **не упасть, но зависнуть**: event-loop застрял, поток в дедлоке,
обработчик крутится в бесконечном цикле. Снаружи процесс «жив» (есть в `ps`), а
запросы не обслуживает — худший вид аварии. **Watchdog** systemd ловит именно это:
сервис обязан **периодически отчитываться** «я жив» (`sd_notify("WATCHDOG=1")`); не
отчитался вовремя — systemd считает его повисшим и **перезапускает** (по
`WatchdogSec=` в юните).

Здесь ты реализуешь **пейсер watchdog**: разбор окружения и логику «пора ли слать
очередной `WATCHDOG=1`».

## Протокол

systemd передаёт сервису:
- `WATCHDOG_USEC` — период watchdog в **микросекундах** (из `WatchdogSec=`). Не
  задан/`0` — watchdog **выключен**.
- `WATCHDOG_PID` — PID, которому предназначен watchdog (как `LISTEN_PID`). Если
  задан и **≠ нашему** — watchdog не наш (выключен). Не задан — считаем, что наш.

**Канонический интервал отправки — половина периода:** `WATCHDOG_USEC / 2`. Половина,
а не весь период, — это запас на джиттер планировщика и сети, чтобы не пропустить
дедлайн из-за случайной задержки.

## Что реализовать

```cpp
struct Wd;                                   // непрозрачный
Wd*  wd_create(int my_pid, const char* watchdog_usec, const char* watchdog_pid);
void wd_destroy(Wd*);
int  wd_enabled(Wd*);                        // 1 если watchdog включён и наш
unsigned long long wd_interval_usec(Wd*);    // период отправки = WATCHDOG_USEC/2 (0 если выкл.)
int  wd_should_ping(Wd*, unsigned long long now_usec);  // пора слать WATCHDOG=1? (0/1)
```

Значения окружения — строками (`nullptr` = не задано), время — инъектируемое
`now_usec` (тест задаёт «часы» детерминированно).

## Логика

- **`wd_create`**: распарсить `watchdog_usec` (беззнаковое). `enabled` ⇔
  `usec > 0` **и** (`watchdog_pid` пуст **или** `число(watchdog_pid) == my_pid`).
  Запомнить `interval = usec / 2`.
- **`wd_should_ping(now)`**:
  - если выключен → `0`;
  - **первый** вызов → `1` (шлём сразу, запоминаем `now` как время последнего пинга);
  - иначе если `now - last_ping >= interval` → `1`, обновить `last_ping = now`;
  - иначе → `0`.

```cpp
int wd_should_ping(Wd* w, unsigned long long now) {
    if (!w->enabled) return 0;
    if (!w->pinged || now - w->last >= w->interval) {
        w->pinged = true; w->last = now; return 1;
    }
    return 0;
}
```

## Почему так

- **Половина периода** даёт двойной запас: даже пропустив один пинг из-за всплеска
  нагрузки, ты успеешь до дедлайна `WATCHDOG_USEC`.
- **Сверка PID** — та же защита, что у socket activation: не принять чужой watchdog.
- **Инъекция времени** (а не `clock_gettime` внутри) делает логику тестируемой и
  детерминированной — в реальном loop ты сам зовёшь `wd_should_ping(now_monotonic())`
  на каждой итерации и при `1` шлёшь `notify_watchdog` (упр. 02).

В реальном loop watchdog-пинг вешают на `timerfd` с периодом `interval` (C4) или
проверяют на каждой итерации `epoll_wait` — главное, чтобы зависший обработчик
**не успевал** отправить пинг, и тогда systemd перезапустит сервис.

## Проверка

Автопрогон (ASan/UBSan): период из `WATCHDOG_USEC` делится пополам; первый пинг —
сразу; следующий — ровно через интервал, не раньше; watchdog выключается при пустом
`WATCHDOG_USEC` и при чужом `WATCHDOG_PID`. Реализуй пейсер — все пройдут.
