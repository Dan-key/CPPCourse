# Задание: false sharing — выравнивание по линии кэша

Память когерентна **линиями по 64 байта**, не по переменным. Если два
независимых счётчика, которые пишут **разные** потоки, попадают в одну линию —
каждая запись инвалидирует линию у соседнего ядра, и линия пинг-понгует между
кэшами (cache line bouncing). Логически конфликта нет (*false*), но железо его
создаёт (*sharing*), и масштабируемость рушится.

## Что реализовать

В `solution.cpp` структура `Cell` сейчас содержит только `std::atomic<long> v` и
**не выровнена**, поэтому `c_[0]` и `c_[1]` лежат в одной линии. Разнеси их по
разным линиям — добавь выравнивание:

```cpp
struct alignas(64) Cell {           // ← каждый Cell с границы линии кэша
    std::atomic<long> v{0};
};
```

В C++17+ каноничнее использовать
`alignas(std::hardware_destructive_interference_size)` — это «правильный» размер
линии для текущей платформы (обычно 64). Эквивалент через padding:

```cpp
struct Cell {
    std::atomic<long> v{0};
    char pad[64 - sizeof(std::atomic<long>)];
};
```

После правки `gap()` (расстояние между счётчиками) обязан стать `>= 64`.

## Замер эффекта (руками — это часть задания из трека)

Собери и засеки время двух вариантов — выровненного и невыровненного:

```sh
# Выровненный (твоё решение):
g++ -std=c++20 -O2 -pthread solution.cpp test.cpp -o good && time ./good

# «Плохой»: временно убери alignas/padding из Cell и пересобери:
g++ -std=c++20 -O2 -pthread solution.cpp test.cpp -o bad  && time ./bad
```

На многоядерной машине выровненная версия обычно в **2–5 раз** быстрее на этом
цикле инкрементов — при идентичной логике. Точную картину покажет:

```sh
perf c2c record ./bad && perf c2c report     # подсветит конфликтующую линию и ядра
perf stat -e cache-misses ./bad ./good       # сравни промахи
```

## C-эквивалент (реальность ядра)

На C11 — `_Alignas(64) _Atomic long v;` или padding-поля; в ядре Linux для этого
есть `____cacheline_aligned` и per-CPU переменные, чтобы счётчики разных ядер
вообще не делили линию. Принцип один: **разделяемая запись в одну линию — точка
сериализации**, убирай её.

## Проверка

Серверный автопрогон (ASan/UBSan) проверяет: `gap() >= 64` (счётчики на разных
линиях) и функциональную корректность — два потока по `ITERS` инкрементов, итог
каждого счётчика обязан быть ровно `ITERS`. Сам **замер** скорости делается
вручную командами выше: ASan искажает тайминги, мерь под `-O2` без санитайзеров.
