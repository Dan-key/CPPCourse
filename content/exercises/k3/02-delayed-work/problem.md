# Задание: отложить обработку НА ВРЕМЯ через delayed_work

В `01-workqueue` ты отложил работу «как можно скорее» (`schedule_work`). Часто работу
надо отложить **на интервал**: опросить устройство через 100 мс, повторить попытку,
выждать таймаут. Для этого — **`delayed_work`** (workqueue + таймер): таймер истекает
через заданную задержку → работа ставится в очередь → `work_fn` бежит в **process
context** (можно спать).

Здесь ты строишь misc-устройство `/dev/cppdw`, где `write` планирует обработку
**через 100 мс** (`schedule_delayed_work`), отложенная работа переводит данные в
верхний регистр и считает число запусков, а `read` дожидается результата
(`flush_delayed_work`). При выгрузке — `cancel_delayed_work_sync` (синхронизация с
остановкой: иначе `kworker` дёрнет `work_fn` после выгрузки → паника).

## Что реализовать

- **`write(text)`** — сохранить данные и **`schedule_delayed_work(&dwork,
  msecs_to_jiffies(100))`** (обработка не сейчас, а через 100 мс);
- **`dwork_fn`** (process context) — под `lock` перевести вход в **верхний регистр** в
  буфер результата, увеличить счётчик запусков `runs`;
- **`read`** — **`flush_delayed_work(&dwork)`** (дождаться отложенной работы) и отдать
  `"<RESULT> runs=<n>\n"`;
- **`exit`** — `misc_deregister` → **`cancel_delayed_work_sync(&dwork)`** (порядок:
  снять устройство, потом добить отложенную работу).

## Ключевые API

- **`INIT_DELAYED_WORK(&dwork, dwork_fn)`** — связать `delayed_work` с функцией.
- **`schedule_delayed_work(&dwork, delay)`** — выполнить `dwork_fn` через `delay`
  (в jiffies; `msecs_to_jiffies(100)` = 100 мс). Под капотом — таймер + workqueue.
- **`flush_delayed_work(&dwork)`** — дождаться завершения (если запланирована/бежит).
- **`cancel_delayed_work_sync(&dwork)`** — отменить **и** дождаться; останавливает
  встроенный таймер. В `exit` — обязательно, **до** освобождения ресурсов.

## Чем отличается от обычной work (`01`)

`schedule_work` ставит работу в очередь сразу («как дойдут руки»). `schedule_delayed_work`
ставит её **через таймер** — с заданной задержкой. Колбэк всё равно бежит в process
context (можно спать), в отличие от колбэка «голого» `timer_list`, который атомарный
(см. `03-timer-source`).

## Проверка

QEMU: `echo -n "abc" > /dev/cppdw` (планирует отложенную работу), `cat` →
`"ABC runs=1"` (работа выполнилась, перевела в верхний регистр); ещё раз
`echo -n "Xy"` → `"XY runs=2"`. `dmesg` должен быть чист.
