# Правила ревью модуля курса «Трек системного программиста»

> Это спецификация для ревьюера (человека или ИИ), парная к `MODULE_AUTHORING.md`.
> `MODULE_AUTHORING.md` отвечает на вопрос «как написать модуль», этот файл — «как
> проверить, что написанное верно». Соблюдай дословно: цель ревью — чтобы студент
> не выучил неправду и не скопировал код, который не компилируется.

---

## 0. Принцип: НИЧЕГО НА ВЕРУ

Два железных правила, нарушение любого = ревью не сделано:

1. **Каждый листинг компилируется.** Не «выглядит правильно» — а собран компилятором
   с теми же флагами, что и прогонщик. «Похоже на рабочий код» — не аргумент.
2. **Каждый факт проверяется по первоисточнику.** `man`, исходники glibc/ядра,
   текст стандарта, а если их недостаточно — **поиск в интернете**. Числа, версии,
   атрибуции («это BSD», «с ядра 2.6.11»), коды ошибок — самые частые места вранья.

Если факт проверить нечем — он не утверждается как факт, а помечается как допущение.

## 1. Оси ревью

| Ось | Инструмент | Что ловим |
|-----|-----------|-----------|
| Формат/структура | `head`, счёт ` ``` `, `wc -l` | «md поедет», недобор объёма |
| Код в лекции | `gcc`/`g++` с флагами §4.1 авторинга | не компилируется, UB, deprecated API |
| Факты | `man`, исходники, стандарт, **веб-поиск** | неверные числа/версии/механизмы |
| Кросс-ссылки | grep по `§` | ссылка ведёт не туда |
| Самопроверка/банк | разбор каждого ответа | ложная посылка, неверный эталон |
| Квиз | `json.load`, разбор `answer` | схема сломана, неверный индекс ответа |
| Упражнения | сборка starter+test и эталона | starter не падает / эталон не проходит |

## 2. Формат и структура (быстрые проверки)

```sh
head -1 content/modules/NN-XX-*.md                 # ровно "# Модуль ..."
python3 -c "s=open('…md').read();print(s.count('\`\`\`')%2==0)"   # ограждения парны
wc -l content/modules/NN-XX-*.md                   # >= 2000 (иначе модуль НЕ готов)
```

Проверь ещё:
- нет YAML-frontmatter (файл начинается с `# `);
- у всех блоков указан язык (` ```c `/` ```cpp `/` ```sh `/` ```text `);
- скелет на месте: Карта модуля → технические разделы → Практика и самопроверка
  (≥10 вопросов) → Банк вопросов (БАЗА/МЕХ/ЭКСПЕРТ/ЗАДАНИЯ) → Что дальше;
- язык кода соответствует этапу (F1–F4 → C17; C1–C6 → двуязычно; K* / EL* → C).

**Недобор строк — это finding уровня модуля.** Фиксируй явно и предлагай, чем
углублять (под капотом, диаграммы, типичные ошибки, реальные CVE).

---

## 3. Код в лекции

1. **Извлеки каждый блок ` ```c `/` ```cpp `** и собери его. Флаги — ровно как у
   прогонщика (`MODULE_AUTHORING.md §4.1`):
   - C: `gcc -std=c17 -Wall -Wextra -Wconversion -Wsign-conversion -fsanitize=address,undefined -O1 -g`
   - C++: `g++ -std=c++20 -Wall -Wextra -fsanitize=address,undefined -O1 -g -pthread`
2. **Предупреждения — это сигнал.** `-Wuninitialized`, `-Wimplicit-function-declaration`,
   `-Wsign-compare` часто = реальный баг (см. историю F1/F3).
3. **«Полный» пример обязан собираться целиком** — включая `#include`. Если листинг
   подан как законченная программа (`int main`, список инклюдов), но использует
   `strerror`/`memcpy`/`NULL` без нужного заголовка — это finding.
4. **Запусти то, что запускается**, под ASan/UBSan — особенно примеры «как надо».
5. **Deprecated/удалённое API.** Современный GCC/Clang по умолчанию = свежий стандарт
   (GCC 15 → C23). Лови то, что отвалилось: `ATOMIC_VAR_INIT` (удалён в C23),
   `gets`, неявные объявления и т.п. Проверяй `__STDC_VERSION__` дефолтного компилятора.
6. **Код vs проза.** Код не должен противоречить правилу, сформулированному рядом
   (пример: один condvar + `signal` на два условия, тогда как текст ниже требует
   `broadcast`). Несогласованность — finding.
7. **«Намеренно битый» код** (демонстрация бага) обязан ломаться **тем механизмом**,
   который описан в комментарии, а не другим. Проверь это компилятором/санитайзером.

## 4. Факты

1. Сверяй с первоисточником в таком приоритете: текст стандарта / `man N` /
   исходники glibc/ядра → авторитетная статья (LWN, cppreference, man7) → прочее.
2. **Обязательно ищи в интернете**, если в утверждении есть:
   - номер/детали CVE;
   - конкретная версия («с ядра X.Y», «POSIX-2008», «Linux 5.3+»);
   - историческая атрибуция («это поведение BSD / System V»);
   - числовая константа по умолчанию (пороги malloc, размеры буферов, диапазоны bin);
   - точная семантика syscall на грани (рестарт после `EINTR`, `connect`, и т.п.).
   Не отвечай по памяти — память врёт именно на числах и версиях.
3. **Указывай источники** в отчёте ссылками — чтобы правку можно было перепроверить.
4. Линуксная специфика vs переносимость: если факт верен только для Linux (или
   наоборот, только portable) — это должно быть сказано явно.

## 5. Кросс-ссылки, самопроверка, банк вопросов

- Каждая внутренняя ссылка `(см. §X)`/`(п. N)` обязана вести **туда, где факт реально
  разобран**. Проверяй переходы (битая ссылка была в F2 §14).
- В «Самопроверке» и «Банке вопросов» разбирай **каждый** ответ/посылку:
  - у открытого вопроса — эталон фактически верен и полон;
  - у вопроса банка без ответа — в формулировке нет ложной посылки
    (пример: «почему `connect()` опасен» — ложно для Linux).
- Опечатки в эталонах тоже фиксируй (они закрепляются студентом как образец).

## 6. Квиз (`quizzes/XX.json`)

```sh
python3 -c "import json;json.load(open('content/quizzes/XX.json'))"   # валиден
```

Сверь со схемой `MODULE_AUTHORING.md §3` и **проверь содержание**:
- `levels` — объект из ровно 4 ключей в порядке БАЗА/МЕХАНИЗМЫ/ЭКСПЕРТ/ЗАДАНИЯ;
- у `mcq` поле `answer` — индекс, и **этот вариант действительно правильный**
  (разбери все options, не доверяй `answer`);
- у `self_grade` `reference` фактически верен;
- `exercise.exercise_id`/`title` согласованы с manifest и папками упражнений;
- у вопросов нет поля `id`.

## 7. Упражнения (`exercises/XX/<NN-slug>/`)

Для каждого упражнения, флагами из `MODULE_AUTHORING.md §4.1`:
1. `starter.*` + `test.*` — **компилируется**, но тесты **проваливаются** (заглушки).
2. Эталонное решение + `test.*` — **компилируется и проходит** (`exit 0`).
3. `solution` не тянет локальный `#include "x.h"` (см. авторинг §4.2 — opaque-указатель).
4. Тест детерминирован под ASan/UBSan; в `problem.md` есть ручной прогон TSan для гонок.

---

## 8. Формат отчёта о ревью

Разделяй жёстко на две части:

**Исправлено (N).** На каждый пункт: где (файл/`§`), что было, **почему неверно**,
чем подтверждено (источник/компилятор), что стало. Правки вноси сразу (не спрашивая)
для модулей-лекций.

**Проверено — верно.** Кратко перечисли значимые утверждения/листинги, которые
сверены и оказались корректны, — чтобы было видно охват, а не только дефекты.

В конце — список источников ссылками для всего, что проверялось в вебе.

## 9. Частые ошибки ревьюера (не делай так)

- **Доверять «выглядит корректно».** Компилируй и проверяй. Все реальные баги
  F1/F3 прошли бы «на глаз».
- **Выдумывать дефекты.** Если раздел верен — так и напиши «верно», не выискивай
  правки ради правок (F1 §14 был чист — туда ушла одна опечатка, и всё).
- **Отвечать про числа/версии по памяти.** Это ось №1 для веб-поиска.
- **Чинить код, ломая урок.** Если листинг намеренно демонстрирует баг — сохрани
  суть урока, исправь лишь неверный механизм/комментарий.
- **Молча менять смысл.** Каждая правка — с объяснением «почему» в отчёте.
